## 1. Executive Summary

### 1.1 Background

在企业数据分析场景中，**业务人员**（产品经理、运营、分析师）经常面临以下痛点：

1. **数据获取门槛高**：不会 SQL，每次查数据都要找数据团队
2. **等待时间长**：简单查询也要排队，影响决策效率
3. **口径不一致**：不同人查同一指标，结果可能不同
4. **工具使用复杂**：传统 BI 工具学习成本高，配置繁琐

**市场机会**：
- NL2SQL/ChatBI 市场快速增长，2024-2025 年成为 AI 应用热门赛道
- 头部产品（Aloudata Agent、数势 SwiftAgent）验证了市场需求
- 企业数字化转型加速，自助数据分析需求旺盛

**为什么现在做**：
- 大模型技术成熟（GPT-4、Claude 3.5 等）NL2SQL 准确率大幅提升
- 京东内部 DataAgent 项目已有技术积累
- 营销中台业务场景明确，可快速验证

### 1.2 Objectives

**Primary Objective**：
让业务人员用自然语言查询数据，实现「问即答，答即准」，将数据获取时间从天级缩短到分钟级。

**Secondary Objectives**：
1. 降低数据分析门槛，非技术人员也能自助查询
2. 统一数据口径，确保查询结果准确一致
3. 提升数据团队效率，减少重复性查询工作
4. 沉淀数据资产，积累常用查询和分析模式

**Success Criteria**：

| 指标        | 目标值     | 说明                    |
| --------- | ------- | --------------------- |
| SQL 生成准确率 | ≥ 85%   | 简单查询 ≥ 90%，复杂查询 ≥ 75% |
| 查询响应时间    | ≤ 40秒   | 从提问到结果展示              |
| 用户满意度     | ≥ 4.5/5 | NPS ≥ 40              |
| 周活跃用户     | ≥ 500 人 | 内部用户渗透率达到 30%         |
| 数据团队提效    | ≥ 50%   | 重复性查询减少               |

### 1.3 Scope

**In Scope**：
- 自然语言转 SQL 查询（NL2SQL）
- 查询结果可视化展示
- 查询解释和 SQL 展示
- 查询历史保存和复用
- 数据字典和 Schema 说明
- 多数据源支持（MySQL、ClickHouse、Hive）
- 基础权限控制（数据源级别）

**Out of Scope**（未来版本考虑）：
- 复杂数据建模和 ETL
- 实时流数据处理
- 高级数据挖掘和预测
- 移动端 App 原生应用
- 外部客户商业化版本

## 3. Requirements

### 3.1 Functional Requirements

#### Feature 1: 自然语言查询（NL2SQL）

**Description**：
用户输入自然语言问题，系统自动理解意图、生成 SQL、执行查询并返回结果。

**User Stories**：
- As a 业务分析师，I want 用自然语言查询数据，so that 无需等待数据团队
- As a 产品经理，I want 快速获取策略效果数据，so that 及时调整策略
- As a 业务分析师，I want 系统理解我的业务术语，so that 不用记技术字段名

**Acceptance Criteria**：
- Given 用户在查询页面，When 输入自然语言问题（如"上周各渠道转化率"），Then 系统在 10 秒内返回查询结果
- Given 用户输入模糊问题，When 系统需要澄清，Then 主动询问用户确认（如"您指的是哪个时间范围？"）
- Given 用户输入复杂问题（多表关联），When 系统生成 SQL，Then 准确率 ≥ 75%
- Given 用户输入简单问题（单表查询），When 系统生成 SQL，Then 准确率 ≥ 90%

**Priority**：P0

#### Feature 2: 查询解释

**Description**：
展示系统生成的 SQL 和查询逻辑，让用户理解结果是如何产生的。

**User Stories**：
- As a 业务分析师，I want 看到生成的 SQL，so that 验证查询逻辑正确性
- As a 产品经理，I want 理解查询步骤，so that 信任查询结果
- As a 数据分析师，I want 查看和编辑 SQL，so that 进行高级分析

**Acceptance Criteria**：
- Given 查询成功返回，When 用户点击"查看 SQL"，Then 展示格式化后的 SQL 语句
- Given 展示 SQL，When SQL 复杂，Then 提供步骤分解说明（如"Step 1: 筛选时间范围"）
- Given 用户查看 SQL，When 发现错误，Then 支持手动编辑并重新执行
- Given 查询执行，When 耗时较长，Then 展示执行进度和预计剩余时间

**Priority**：P0

#### Feature 3: 结果可视化

**Description**：
根据查询结果自动推荐合适的图表类型，并支持一键导出。

**User Stories**：
- As a 业务分析师，I want 看到数据可视化，so that 快速理解数据趋势
- As a 产品经理，I want 一键生成汇报图表，so that 制作 PPT 更高效
- As a 业务分析师，I want 导出图表和数据，so that 在 Excel 中二次分析

**Acceptance Criteria**：
- Given 查询返回结果，When 系统自动推荐图表，Then 根据数据类型推荐（趋势→折线图，占比→饼图，对比→柱状图）
- Given 展示图表，When 用户不满意推荐，Then 支持切换其他图表类型
- Given 用户查看图表，When 点击导出，Then 支持 PNG、PDF、Excel 格式
- Given 导出图表，When 选择 PPT 格式，Then 生成可直接使用的 PPT 页面

**Priority**：P0

#### Feature 4: 查询历史

**Description**：
保存用户的查询历史，支持搜索、复用和收藏常用查询。

**User Stories**：
- As a 业务分析师，I want 保存常用查询，so that 下次快速复用
- As 产品经理，I want 查看之前的查询，so that 对比不同时间的数据
- As a 业务分析师，I want 收藏重要查询，so that 建立个人数据看板

**Acceptance Criteria**：
- Given 用户查询成功，When 系统自动保存，Then 在历史记录中展示（保留 90 天）
- Given 用户查看历史，When 搜索关键词，Then 匹配查询语句和结果
- Given 用户找到历史查询，When 点击复用，Then 带入原问题并允许编辑
- Given 用户收藏查询，When 进入收藏夹，Then 按分类展示收藏的查询

**Priority**：P1

#### Feature 5: 数据字典

**Description**：
提供数据 Schema 说明和业务术语映射，帮助用户理解可用数据。

**User Stories**：
- As a 业务分析师，I want 查看有哪些数据表，so that 知道能查什么
- As a 业务分析师，I want 理解字段含义，so that 准确描述查询需求
- As a 数据工程师，I want 维护数据字典，so that 确保口径统一

**Acceptance Criteria**：
- Given 用户进入数据字典，When 浏览数据表，Then 展示表名、描述、字段列表
- Given 用户查看字段，When 字段有业务术语映射，Then 展示业务含义和示例值
- Given 用户查询时，When 输入业务术语，Then 自动映射到技术字段
- Given 数据工程师维护，When 更新字典，Then 实时同步到查询系统

**Priority**：P1

#### Feature 6: 多轮对话

**Description**：
支持上下文感知的连续对话，用户可以通过追问逐步细化查询。

**User Stories**：
- As a 业务分析师，I want 基于上次结果追问，so that 深入分析数据
- As a 产品经理，I want 系统记住上下文，so that 不用重复描述背景
- As a 业务分析师，I want 系统主动建议，so that 发现更多分析角度

**Acceptance Criteria**：
- Given 用户完成一次查询，When 输入追问（如"只看华东区"），Then 系统理解上下文并追加条件
- Given 多轮对话，When 用户切换话题，Then 系统检测并提示是否开启新对话
- Given 对话历史，When 达到 10 轮，Then 提示用户保存或开启新对话
- Given 对话结束，When 用户满意结果，Then 支持一键保存完整对话为报告

**Priority**：P1

### 3.2 Non-Functional Requirements

#### Performance

| 指标       | 要求      | 说明       |
| -------- | ------- | -------- |
| 查询响应时间   | ≤ 40 秒  | 从提交到结果展示 |
| SQL 生成时间 | ≤ 20 秒  | 大模型推理时间  |
| 页面加载时间   | ≤ 2 秒   | 首屏加载     |

- **Audit Log**：记录所有查询行为，支持审计追溯
- **SQL 注入防护**：生成 SQL 经过安全校验，禁止危险操作

#### Compatibility

- **Browser**：Chrome 90+、Firefox 88+、Safari 14+、Edge 90+
- **Device**：桌面端优先，支持平板（响应式）
- **Data Sources**：MySQL 5.7+、ClickHouse、Hive、StarRocks
- **Integration**：支持接入京东内部 BI 系统、企业微信机器人

## 4. User Experience

### 4.1 User Flows

#### Flow 1: 单次查询流程

```
1. 用户进入智能问数首页
2. 在输入框输入自然语言问题
3. 系统实时显示理解结果（意图、时间范围、指标）
4. 用户确认或修改理解结果
5. 系统生成 SQL 并执行查询
6. 展示查询结果和可视化图表
7. 用户查看、导出或分享结果
```

#### Flow 2: 多轮对话流程

```
1. 用户完成首次查询
2. 系统展示结果并推荐相关问题
3. 用户选择追问或输入新问题
4. 系统理解上下文并执行查询
5. 展示追加条件后的结果
6. 用户可以继续追问或结束对话
7. 支持保存完整对话为报告
```

#### Flow 3: 历史查询复用流程

```
1. 用户进入查询历史页面
2. 搜索或浏览历史查询
3. 找到目标查询
4. 点击"复用"带入原问题
5. 编辑问题（如修改时间范围）
6. 重新执行查询
7. 保存为新查询或更新原查询
```

### 4.2 Key Interactions

#### 交互 1: 智能输入提示
- 输入时实时显示热门查询推荐
- 识别业务术语并提示对应技术字段
- 检测时间范围并高亮显示

#### 交互 2: 查询理解确认
- 生成 SQL 前展示理解结果
- 用户可以修改理解参数
- 支持"重新理解"和"手动编辑 SQL"

#### 交互 3: 结果交互
- 表格支持排序、筛选、分页
- 图表支持切换类型、调整样式
- 点击图表元素下钻详情

#### 交互 4: 异常处理
- 查询失败时展示友好错误信息
- 提供常见错误解决方案
- 支持一键反馈问题

### 4.3 Design Requirements

#### Visual Design
- **风格**：简洁、专业、数据感
- **主色**：京东红（#E1251B）+ 科技蓝（#1890FF）
- **字体**：PingFang SC、Roboto
- **布局**：三栏布局（导航、主内容、辅助面板）

#### Interaction Patterns
- **查询输入**：类搜索框设计，支持语音输入
- **结果展示**：卡片式布局，信息层级清晰
- **反馈机制**：Toast 提示、进度条、动画过渡
- **手势支持**：桌面端快捷键，移动端滑动操作

#### Accessibility
- WCAG 2.1 Level AA 标准
- 键盘导航支持
- 屏幕阅读器兼容
- 高对比度模式

### 5.2 Data Model

#### 核心实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| User | 用户 | user_id, name, dept, role |
| Query | 查询记录 | query_id, user_id, nl_query, sql_query, status |
| DataSource | 数据源 | ds_id, name, type, connection_info |
| Schema | 数据 Schema | schema_id, ds_id, table_name, fields |
| Metric | 指标定义 | metric_id, name, formula, description |
| Report | 报告 | report_id, user_id, queries, charts |

#### 关键关系
- User → Query (1:N)
- Query → DataSource (N:1)
- DataSource → Schema (1:N)
- Schema → Metric (1:N)
- User → Report (1:N)

## 7. Risks and Mitigation

| 风险 | 影响 | 概率 | 缓解措施 | 负责人 |
|------|------|------|----------|--------|
| **SQL 准确率不达预期** | 高 | 中 | 1. 构建语义层提升准确性<br>2. 提供人工校验机制<br>3. 逐步扩大查询复杂度 | 算法团队 |
| **大模型 API 稳定性** | 高 | 中 | 1. 多模型备份（bailian code plan）<br>2. 本地缓存常用查询<br>3. 降级到规则引擎 | 工程团队 |
| **数据安全合规** | 高 | 低 | 1. 敏感数据脱敏<br>2. 行级权限控制<br>3. 审计日志记录 | 安全团队 |
| **用户接受度低** | 中 | 中 | 1. 早期用户共创<br>2. 充分培训推广<br>3. 快速迭代反馈 | 产品团队 |
| **性能不达预期** | 中 | 中 | 1. 查询结果缓存<br>2. 预计算常用指标<br>3. 异步查询大结果 | 工程团队 |
| **数据源接入延迟** | 中 | 低 | 1. 提前沟通排期<br>2. 准备测试数据<br>3. 分阶段接入 | 数据团队 |
