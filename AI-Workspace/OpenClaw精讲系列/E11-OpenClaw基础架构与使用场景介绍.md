## 一、四大核心架构

OpenClaw 的核心架构围绕四个关键系统展开：

```
┌─────────────────────────────────────────────────────────┐
│                  OpenClaw 核心架构                       │
├─────────────────────────────────────────────────────────┤
│  1. Gateway 网关系统 → 多渠道消息接入                    │
│  2. Tools 内置工具系统 → 底层核心功能                    │
│  3. Skills 插件系统 → 自由组装外部功能                   │
│  4. 人格与记忆系统 → 个性化与持久化                      │
└─────────────────────────────────────────────────────────┘
```

## 二、Gateway 网关系统详解

### 2.1 为什么需要 Gateway？

**传统 Agent 想象**：
- Web 端聊天界面
- 左侧对话列表
- 单一入口

**OpenClaw 创新思路**：
- 打破传统单入口模式
- 支持多设备、多渠道、多节点接入
- 24 小时待命的持续连接

### 2.2 Node 概念

**核心概念**：所有外部设备/渠道都被视为 Node

| Node 类型 | 示例 |
|-----------|------|
| **远程设备** | 在 Mac 上运行 OpenClaw，手机作为 Node |
| **通信软件** | 飞书、微信、WhatsApp |
| **主服务器** | 本地运行的 OpenClaw 核心 |

### 2.3 Gateway 核心功能

```
外部消息 → Gateway → Session 管理 → Agent 处理
              ↓
         维持长期连接
         会话状态管理
         定时任务调度
         事件回调处理
```

**技术特点**：
- 占据 OpenClaw 代码很大一部分
- 实现跨渠道、跨设备的消息统一管理
- 支持 Session 按场景划分
- 每个 Session 可独立重启

## 三、PyAgent 运行时

### 3.1 什么是 PyAgent？

**定义**：嵌入式的 Agent 运行时系统

**特点**：
- 不同于传统的命令-响应模式
- 持续运行的 Agent Loop
- 随时有消息随时响应
- 参数可实时更新调整

### 3.2 与传统 Agent 的区别

| 传统 Agent | PyAgent |
|------------|---------|
| 单次命令执行 | 持续运行循环 |
| 独立后端服务 | 嵌入式系统 |
| 参数固定 | 实时可调整 |
| 被动响应 | 主动监听 |

### 3.3 可动态调整的组件

- **工具列表**：实时增减可用工具
- **记忆系统**：动态更新历史消息
- **人格设定**：随时修改角色特征
- **技能配置**：按需加载 Skills

## 四、工具系统架构

### 4.1 内置工具（固定）

参考 MaCal 设计，包含四大基础工具：

| 工具 | 功能 |
|------|------|
| **命令行工具** | 执行终端命令 |
| **文件系统工具** | 读写本地文件 |
| **Brave Search** | 网络搜索 |
| **Fetch** | 获取网页内容 |

**特点**：内置固化，不可修改

### 4.2 Skills 系统（动态）

**Skills vs MCP 的本质区别**：

| 维度 | MCP | Skills |
|------|-----|--------|
| **解决问题** | 工具接入 | 系统能力 |
| **包含内容** | 仅工具 | 上下文 + 工具 + 指南 |
| **使用方式** | 技术集成 | 拖拽组装 |

**Skills 示例 - 本地文件操作**：
- ✅ 包含操作工具（MCP 部分）
- ✅ 包含操作脚本
- ✅ 包含使用指南
- ✅ 包含场景说明

### 4.3 插件系统（高级）

**Plugin > Skills > Tools 的层级**：

```
Plugin（插件）
  ├── 底层运行逻辑
  ├── 上下文工程
  ├── 工具集
  └── 完整功能体系

Skills（技能）
  ├── 上下文工程
  ├── 工具集
  └── 操作指南

Tools（工具）
  └── 单一功能
```

**实际案例**：飞书接入 = Plugin 级别的功能扩展

## 五、记忆系统设计

### 5.1 多层记忆架构

```
┌────────────────────────────────┐
│          记忆系统              │
├────────────────────────────────┤
│  不可变层 → 核心人格/能力      │
│  可调整层 → 用户偏好/历史      │
│  动态层 → 当前对话上下文       │
└────────────────────────────────┘
```

### 5.2 记忆压缩机制

**触发条件**：上下文窗口过大

**处理流程**：
```
检测上下文大小 → 超阈值 → 执行压缩 → Summary 替换原始消息
```

## 七、技术栈特点

### 7.1 非 Python 架构

**技术选型**：
- TypeScript / JavaScript
- 没有 Python 后端
- 没有 LangChain 等专业框架

**设计理念**：
- 偏轻量架构
- 后端功能相对简单
- 通用性强，多场景适用

### 7.2 与专业 Agent 的区别

| 专业 Agent | OpenClaw |
|------------|----------|
| Python + LangChain | TypeScript |
| 完善后端架构 | 轻量设计 |
| 专业场景定制 | 通用场景覆盖 |

---

## 八、Session 管理机制

### 8.1 会话划分逻辑

```
不同设备 → 不同 Session
不同渠道 → 不同 Session
不同聊天群 → 不同 Session
```

### 8.2 灵活重启

- 每个 Session 独立管理
- 可单独重启不影响其他
- 支持跨 Session 共享记忆

## 九、核心要点总结

| 模块 | 核心价值 |
|------|----------|
| **Gateway** | 打破单入口，实现多渠道统一管理 |
| **PyAgent** | 嵌入式运行时，持续监听响应 |
| **内置工具** | 四大基础工具，固定不可变 |
| **Skills** | 动态技能，拖拽即用 |
| **Plugin** | 高级扩展，完整功能体系 |
| **记忆系统** | 多层设计 + 自动压缩 |
