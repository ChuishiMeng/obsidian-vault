# P1 问题定义：数据查询系统中的意图消歧

> 最后更新：2026-03-20
> 状态：初稿

---

## 1. 研究背景

### 1.1 场景描述

数据作为企业的核心资产，支撑着业务决策。业务人员和管理者有着普遍的**快速获取数据信息**的需求。

**自然语言交互的优势**：
- 缩短数据传递链路
- 高效获取信息
- 确保数据时效性

### 1.2 核心矛盾

**语境信息量不一致**：

| 视角 | 特征 | 信息量 |
|------|------|--------|
| **用户视角** | 结果导向语言 | 隐含信息丰富 |
| **系统视角** | 原子数据存储 | 仅结构信息 |

**具体表现**：
- 用户：关心场景指标（如"复购率"），默认系统知道背景、业务知识、计算逻辑
- 系统：只存储原子数据（如订单记录），不知道如何组合成用户口中的指标

**结果**：用户意图与查询结果不一致

---

## 2. 问题定义

### 2.1 核心问题

> **如何在用户自然语言表达与数据查询系统之间，构建有效的语义对齐机制？**

### 2.2 歧义分类体系

#### 2.2.0 三层歧义框架

Data Agent 问答场景的歧义可归纳为**三个本质层面**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    用户层：个人语言特征                             │
│   表达习惯 │ 口语化/方言 │ 指代省略 │ 意图偏好 │ 角色视角            │
│                     ↓ 需要用户画像和历史学习                         │
└─────────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────────┐
│                    业务层：领域业务知识                              │
│   概念定义 │ 计算逻辑 │ 业务规则 │ 行业术语 │ 维度约定 │ 时间规则    │
│                     ↓ 需要业务知识图谱                               │
└─────────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────────┐
│                    数据层：Schema 歧义                              │
│   表映射 │ 字段对应 │ 值解析 │ 关联路径                             │
│                     ↓ 需要增强元数据                                 │
└─────────────────────────────────────────────────────────────────────┘
```

**三层框架的本质**：

| 层面      | 定位         | 特征        | 解决方案            |
| ------- | ---------- | --------- | --------------- |
| **用户层** | 歧义的**输入端** | 个性化、上下文依赖 | 用户画像、历史学习、交互确认  |
| **业务层** | 歧义的**语义层** | 领域性、跨部门差异 | 业务知识图谱、规则库      |
| **数据层** | 歧义的**实现端** | 结构性、技术性   | 增强元数据、Schema 描述 |

---

#### 2.2.1 用户层歧义：个人语言特征

**定义**：用户表达习惯、角色视角、上下文依赖导致的歧义

| 歧义类型 | 定义 | 示例 |
|----------|------|------|
| **表达习惯** | 个人语言风格差异 | "大盘" → 某些用户指整体数据，某些指核心指标看板 |
| **口语化/方言** | 非正式表达、简称、行业黑话 | "数科" → 京东数字科技控股股份有限公司 |
| **指代省略** | 上下文依赖、信息省略 | "查它的销售额" → "它"指代哪个主体？ |
| **意图模糊** | 真实目的不明确 | "最近销售怎么样" → 想看趋势？对比？异常？ |
| **角色视角** | 不同角色有不同期望 | "查销售额" → 总监看全局汇总，代表看个人业绩 |
| **偏好差异** | 个性化理解习惯 | 不同用户对"优质用户"有不同默认定义 |

**用户层歧义的上下文依赖示例**：

```
用户A（销售总监）："查销售额"
        → 期望：全公司销售汇总，按区域/产品线分解

用户B（销售代表）："查销售额"
        → 期望：个人销售业绩，按客户/时间分解

用户C（财务）："查销售额"
        → 期望：已确认收入，含税/不含税区分
```

---

#### 2.2.2 业务层歧义：领域业务知识

**定义**：业务概念定义、规则约定缺失或不一致导致的歧义

| 歧义类型 | 定义 | 示例 |
|----------|------|------|
| **概念歧义** | 业务术语定义不统一 | "优质用户" → 柜台：存款达标；理财：理财达标；信贷：按时还款 |
| **逻辑歧义** | 计算逻辑有多种可能 | "复购率" → 用户复购率 / 订单复购率 / 商品复购率？ |
| **规则歧义** | 业务规则隐含且不明确 | "月末" → 自然月末 / 财务结算日 / 业务统计日？ |
| **维度约定** | 隐含的维度条件 | "签订合同额" → 隐含"本部门、本年度" |
| **时间规则** | 时间范围定义不一 | "上个月" → 自然月/财务月？包含今天吗？ |
| **同义歧义** | 多个词汇指代同一概念 | "客户"/"用户"/"会员"/"账户" → 可能指向同一实体 |

**业务层歧义的知识图谱视角**：

```
         ┌──────────────┐
         │  "优质用户"  │ (用户查询)
         └──────┬───────┘
                │
    ┌───────────┼───────────┐
    │           │           │
    ▼           ▼           ▼
┌───────┐  ┌───────┐  ┌───────┐
│ 存款   │  │ 理财   │  │ 信贷   │
│ 达标   │  │ 达标   │  │ 达标   │
└───────┘  └───────┘  └───────┘
    │           │           │
    ▼           ▼           ▼
存款>50万  理财>100万  按时还款率>95%
```

#### 2.2.3 数据层歧义：Schema 歧义

**定义**：数据库结构映射过程中产生的歧义

| 歧义类型 | 定义 | 示例 |
|----------|------|------|
| **表歧义** | 同一概念可能对应多张表 | "订单" → 订单主表？订单明细表？订单快照表？ |
| **字段歧义** | 同一业务概念对应多个字段 | "销售额" → 销售金额？销售数量？含税销售额？ |
| **值歧义** | 字段值有多种解释 | "状态=1" → 成功？待处理？需查数据字典 |
| **路径歧义** | 多表关联路径不唯一 | 查"用户购买的商品类别" → 用户→订单→商品→类别？用户→购物车→商品→类别？ |

**数据层歧义的典型场景**：

```
问题："查询用户数量"
          │
          ├── 用户表.user_id (去重计数)
          ├── 用户表.id (主键计数)
          ├── 注册用户表.user_id
          ├── 活跃用户表.user_id
          └── 历史用户表.user_id (需要时间条件)

问题："查询销售额"
          │
          ├── order.amount (订单金额)
          ├── order_detail.price * quantity (明细金额)
          ├── sales.daily_amount (汇总金额)
          └── revenue.recognized_amount (确认收入)
```

---

#### 2.2.4 歧义传播与叠加

**歧义不是孤立存在的，而是会传播和叠加**：

```
用户问题："查上个月优质用户的销售额"

歧义传播链：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ 上个月     │ ──→ │ 优质用户   │ ──→ │ 销售额     │
│ (业务层)   │     │ (业务层)   │     │ (数据层)   │
└─────────────┘     └─────────────┘     └─────────────┘
      │                   │                   │
      ▼                   ▼                   ▼
  时间规则            概念定义            字段映射
  3种解释            3种定义            4种字段
```

**歧义叠加效应**：
- 单一歧义：O(n) 种解释
- 两个歧义：O(n²) 种组合
- k 个歧义：O(nᵏ) 种可能

**这解释了为什么简单的"让模型猜测"是不可靠的**

### 2.3 歧义来源与解决思路

| 层面 | 歧义来源 | 说明 | 解决思路 |
|------|----------|------|----------|
| **用户层** | 表达习惯、口语化/方言、角色视角 | 不同用户表达习惯不同 | 用户画像学习、交互确认 |
| **业务层** | 概念定义、规则约定、行业术语 | 领域知识缺失 | 业务知识图谱构建 |
| **数据层** | 表结构复杂、字段语义不清 | 数据库设计不友好 | 增强元数据 |

| 问题 | 描述 | 难度 |

| **歧义定位** | 如何确定歧义出现在哪个层面？ | ⭐⭐⭐⭐ |

## 3. 现有方法与局限

### 3.1 基于指标的方法

**方法**：人工构建指标系统，封装复杂计算逻辑

| 优点 | 缺点 |
|------|------|
| 标准化输出 | 人力成本高 |
| 结果可控 | 难以应对探索性分析 |
| 业务语义明确 | 无法覆盖快速增长的非标准化需求 |

### 3.2 Text2SQL 方法

**方法**：自然语言 + 元数据 → 大模型 → SQL

| 优点 | 缺点 |
|------|------|
| 灵活、通用 | 语法正确但业务错误 |
| 无需预定义指标 | 倾向于猜测而非理解 |
| 支持任意问题 | 面对业务歧义时不可靠 |

**核心问题**：大模型对用户真实意图的理解偏差

## 4. 研究目标

### 4.1 目标定位

> **构建语义驱动的意图理解智能体**：
> - 理解业务知识
> - 具有计算逻辑生成能力
> - 作为用户表达与元数据之间的中间件

### 4.2 设计理念

| 理念 | 定义 | 实现方式 |
|------|------|----------|
| **准确可信** | 结果与用户真实意图一致 | 不确定时确认，而非盲目回答 |
| **个性化** | 感知用户画像和场景 | 查询上下文知识确认意图 |
| **最小交互** | 必要时确认意图 | 反馈机制强化个性化知识 |
| **可解释** | 展示意图理解过程 | 显式提供查询与数据的关联方式 |

### 4.3 预期成果

| 维度 | 指标 | 目标 |
|------|------|------|
| **准确性** | 意图澄清准确率 | 提升 XX% |
| **效率** | 单次取数交互轮次 | 降低 XX% |
| **效率** | 单次取数时间 | 减少 XX% |

## 5. 技术挑战

### 5.1 知识获取与表示

**问题**：企业业务逻辑分散（口头约定、多文档），缺乏统一存储管理

**挑战**：
- 如何初始化获取知识（用户、行业）
- 如何更新、维护知识
- 如何高效利用知识

### 5.2 歧义复杂度

**问题**：概念和关联的多解释组合

**特征**：
- 复杂度随问题规模指数增长
- 每个概念可能有多种解释
- 概念间关联也有多种可能

### 5.3 交互式确认

**问题**：同时满足"确认意图"和"最小交互"两个目标

**矛盾**：
- 过于自信 → 幻觉
- 过于谨慎 → 用户体验差

## 6. 技术价值

### 6.1 业务价值

| 价值 | 说明 |
|------|------|
| **可信可用** | 解释"为什么这么查"，解决"幻觉"和"不可控"顾虑 |
| **释放数据价值** | 业务人员直接自然语言对话，聚焦业务信息 |
| **降低分析成本** | 减少业务与数据部门的反复沟通 |

### 6.2 技术价值

- 填补用户表达与系统理解之间的语义鸿沟
- 构建三层知识图谱（用户-语义-数据）
- 实现知识感知的意图推理

## 7. 初步框架设想

### 7.1 场景知识图谱

**三层异构结构**：

```
┌─────────────────────────────────────┐
│          用户层 (User)              │
│   用户节点：用户群体/单个用户         │
└──────────────┬──────────────────────┘
               │
┌──────────────┴──────────────────────┐
│          语义层 (Semantic)          │
│   业务概念 + 计算逻辑                │
└──────────────┬──────────────────────┘
               │
┌──────────────┴──────────────────────┐
│          数据层 (Data)              │
│   元数据（表/列）+ 关键值            │
└─────────────────────────────────────┘
```

**边类型**：
- **确定边**：客观定义和映射，权重固定
- **概率边**：个性化知识，权重可变（历史查询 + 动态反馈）

### 7.2 交互式意图推理

**流程**：
1. 从自然语言提取实体和业务概念
2. 在知识图谱中多跳检索
3. 基于用户身份更新链路
4. 生成候选意图子图
5. 歧义判别 + 交互确认
