## 1. 歧义检测方法

### 核心思路
- 采用**预训练 LLM + 上下文学习** 进行歧义识别
- 基于**细粒度歧义分类体系**对歧义短语进行分类

### 歧义分类体系 (两维度)

| 维度 | 子类别 | 说明 | 示例 |
|------|--------|------|------|
| **DB 相关歧义** | 不明确 schema 引用 | 无法确定使用哪个表/列 | "最老用户"可指年龄或注册日期 |
| | 不明确值引用 | 问题中的值与数据库实际值不对应 | 查询"New York City"但数据库存储"NYC" |
| | 缺失 SQL 关键词 | 缺少明确 SQL 操作的关键词 | "按注册日期显示用户"可指 ORDER BY/GROUP BY/WHERE |
| **LLM 相关歧义** | 不明确知识来源 | 未指定信息应从数据库检索还是 LLM 推理 | "女性员工"是查询 gender 列还是语义分析姓名 |
| | 推理上下文不足 | 缺乏足够信息引导 LLM 推理 | 查询"当前汇率"未指定货币或日期 |
| | 知识冲突 | 问题中的假设与事实/数据库内容矛盾 | 查询从未发生的事件的参与者 |
| | 模糊时空范围 | 时空约束未明确指定粒度 | "2018 世界杯后"可指决赛后或整年后 |

## 2. 交互式消歧流程

### 两阶段流水线

```
Stage 1: 歧义识别
  歧义检测模块 → 澄清问题生成模块
                    ↓
Stage 2: 迭代优化
  偏好更新模块 → 问题优化模块 → 查询重写
```

### 关键步骤

**1. 澄清问题生成**
- 将每个歧义方面转化为**针对性多选题**
- 补充相关描述或数据库 schema 片段

**2. 偏好更新**
- 使用**树形结构**索引和存储用户澄清
- 若发生语义冲突，LLM 自动更新偏好

**3. 问题优化**
- 处理用户额外约束
- 冲突时优先保留额外约束中的条件

**4. 查询重写**
- 整合原始问题 + 用户澄清 + 额外约束
- 生成清晰无歧义的重写查询

## 3. 实验结果

### SQL 生成准确率提升

| 数据集 | 无 AmbiSQL | 有 AmbiSQL | 提升幅度 |
|--------|-----------|-----------|---------|
| **平均** | 42.5% | 92.5% | **+50%** |
| TAG 样本 | 20.0% | 85.0% | +65% |
| BIRD 样本 | 75.0% | 100.0% | +25% |

### 歧义检测准确率

| 指标 | 总体 | DB 相关 | LLM 相关 |
|------|------|--------|---------|
| **精确率** | 87.2% | 83.9% | 93.8% |
| **召回率** | 89.1% | 92.9% | 83.3% |
| **F1 分数** | 88.2% | 88.1% | 88.2% |

## 4. 关键技术点

1. **细粒度歧义分类体系**: 覆盖 DB 映射和 LLM 推理两类歧义
2. **LLM 上下文学习**: 无需微调，通过 prompt 工程实现歧义检测
3. **树形偏好存储**: 层次化索引用户澄清
4. **交互式多选题**: 降低用户回答不确定性
5. **轻量级集成**: 可作为预处理模块与现有系统集成
