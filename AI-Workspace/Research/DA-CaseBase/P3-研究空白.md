### 1.3 矩阵核心发现

**四个关键洞察**：

1. **案例定义空白**：没有任何现有方法给出完整的案例形式化定义。CBR-to-SQL 最接近（模板+实体分离），但缺少语义上下文、执行验证、质量评分等元信息。
2. **案例表示理论缺失**：CBR-to-SQL 的实体掩码是唯一的模板抽象方法，但缺乏理论指导——什么该掩码、什么不该掩码？复杂 SQL 的泛化边界不清晰。
3. **案例匹配单一**：所有方法都依赖纯 Embedding 匹配，没有结构化+语义的混合匹配方案。LangCache 的 92% Precision 意味着 8% 误匹配，在企业场景不可接受。
4. **案例应用两极化**：要么人工保证质量（CBR-to-SQL、RubikSQL），要么自动但无质量控制（GPTCache、LangCache）。没有"自动+高质量"的方案。

## Step 2：核心证据支撑

### 2.1 案例定义——CBR-to-SQL 的两阶段分离

**CBR-to-SQL (P02) 的案例结构**：
- **已有**：NL问题 → SQL模板 + 实体映射表
- **缺失**：
  - 语义上下文（该查询的业务意图、适用场景）
  - 执行验证结果（SQL 是否可执行、结果是否合理）
  - 质量评分（案例的可靠程度、使用频次、成功率）
  - 版本信息（对应的 Schema 版本、创建/更新时间）
  - 关联关系（与其他案例的相似/互斥/依赖关系）

**NL2SQL-BUGs (P09) 的警示**：即使 SQL 执行成功，仍有 **25% 的语义错误未检出**。这意味着没有执行验证元信息的案例定义，无法保证案例质量。

### 2.2 案例表示——实体掩码的泛化边界

**CBR-to-SQL 实体掩码方法的能力边界**（Opus API 技术分析，2026-04-14）：

| SQL 复杂度 | 模板匹配成功率 | 根因 |
|-----------|:---:|------|
| 简单查询 | ~95% | 掩码规则明确 |
| 1层嵌套 | ~82% | 基本可处理 |
| 2层嵌套 | ~61% | 模板爆炸开始 |
| 3层嵌套 | ~40% | 掩码边界模糊 |
| 2表JOIN | 良好 | 模板充足 |
| 3表JOIN | 中等 | JOIN顺序组合增多 |
| 4+表JOIN | 较差 | JOIN路径歧义+模板稀疏 |
| 窗口函数 | ~35-40% | 函数类型×PARTITION×ORDER×FRAME组合空间庞大 |

**核心问题**：缺乏理论指导，不知道什么该掩码、什么不该掩码。例如：
- `WHERE department = 'Sales'`：`'Sales'` 应该掩码 ✅
- `ORDER BY salary DESC LIMIT 10`：`10` 和 `DESC` 该掩码吗？❓
- `JOIN employees ON e.dept_id = d.id`：JOIN 条件该掩码吗？❓

### 2.3 案例匹配——语义边界问题

**核心挑战**：语义相似但逻辑不同的查询

| 查询对 | Embedding距离 | 逻辑差异 | 能区分？ |
|--------|:---:|---------|:---:|
| "销售额前10" vs "销售额后10" | 极近 | 排序方向相反 | ❌ |
| "本月新增用户" vs "本月活跃用户" | 近 | 过滤条件不同 | ❌ |
| "上季度华东区销售额" vs "上季度华东区订单数" | 近 | 聚合字段不同 | ⚠️ |
| "高血压患者" vs "血压高的患者" | 相同 | 无差异（应匹配） | ✅ |

**LangCache (P01)**：Precision 92% → 8% 误匹配率在企业场景不可接受
**Semantic Cache OLAP (P15)**：查询规范化提升命中率至 76%，但仅验证 OLAP 场景

**根本原因**：纯 Embedding 匹配无法捕捉结构化语义特征（聚合类型、排序方向、时间范围、过滤条件）。

### 2.4 案例应用——质量保证的两难困境

| 方法 | 沉淀方式 | 质量保证 | 问题 |
|------|---------|---------|------|
| CBR-to-SQL | 手动构建 | ✅ 人工审核 | 不可扩展，成本高 |
| RubikSQL | 半自动+审核 | ✅ 人工审核 | 更新频率低（周/月），延迟高 |
| GPTCache | 自动缓存 | ❌ 无验证 | 缓存错误答案无法发现 |
| LangCache | 自动缓存 | ❌ 无验证 | 同上 |
| Semantic Cache OLAP | 自动缓存 | ❌ 无验证 | 同上 |

**RubikSQL (P06) 的 KB 更新机制**：批量更新（周/月级），人工参与度高，非实时。

**退化问题**：
- BEAVER (P17) 报告表检索失败率 **51.3%**，表明 Schema 变更频繁
- 现有方法要么依赖 TTL 过期（粗粒度），要么依赖人工发现（高成本）
- 没有自动检测案例退化（Schema 变更导致失效、案例冲突、冗余积累）的机制

## Step 3：研究空白声明

### Gap 1：案例定义不完整

> 现有方法对 NL2SQL 案例的定义不完整。CBR-to-SQL [P02] 提出了最接近的案例结构——两阶段分离（SQL 模板 + 实体映射），但案例仅包含 NL-SQL 对，**缺少语义上下文、执行验证结果、质量评分、版本信息等元信息**。RubikSQL [P06] 的知识库条目为非结构化文本，更无法称为形式化案例定义。NL2SQL-BUGs [P09] 表明 25% 的语义错误在执行成功时仍未被检出，凸显了执行验证元信息的必要性。
>
> **研究空白**：NL2SQL 场景下缺少完整的案例形式化定义——现有定义仅覆盖"问题-解决方案"对，缺少支撑质量保证和生命周期管理的元信息结构。
>
> **本研究方案**：提出 NL2SQL 案例的多层形式化定义，包含：核心层（NL-SQL模板-实体映射）、上下文层（业务意图、适用场景、Schema版本）、质量层（执行验证结果、置信度评分、使用统计）、关联层（相似/互斥/依赖关系）。

### Gap 2：案例表示的抽象粒度缺乏理论指导

> CBR-to-SQL [P02] 的实体掩码是目前唯一的 SQL 模板抽象方法，在简单查询上效果良好（~95%），但在复杂 SQL 上显著退化：嵌套子查询 3层仅~40%，多表 JOIN 4+表较差，窗口函数总体仅~35-40%。**根本问题不在于方法实现，而在于缺乏理论指导**——什么 SQL 元素该被抽象（掩码）、什么不该？掩码粒度的选择依据是什么？不同 SQL 结构（嵌套/JOIN/窗口函数）的泛化边界在哪里？
>
> **研究空白**：SQL 模板抽象缺乏理论框架——掩码粒度的选择无理论依据，复杂 SQL 结构的泛化边界不清晰。
>
> **本研究方案**：提出多粒度案例表示理论，定义抽象层级（值级→条件级→子句级→结构级），建立各层级的泛化收益-风险模型，为不同 SQL 复杂度提供理论最优的抽象粒度选择。

### Gap 3：案例匹配的语义边界区分能力不足

> 现有语义匹配方法（GPTCache [P04]、LangCache [P01]、Semantic Cache OLAP [P15]）均基于纯 Embedding 相似度。LangCache 达到 Precision 92% [P01]，Semantic Cache OLAP 通过查询规范化将命中率提升至 76% [P15]。但**纯 Embedding 无法区分"语义相似但逻辑不同"的查询**（如"top 10" vs "bottom 10"），8% 的误匹配率在企业数据查询场景不可接受。Semantic Cache OLAP 的规范化方法仅在 OLAP 场景验证，未推广到通用 NL2SQL。
>
> **研究空白**：NL2SQL 案例匹配缺少结构化+语义的混合匹配机制——纯 Embedding 无法捕捉聚合类型、排序方向、时间范围等结构化语义特征，导致逻辑不同的查询被错误匹配。
>
> **本研究方案**：设计语义+结构混合匹配算法——先通过 Embedding 粗筛候选案例，再通过结构化特征提取（聚合函数、排序方向、过滤条件、时间范围）进行精确验证，解决语义边界区分问题。

### Gap 4：案例自动沉淀缺乏质量保证与退化检测

> 现有案例/知识沉淀存在两极化困境：CBR-to-SQL [P02] 手动构建案例库，RubikSQL [P06] 需人工审核入库（批量更新，周/月级）——质量有保证但不可扩展；GPTCache [P04]、LangCache [P01] 自动缓存——可扩展但无质量控制，错误案例无法自动发现。同时，BEAVER [P17] 报告企业数据库表检索失败率 51.3%，表明 Schema 变更频繁，但**没有方法能自动检测案例退化**（Schema 变更导致失效、案例冲突、冗余积累），仅依赖 TTL 过期或人工发现。
>
> **研究空白**：NL2SQL 案例的自动沉淀缺乏无人工干预的质量保证机制，同时缺少案例退化的自动检测与修复能力。
>
> **本研究方案**：设计自动案例沉淀与演化机制——多维度自动验证（执行验证+结果合理性+一致性验证+对抗验证）保证入库质量；Schema 变更感知的退化检测算法 + 定期一致性检查保证案例库健康度；置信度评分驱动的案例生命周期管理（新增→验证→成熟→退化→淘汰）。

## Step 4：研究问题（RQ）

### RQ 与 Gap 一一对应

| RQ | 对应 Gap | 核心问题 |
|----|---------|---------|
| **RQ1** | Gap 1（案例定义） | 如何形式化定义 NL2SQL 案例，使其支撑完整的案例生命周期管理？ |
| **RQ2** | Gap 2（案例表示） | SQL 模板的多粒度抽象理论——不同 SQL 结构的最优抽象粒度是什么？泛化边界在哪里？ |
| **RQ3** | Gap 3（案例匹配） | 如何设计语义+结构混合匹配算法，在保证 Precision ≥ 95% 的前提下最大化案例命中率？ |
| **RQ4** | Gap 4（案例应用） | 如何在无人工干预下实现案例的自动沉淀、质量保证与退化检测？ |

### RQ 详细描述

**RQ1（案例定义的形式化）**：
> 在 NL2SQL 场景中，一个"案例"应包含哪些信息？如何形式化定义案例结构，使其不仅承载"问题-解决方案"映射，还能支撑质量评估、版本管理、关联分析等生命周期管理需求？

**RQ2（案例表示的多粒度方法）**：
> SQL 模板抽象存在多个粒度层级（值级、条件级、子句级、结构级），不同粒度的泛化能力和风险如何权衡？对于不同复杂度的 SQL 结构（简单查询、嵌套子查询、多表JOIN、窗口函数），最优的抽象粒度是什么？是否存在理论上的泛化边界？

**RQ3（案例匹配的混合算法）**：
> 如何结合语义 Embedding 和结构化特征（聚合类型、排序方向、过滤条件、时间范围等），设计混合匹配算法，精确区分"语义相似且逻辑相同"与"语义相似但逻辑不同"的查询？该算法能否在保证 Precision ≥ 95% 的前提下实现高命中率？

**RQ4（案例应用的自动沉淀与演化）**：
> 如何设计无人工干预的自动案例沉淀机制，使新生成的 NL2SQL 结果经过多维度自动验证后入库？如何自动检测案例退化（Schema 变更失效、案例冲突、冗余积累），并实现案例库的持续健康演化？

## Step 5：创新点凝练

### 三大创新点

| 创新点 | 对应 RQ/Gap | 核心贡献 | 与现有方法差异 |
|--------|-----------|---------|--------------|
| **I1: 案例的多粒度形式化表示理论** | RQ1+RQ2 / Gap1+Gap2 | 提出 NL2SQL 案例的多层形式化定义 + 多粒度模板抽象理论 | vs CBR-to-SQL 仅模板+实体分离，无元信息；vs 无理论指导的掩码 |
| **I2: 语义+结构混合匹配算法** | RQ3 / Gap3 | 解决纯 Embedding 的语义边界问题，Precision 目标 ≥ 95% | vs GPTCache/LangCache 纯 Embedding（92% Precision）；vs P15 仅 OLAP |
| **I3: 自动案例沉淀与演化机制** | RQ4 / Gap4 | 无人工干预的质量保证 + 退化检测 + 生命周期管理 | vs CBR-to-SQL 手动构建；vs RubikSQL 人工审核；vs GPTCache 无质量控制 |

### 创新点详细说明

**I1: 案例的多粒度形式化表示理论**

核心思想：
- **案例定义**：多层结构（核心层→上下文层→质量层→关联层），每层承载不同的生命周期管理功能
- **案例表示**：多粒度抽象层级（值级→条件级→子句级→结构级），建立泛化收益-风险模型
- 理论贡献：首次为 NL2SQL 案例提供完整的形式化定义和抽象理论框架

**I2: 语义+结构混合匹配算法**

核心思想：
- **粗筛阶段**：Embedding 语义匹配，快速召回候选案例
- **精排阶段**：结构化特征提取（聚合函数类型、排序方向、过滤条件模式、时间范围表达），精确验证语义边界
- 解决问题：区分 "top 10" vs "bottom 10"、"新增用户" vs "活跃用户" 等语义近但逻辑异的查询

**I3: 自动案例沉淀与演化机制**

核心思想：
- **自动沉淀**：多维度验证（执行验证→结果合理性→一致性验证→对抗验证）→ 置信度评分 → 自动入库
- **退化检测**：Schema 变更感知 + 定期一致性检查 + 使用效果监控
- **生命周期管理**：新增 → 验证 → 成熟 → 退化 → 淘汰，全程自动化
- 解决问题：打破"人工保证质量"与"自动但无质量控制"的两难困境

## Step 6：论文 Story

### 新 Story

> **问题**：Data Agent 在企业场景中需要持续积累和复用 NL2SQL 查询知识，但现有方法存在根本性缺陷——要么依赖人工构建和审核案例（CBR-to-SQL、RubikSQL），不可扩展；要么自动缓存但无质量控制（GPTCache、LangCache），不可信赖。
>
> **洞察**：现有工作将案例库视为工程组件（缓存/知识库），缺少系统性的理论框架。案例库的四个核心子问题——定义、表示、匹配、应用——各自存在理论空白，需要统一的生命周期理论。
>
> **方案**：我们提出 NL2SQL 案例库的完整生命周期理论（DA-CaseBase），包含三大创新：
> 1. **多粒度形式化表示理论**——定义案例的多层结构和多粒度抽象方法
> 2. **语义+结构混合匹配算法**——解决纯 Embedding 的语义边界问题
> 3. **自动案例沉淀与演化机制**——无人工干预的质量保证和退化检测
>
> **价值**：使 Data Agent 能**自动积累和复用查询知识**，随使用时间增长而变得更快、更准、更省。

### Story 对比

| 维度 | 旧 Story | 新 Story |
|------|---------|---------|
| 聚焦点 | 三层架构比 CHESS 更快 | 案例库的完整生命周期理论 |
| 贡献类型 | 系统/工程 | 理论+方法 |
| 核心论点 | 缓存加速 NL2SQL | Data Agent 能自动积累和复用查询知识 |
| 竞争优势 | 速度和成本 | 理论完整性 + 自动化程度 |
