## 一、数据集选择

### 1.1 核心数据集

#### BIRD-SQL（主实验平台）

| 属性 | 说明 |
|------|------|
| **全称** | BIg Bench for LaRge-scale Database Grounded Text-to-SQL Evaluation |
| **规模** | dev set ~1,534 条查询，95 个数据库 |
| **难度分级** | simple / moderate / challenging |
| **核心优势** | 提供 **knowledge evidence**（领域知识注释） |
| **适用性** | ⭐⭐⭐⭐⭐ |

**选择理由**：
1. **knowledge evidence 直接对应案例库的领域知识沉淀假设**——BIRD 的 evidence 可类比为"已沉淀的案例知识"，验证案例库在有/无领域知识时的效果差异
2. **难度分级支持分层评估**——案例库对 simple/moderate/challenging 三类查询的增益可分别量化
3. **企业级复杂度**——涉及多表 JOIN、嵌套查询、领域特定函数，贴近实际场景
4. **社区广泛采用**——便于与 CHASE-SQL、CHESS、DIN-SQL 等 SOTA 方法直接对比

#### Spider（基准对比平台）

| 属性 | 说明 |
|------|------|
| **规模** | 200 个数据库，1,018 条测试查询，7,000+ 训练查询 |
| **难度分级** | easy / medium / hard / extra hard |
| **核心优势** | 学术界最广泛使用的 Text-to-SQL 基准 |
| **适用性** | ⭐⭐⭐⭐ |

**选择理由**：
1. **广泛的 baseline 覆盖**——几乎所有 NL2SQL 方法都在 Spider 上报告了结果，便于横向对比
2. **数据库多样性高**——200 个不同领域的数据库，适合验证案例库的跨领域泛化能力
3. **复杂查询子集丰富**——hard/extra hard 查询涉及嵌套、多表 JOIN、窗口函数，适合测试多粒度表示的极限

**在本研究中的定位**：Spider 作为**辅助验证平台**，主要用于：
- 与已有方法（CBR-to-SQL 等）在相同数据集上对比
- 验证案例库方法在不同基准上的泛化一致性

### 1.2 辅助数据集

#### Spider 2.0（跨领域泛化验证）

| 属性 | 说明 |
|------|------|
| **规模** | 600+ 企业级工作流任务 |
| **特点** | 跨数据库、跨方言（BigQuery/Snowflake/PostgreSQL） |
| **适用性** | ⭐⭐⭐ |

**用途**：评估案例库方法在更复杂的企业级工作流场景下的适用性，特别是跨数据库方言时案例表示和匹配的鲁棒性。

#### BEAVER（真实企业场景验证）

| 属性 | 说明 |
|------|------|
| **特点** | 来自真实企业环境的 NL2SQL 交互数据 |
| **适用性** | ⭐⭐（如数据可获取） |

**用途**：如数据可获取，用于验证案例库在真实企业交互模式下的表现——特别是重复查询率、查询模式分布等假设的真实性。

### 1.3 数据集选择理由汇总

| 选择原则 | 对应数据集 | 验证目标 |
|---------|-----------|---------|
| **领域知识沉淀假设** | BIRD-SQL (knowledge evidence) | 案例库沉淀领域知识 → 提升准确率 |
| **跨领域泛化能力** | Spider (200 databases) | 案例库在新领域的冷启动和泛化 |
| **企业级复杂度** | Spider 2.0 (enterprise workflows) | 案例库在真实工作流中的鲁棒性 |
| **社区可比性** | BIRD + Spider | 与 SOTA 方法直接对比 |

**重要约束**：本研究**不使用京东内部数据**，全部基于公开数据集，确保可重复性和学术透明度。

### 1.4 案例库专用数据构建

由于现有数据集并非为案例库生命周期评估设计，需要在公开数据集基础上构建**案例库专用评估数据**：

| 构建数据 | 来源 | 构建方法 | 规模估计 |
|---------|------|---------|---------|
| **等价查询对** | BIRD dev set | LLM 改写 + 人工验证 | ~500 对 |
| **语义边界对** | BIRD dev set | 构造"相似但不等价"的查询对 | ~300 对 |
| **多粒度模板** | Spider train set | 自动模板抽取 + 人工审核 | ~2,000 模板 |
| **Schema 变更场景** | Spider databases | 模拟列名/表名/类型变更 | ~50 个变更场景 |
| **查询模式分布** | BIRD dev set | 聚类分析高频查询模式 | ~100 种模式 |

## 二、评估指标体系

### 2.1 核心指标（对应四个 RQ）

#### RQ1：案例定义——等价性判别指标

| 指标 | 定义 | 计算方法 | 目标值 |
|------|------|---------|--------|
| **等价判别 Precision** | 判定为等价的案例对中，实际等价的比例 | TP / (TP + FP) | ≥ 0.95 |
| **等价判别 Recall** | 实际等价的案例对中，被正确判定的比例 | TP / (TP + FN) | ≥ 0.90 |
| **等价判别 F1** | Precision 和 Recall 的调和平均 | 2·P·R / (P+R) | ≥ 0.92 |
| **分层判别准确率** | 三层判别（结构/语义/意图）各层的准确率 | 各层 Accuracy | 结构≥0.98, 语义≥0.92, 意图≥0.85 |

**评估方法**：在人工标注的等价查询对（~500 对）和语义边界对（~300 对）上评估，标注由 2 名标注者独立完成，Cohen's κ ≥ 0.8 视为标注可靠。

#### RQ2：案例表示——多粒度表示指标

| 指标 | 定义 | 计算方法 | 目标值 |
|------|------|---------|--------|
| **覆盖率 (Coverage)** | 测试集中能被案例库模板覆盖的查询比例 | 命中查询数 / 总查询数 | L1≥0.30, L2≥0.50, L3≥0.70 |
| **实例化准确率 (Instantiation Acc.)** | 模板实例化后 SQL 执行正确的比例 | 实例化正确数 / 实例化总数 | L1≥0.95, L2≥0.85, L3≥0.70 |
| **泛化-精确 Pareto 曲线** | 不同粒度下覆盖率 vs 实例化准确率的权衡 | 绘制 Pareto 前沿 | 支配现有方法 |
| **复杂查询表示率** | 嵌套/多表JOIN/窗口函数的有效模板化比例 | 按复杂度分层统计 | 嵌套≥0.60, JOIN≥0.75 |

**三个粒度层次**（对应 P4 方案设计）：

| 粒度层 | 名称 | 说明 | 泛化度 | 精确度 |
|--------|------|------|--------|--------|
| **L1** | 实例级 | 原始 NL-SQL 对，实体槽位化 | 低 | 高 |
| **L2** | 模板级 | SQL 模板（参数化 + 结构骨架） | 中 | 中 |
| **L3** | 模式级 | 查询意图模式（如"分组聚合+排序+TopN"） | 高 | 低 |

#### RQ3：案例匹配——匹配效果指标

| 指标 | 定义 | 计算方法 | 目标值 |
|------|------|---------|--------|
| **Hit@1** | 排名第一的候选即为正确匹配的比例 | 正确@1 / 总查询 | ≥ 0.75 |
| **Hit@5** | 前5个候选中包含正确匹配的比例 | 正确@5 / 总查询 | ≥ 0.90 |
| **匹配 Precision** | 返回的候选中实际匹配正确的比例 | 正确匹配数 / 返回候选数 | ≥ 0.85 |
| **边界区分率** | 语义边界附近的查询对能被正确区分的比例 | 边界正确区分数 / 边界总对数 | ≥ 0.92（vs GPTCache 92%→目标降误判至<3%） |
| **检索延迟 (Latency)** | 从查询到返回 Top-K 候选的时间 | P50 / P99 延迟 | P50<50ms, P99<200ms (10K案例库) |

**边界区分率的详细定义**：

构造"top10 vs bottom10"难例对——语义相似度在 0.85-0.95 区间但实际不等价的查询对，评估匹配算法区分正确匹配和边界误匹配的能力。这直接回应 GPTCache 报告的 8% 边界误判问题（P3-研究空白 Gap2）。

#### RQ4：案例应用与演化——应用效果指标

| 指标 | 定义 | 计算方法 | 目标值 |
|------|------|---------|--------|
| **SQL 执行准确率 (EX)** | 案例匹配后实例化 SQL 的执行正确率 | 执行正确数 / 匹配命中数 | ≥ 0.90 |
| **案例命中率 (Case Hit Rate)** | 总查询中能被案例库命中的比例 | 命中查询数 / 总查询数 | ≥ 0.40（稳态） |
| **Token 节省率** | 案例命中场景下 LLM Token 消耗的降低比例 | (基线Token - 命中Token) / 基线Token | ≥ 0.50 |
| **自动沉淀质量** | 自动入库案例中质量合格的比例 | 合格案例数 / 自动入库总数 | ≥ 0.90 |
| **退化检测召回率** | Schema 变更后失效案例被检出的比例 | 检出失效数 / 实际失效数 | ≥ 0.90 |

### 2.2 端到端指标

| 指标 | 定义 | 计算方法 | 对比方式 |
|------|------|---------|---------|
| **整体 EX (Execution Accuracy)** | 案例库增强后的 NL2SQL 整体执行准确率 | 执行正确数 / 总查询数 | CaseBase + NL2SQL vs 纯 NL2SQL |
| **分难度 EX** | 按 simple/moderate/challenging 分层的 EX | 各难度层独立计算 | 分层对比增益 |
| **响应时间（命中）** | 案例命中时的平均响应时间 | 从查询到返回 SQL 的端到端时间 | vs 纯 LLM 生成时间 |
| **响应时间（未命中）** | 案例未命中时的端到端时间 | 含案例库检索开销 | vs 纯 NL2SQL（评估额外开销） |
| **质量评分有效性** | 自动质量评分与人工评分的相关性 | Spearman rank correlation | ρ ≥ 0.80 |
| **案例库增长曲线** | 随使用量增加，案例库规模和命中率的变化 | 绘制增长曲线 | 验证"渐进沉淀"假设 |

### 2.3 指标与 RQ/贡献的对应关系

| 贡献 | 对应 RQ | 核心验证指标 | 辅助指标 |
|------|--------|-------------|---------|
| **C1: 形式化定义与等价性判别** | RQ1 | 等价判别 F1, 分层判别准确率 | 去重率, 标注一致性 |
| **C2: 多粒度表示** | RQ2 | Pareto 曲线, 覆盖率, 实例化准确率 | 复杂查询表示率 |
| **C3: 混合匹配算法** | RQ3 | Hit@1, Hit@5, 边界区分率 | 检索延迟, Precision |
| **C4: 自动沉淀与演化** | RQ4 | 沉淀质量, 退化检测召回率, 案例命中率 | Token 节省率, 增长曲线 |

## 三、Baseline 选择

| Baseline | 来源 | 覆盖的案例库环节 | 对应的对比维度 |
|---------|------|----------------|--------------|

| Baseline | 说明 | 对比目的 |
|---------|------|---------|
| **纯 NL2SQL（无案例库）** | 下界——仅用 LLM 生成 SQL，无任何案例支持 | 证明案例库的整体增益 |

| **DIN-SQL** | 任务分解 + CoT | 验证案例库作为 few-shot 来源的增益 |
| **DAIL-SQL** | In-context Learning | 验证案例库提供的上下文质量 |

### 3.3 消融 Baseline（自身变体）

| 变体 | 说明 | 对比目的 |
|------|------|---------|
| **Ours-SemanticOnly** | 仅语义匹配，无结构特征 | 验证混合匹配（I2）的增益 |
| **Ours-StructureOnly** | 仅结构匹配，无语义特征 | 验证语义通道的必要性 |
| **Ours-L1Only** | 仅 L1 实例级表示 | 验证多粒度表示（I1）的增益 |
| **Ours-L2Only** | 仅 L2 模板级表示 | 验证多粒度协同的价值 |
| **Ours-NoEvolution** | 关闭自动沉淀和演化 | 验证演化机制（I3）的增益 |
| **Ours-NoQuality** | 关闭质量验证，直接入库 | 验证质量保证的必要性 |
| **Ours-ExecVerifyOnly** | 仅执行验证，无多维质量评分 | 验证多维质量评分的增益 |

### 3.4 Baseline 选择理由

| 选择原则 | 说明 |
|---------|------|
| **直接可比** | CBR-to-SQL 是唯一显式使用案例推理的 NL2SQL 方法，是最直接的对比对象 |
| **覆盖主要范式** | 缓存（GPTCache）、知识库（RubikSQL）、工作负载（TailorSQL）三种范式均覆盖 |
| **SOTA 集成** | 与 CHASE-SQL 等 SOTA 集成，证明案例库是"正交增益"而非替代方案 |
| **充分消融** | 7 个自身变体，分别验证三个创新点（I1/I2/I3）的独立贡献 |

## 四、实验协议设计

### 4.1 案例库构建协议

#### 种子案例构建

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 查询模式聚类 | 对 BIRD dev set 进行查询模式聚类，识别高频模式 |
| 2 | 代表性案例选取 | 每种模式选取 1-3 个代表性查询作为种子案例 |
| 3 | 模板抽取 | 对种子案例执行 L1/L2/L3 三层抽象 |
| 4 | 质量验证 | 对种子案例执行执行验证 + 人工审核 |
| 5 | 案例库初始化 | 将验证通过的案例入库 |

#### 案例库规模控制实验

| 案例库规模 | 构建方式 | 验证目的 |
|-----------|---------|---------|
| **100 案例** | 手动精选高频模式 | 冷启动最小可用规模 |
| **500 案例** | 自动聚类 + 人工审核 | 中等规模效果 |
| **1,000 案例** | 自动沉淀 + 质量过滤 | 接近稳态的效果 |
| **5,000 案例** | 全自动沉淀（模拟长期使用） | 大规模案例库的性能和质量 |

#### 冷启动实验

从 **0 案例** 开始，模拟真实使用场景：
1. 按时间顺序逐条处理 BIRD dev set 查询
2. 每条查询先尝试匹配案例库 → 未命中则调用 NL2SQL → 验证通过后自动入库
3. 记录每个阶段（0/50/100/200/500/1000 案例）的命中率和 EX 变化
4. 绘制**案例库增长曲线**和**命中率增长曲线**

### 4.2 消融实验设计

#### 实验 A：匹配算法消融（验证 I2 混合匹配）

| 配置 | 语义通道 | 结构通道 | 融合方式 | 对应 RQ |
|------|---------|---------|---------|--------|
| A1: 纯语义 | ✅ Embedding | ❌ 关闭 | — | RQ3 |
| A2: 纯结构 | ❌ 关闭 | ✅ 指纹+AST | — | RQ3 |
| A3: 混合（加权） | ✅ | ✅ | 固定权重 α=0.6 | RQ3 |
| A4: 混合（自适应） | ✅ | ✅ | 复杂度自适应权重 | RQ3 |

**预期结论**：A4 > A3 > A1 > A2（语义为主，结构补充边界区分）

#### 实验 B：粒度消融（验证 I1 多粒度表示）

| 配置 | L1 实例级 | L2 模板级 | L3 模式级 | 对应 RQ |
|------|---------|---------|---------|--------|
| B1: 仅 L1 | ✅ | ❌ | ❌ | RQ2 |
| B2: 仅 L2 | ❌ | ✅ | ❌ | RQ2 |
| B3: 仅 L3 | ❌ | ❌ | ✅ | RQ2 |
| B4: L1 + L2 | ✅ | ✅ | ❌ | RQ2 |
| B5: L1 + L2 + L3（完整） | ✅ | ✅ | ✅ | RQ2 |

**预期结论**：B5 在 Pareto 曲线上支配其他配置——兼顾覆盖率和精确度

#### 实验 C：质量验证消融（验证 I3 自动演化中的质量保证）

| 配置 | 执行验证 | 多维质量评分 | 反馈调优 | 对应 RQ |
|------|---------|------------|---------|--------|
| C1: 无验证 | ❌ | ❌ | ❌ | RQ4 |
| C2: 仅执行验证 | ✅ | ❌ | ❌ | RQ4 |
| C3: 执行 + 质量评分 | ✅ | ✅ | ❌ | RQ4 |
| C4: 完整（含反馈调优） | ✅ | ✅ | ✅ | RQ4 |

**预期结论**：C4 的长期案例库质量显著优于 C1-C3，尤其在 1000+ 案例时差距明显

#### 实验 D：案例库规模消融

| 配置 | 案例库大小 | 主要观察指标 | 对应 RQ |
|------|-----------|------------|--------|
| D1 | 100 | 冷启动命中率, EX | RQ4 |
| D2 | 500 | 命中率增长拐点 | RQ4 |
| D3 | 1,000 | 接近稳态表现 | RQ4 |
| D4 | 5,000 | 大规模性能, 检索延迟 | RQ3, RQ4 |

**预期结论**：命中率在 500-1000 区间出现显著拐点，5000 案例时接近稳态（边际收益递减）

### 4.3 主实验设计

#### 实验 E：端到端效果评估（BIRD-SQL）

| 实验配置 | 说明 | 评估指标 |
|---------|------|---------|
| E1: 纯 NL2SQL | 无案例库，直接 LLM 生成 | EX, 响应时间, Token 消耗 |
| E2: CaseBase + NL2SQL | 本研究完整方法 | EX, 命中率, 响应时间, Token 节省率 |
| E3: CBR-to-SQL | 直接对比 | EX, 匹配准确率 |
| E4: GPTCache + NL2SQL | 纯缓存对比 | EX, 命中率, 边界误判率 |
| E5: CaseBase + CHASE-SQL | 与 SOTA 集成 | EX（vs 纯 CHASE-SQL） |
