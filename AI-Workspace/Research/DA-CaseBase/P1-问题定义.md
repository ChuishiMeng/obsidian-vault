## 一、Background（研究背景）

### 1.1 场景描述

**研究领域**：企业数据分析与自然语言数据库查询

**应用场景**：

企业数据分析场景中，业务人员（非技术用户）需要通过自然语言查询业务数据获取业务洞察。Data Agent 作为自然语言与数据库之间的桥梁，负责将用户的自然语言问题转换为可执行的 SQL 查询，并返回准确的结果。

**典型应用案例**：

| 案例 | 用户问题 | 期望结果 |
|------|---------|---------|
| 销售分析 | "上季度华东区各产品线销售额是多少？" | 准确的销售数据报表 |
| 用户洞察 | "过去30天新增用户中，有多少完成了首单？" | 用户转化率数据 |
| 库存管理 | "库存周转率低于行业平均的SKU有哪些？" | 库存预警清单 |

**利益相关者**：

| 角色 | 需求 | 痛点 |
|------|------|------|
| 业务分析师 | 快速获取数据洞察 | SQL 学习成本高，等待数据团队排期 |
| 数据团队 | 提高数据服务效率 | 重复查询消耗大量精力 |
| 企业管理层 | 数据驱动决策 | 数据获取时效性不足 |

---

### 1.2 面临瓶颈

**核心挑战**：Data Agent 的问数准确度与效率不足

#### 问题一：准确度不足

**现象描述**：
- 相同问题，LLM 可能生成不同的 SQL，导致结果不一致
- NL2SQL/NL2Python 生成式方法无法保证 100% 准确率
- 问数查询对准确性要求极高，不允许歧义

**问题严重程度**：

| 维度 | 量化数据 | 来源 |
|------|---------|------|
| SOTA 执行准确率 | Spider: 85.7%, BIRD: 59.8% | SQLForge (2025) |
| 语义错误检出率 | 75.16%（仍有 25% 未检出） | NL2SQL-BUGs (2025) |
| 企业场景需求 | 95%+ 准确率 | 行业实践 |

**根本原因**：
1. **语义鸿沟**：自然语言与 SQL 之间存在语义对齐难题
2. **Schema 复杂性**：企业数据库 Schema 庞大，字段语义难以理解
3. **LLM 不确定性**：生成式方法固有的随机性

#### 问题二：效率不足

**现象描述**：
- 每次查询都依赖 LLM 推理，响应时间长（30~50秒）
- Token 消耗大，API 成本高
- 相似问题重复计算，资源浪费

**量化数据**：

| 维度 | 当前状态 | 目标状态 |
|------|---------|---------|
| 响应时间 | 30~50 秒 | < 1s（案例命中） |
| Token 消耗 | 500-2000 tokens/查询 | 0 tokens（案例命中） |
| 成本 | $0.01-0.05/查询 | $0（案例命中） |

**数据支持**：
- 约 33% 的查询是重复的（Web 搜索研究，Lempel & Moran, 2003）
- 语义缓存可降低 90% 成本（Text2SQL Hub, 2024）

#### 问题三：知识无法沉淀

**现象描述**：
- 高频查询无法积累复用
- 专家知识依赖个人经验
- 系统无法自我进化

**现有方案不足**：

| 方案 | 优点 | 局限性 |
|------|------|--------|
| 纯 LLM 生成 | 灵活、泛化强 | 准确率不稳定、成本高 |
| 规则系统 | 准确率可控 | 维护成本高、覆盖有限 |
| RAG 增强 | 知识可更新 | 检索质量依赖向量库 |

---

### 1.3 发展现状

#### 主流技术路线

**路线一：LLM 直接生成（NL2SQL）**

代表方法：DIN-SQL, DAIL-SQL, SQLForge

| 方法 | 核心思路 | 效果 |
|------|---------|------|
| DIN-SQL | 任务分解 + Chain-of-Thought | Spider Dev: 85.3% |
| DAIL-SQL | In-context Learning | Spider Dev: 86.2% |
| SQLForge | 数据合成 + Fine-tuning | Spider Dev: 85.7% |

**局限性**：每次查询都需 LLM 推理，成本高、延迟大

**路线二：语义缓存（Semantic Caching）**

代表方法：GPTCache, Semantic Cache for OLAP

| 方法 | 核心思路 | 效果 |
|------|---------|------|
| GPTCache | Embedding 相似度匹配 | 成本降低 90%，响应提速 9x |
| LangCache | 领域专用 Embedding | Precision: 0.92, Recall: 0.90 |

**局限性**：冷启动问题、语义边界模糊

**路线三：案例/模板驱动（Case-Based）**

代表方法：CBR-to-SQL (2026), TailorSQL (2025)

| 方法 | 核心思路 | 效果 |
|------|---------|------|
| CBR-to-SQL | Masked SQL 模板 + Few-shot | 提升泛化能力 |
| TailorSQL | 历史查询工作负载利用 | 准确率提升 2x |

**局限性**：模板库构建成本、泛化能力有限

#### 技术演进趋势

```
2020-2022: PLM-based (BERT, T5)
    ↓
2023-2024: LLM + Prompt Engineering
    ↓
2024-2025: 语义缓存 + 历史工作负载利用
    ↓
2025-2026: 案例驱动 + 持续学习（本研究方向）
```

#### SOTA 方法总结

| 类别 | 方法 | 准确率 | 成本 | 知识沉淀 |
|------|------|--------|------|---------|
| 生成式 | SQLForge | 高 | 高 | ❌ |
| 缓存式 | GPTCache | 命中时 100% | 低 | ✅ |
| 模板式 | CBR-to-SQL | 高 | 中 | ✅ |
| 工作负载 | TailorSQL | 高 | 中 | ✅ |

**研究机会**：整合语义缓存 + 案例模板 + 历史工作负载，形成多层次知识沉淀体系

## 二、What（研究目标）

### 2.1 Goal（定位与理念）

**研究定位**：

本研究定位于 **Data Agent 问数案例自动沉淀和匹配应用**，聚焦于通过案例库机制提升问数准确度与效率，填补「纯生成式方法」与「规则系统」之间的技术空白。

**定位图谱**：

```
           准确率
             ↑
             │    ┌─────────────┐
             │    │ 规则系统    │ ← 维护成本高
             │    │ (100%)      │
             │    └─────────────┘
             │         
             │           ┌─────────────┐
             │           │ ★ 本研究    │ ← 目标位置
             │           │ (95%+)      │
             │    ┌──────┴─────────────┘
             │    │ 案例库驱动    │
             │    └───────────────┘
             │         
             │    ┌─────────────┐
             │    │ LLM生成式   │ ← 准确率不稳定
             │    │ (80-90%)    │
             │    └─────────────┘
             └──────────────────────→ 灵活性
```

**理念与原则**：

| 原则       | 说明           | 实践方式        |
| -------- | ------------ | ----------- |
| **准确优先** | 企业场景对准确率要求极高 | 案例经人工验证后入库  |
| **渐进沉淀** | 案例库随使用自然增长   | 自动沉淀 + 人工审核 |
| **高效复用** | 重复查询零成本命中    | 三层缓存架构      |
| **可解释性** | SQL 可审计、可修正  | 模板化存储       |

## 四、研究问题

基于以上分析，提出以下研究问题：

### RQ1: 语义相似度匹配

> **如何准确识别语义相似的数据库查询？**

- 子问题 1.1：不同 Embedding 模型在查询相似度匹配上的效果差异？
- 子问题 1.2：如何设定相似度阈值以平衡准确率与召回率？
- 子问题 1.3：如何处理语义相似但逻辑不同的查询（如"销售额前10" vs "销售额后10"）？

### RQ2: 案例模板表示与泛化

> **如何设计 SQL 模板表示方法以最大化复用能力？**

- 子问题 2.1：Masked SQL 模板的最佳表示形式？
- 子问题 2.2：如何从具体 SQL 抽象为可泛化的模板？
- 子问题 2.3：模板库的组织结构与检索策略？

### RQ3: 三层架构设计

> **如何整合语义缓存、模板库、LLM 生成形成高效系统？**

- 子问题 3.1：三层之间的优先级与回退策略？
- 子问题 3.2：如何在保证准确率的前提下最大化缓存命中率？
- 子问题 3.3：系统的可扩展性与实时更新机制？

### RQ4: 企业场景验证

> **案例库驱动的 Data Agent 在企业场景中的实际效果如何？**

- 子问题 4.1：高频查询的覆盖率与命中率？
- 子问题 4.2：与传统 LLM 生成方法的效果对比？
- 子问题 4.3：用户满意度与采纳率？

## 五、研究范围与边界

### 研究范围

| 维度 | 范围界定 |
|------|---------|
| **查询类型** | Select 查询（统计、聚合、筛选） |
| **数据库类型** | 关系型数据库（MySQL、PostgreSQL） |
| **语言** | 中文自然语言查询 |
| **场景** | 企业内部数据分析 |

### 研究边界（不包含）

| 排除项 | 原因 |
|--------|------|
| Insert/Update/Delete 操作 | 企业场景以查询为主，写操作风险高 |
| NoSQL 数据库 | 聚焦关系型数据库，简化研究范围 |
| 多轮对话 | 聚焦单轮查询，多轮对话作为未来工作 |
| 图像/语音输入 | 聚焦文本查询 |
