## 一、Background（研究背景）

### 1.1 场景描述

**研究领域**：企业数据分析中的案例知识管理与复用

**应用场景**：

企业数据分析场景中，业务人员（非技术用户）通过 Data Agent 以自然语言查询业务数据。在长期使用过程中，大量成功的查询-SQL 映射构成了宝贵的**案例知识**——它们承载了领域理解、Schema 映射经验和业务逻辑。然而，这些案例知识目前**无法系统性地沉淀和复用**：每次查询仍依赖 LLM 从零推理，已有经验不断被丢弃。

**典型应用案例**：

| 案例 | 用户问题 | 期望结果 |
|------|---------|---------|
| 销售分析 | "上季度华东区各产品线销售额是多少？" | 准确的销售数据报表 |
| 用户洞察 | "过去30天新增用户中，有多少完成了首单？" | 用户转化率数据 |
| 库存管理 | "库存周转率低于行业平均的SKU有哪些？" | 库存预警清单 |

**利益相关者**：

| 角色 | 需求 | 痛点 |
|------|------|------|
| 业务分析师 | 快速获取数据洞察 | SQL 学习成本高，等待数据团队排期 |
| 数据团队 | 提高数据服务效率 | 重复查询消耗大量精力，知识无法传承 |
| 企业管理层 | 数据驱动决策 | 数据获取时效性不足，经验依赖个人 |

---

### 1.2 案例库理论缺失的现状

**核心问题**：Data Agent 中的案例知识无法沉淀和复用，缺乏系统化的案例库理论

#### 问题一：案例定义缺失——什么是"案例"？

**现象描述**：
- 现有系统将查询-SQL 对视为简单的键值缓存，缺乏对"案例"的形式化定义
- 两个表面不同但语义等价的查询无法被识别为同一案例
- 缺乏案例等价性判别标准，导致案例库中大量冗余或遗漏

**问题量化**：

| 维度 | 现状 | 根本原因 |
|------|------|---------|
| 冗余率 | 同一意图可能产生数十个"不同"缓存条目 | 无等价性判别 |
| 遗漏率 | 语义相同但措辞不同的查询无法命中 | 缺乏形式化定义 |
| 边界误判 | GPTCache 报告 8% 的边界误判率 | 缺乏理论指导的相似度阈值 |

#### 问题二：案例表示粗糙——如何抽象和泛化？

**现象描述**：
- 案例以原始 SQL 字符串存储，无法泛化到相似但不同的查询
- 缺乏多粒度的表示方法：精确匹配和模糊匹配之间无中间层
- 嵌套查询、多表 JOIN 等复杂结构的泛化能力急剧下降（CBR-to-SQL: 82%→40%）

**现有方案对比**：

| 表示方法 | 泛化能力 | 精确性 | 局限 |
|---------|---------|--------|------|
| 原始 SQL 字符串 | 极低 | 极高 | 只能精确匹配 |
| Embedding 向量 | 高 | 低 | 语义边界模糊 |
| Masked SQL 模板 | 中 | 中 | 粒度固定，缺乏理论指导 |

#### 问题三：案例匹配不可靠——如何准确检索？

**现象描述**：
- 纯向量相似度匹配无法区分语义相似但逻辑相反的查询
- 匹配算法缺乏结构信息利用，忽略 SQL 的组合语义
- 语义边界问题：相似度 0.85 和 0.90 的案例可能有本质不同

**问题严重程度**：

| 维度 | 量化数据 | 来源 |
|------|---------|------|
| 语义边界误判 | 8% 边界误判率 | GPTCache (2023) |
| 逻辑反转误匹配 | "销售额最高" vs "销售额最低" 相似度 > 0.9 | 向量相似度固有缺陷 |
| 复杂查询匹配 | 嵌套查询匹配准确率下降 50%+ | CBR-to-SQL (2026) |

#### 问题四：案例无法演化——知识如何持续积累？

**现象描述**：
- 案例入库后缺乏质量保证机制，错误案例可能长期存在
- Schema 变更导致案例批量失效，无自动检测和修复机制
- 案例库随时间退化，缺乏自维护能力

**量化数据**：
- 约 33% 的查询是重复的（Lempel & Moran, 2003），但无法有效积累
- Schema 变更可能导致 30-60% 的已有案例失效（企业实践估算）
- 案例成功率随时间自然衰减，类似软件老化现象

### 1.3 相关技术现状

#### NL2SQL 方法——案例库的消费者与集成环境

NL2SQL 方法（如 DIN-SQL、DAIL-SQL、CHESS、SQLForge 等）是将自然语言转换为 SQL 的基础设施。本研究**不聚焦于改进 NL2SQL 方法本身**，而是将其视为**案例库的消费者和集成环境**——案例库为 NL2SQL 方法提供高质量的先验知识。

| 方法 | 核心思路 | 与案例库的关系 |
|------|---------|--------------|
| DIN-SQL | 任务分解 + Chain-of-Thought | 可从案例库获取 few-shot 示例 |
| DAIL-SQL | In-context Learning | 依赖高质量案例作为上下文 |
| CHESS | Schema 过滤 + 候选生成 | 案例库可预筛选 Schema |
| SQLForge | 数据合成 + Fine-tuning | 案例库可提供高质量训练数据 |

**关键洞察**：这些方法的准确率瓶颈（Spider: 85-86%, BIRD: ~60%）部分源于缺乏可靠的案例知识支撑。案例库理论的突破可以系统性地提升所有下游 NL2SQL 方法的效果。

#### 现有案例/缓存方法——缺乏理论基础

| 方法 | 覆盖的生命周期环节 | 缺失的环节 |
|------|------------------|-----------|
| GPTCache | 匹配（向量相似度） | 定义、表示、演化 |
| LangCache | 匹配（领域 Embedding） | 定义、表示、演化 |
| CBR-to-SQL | 表示（Masked SQL）、匹配 | 定义、演化 |
| TailorSQL | 匹配（工作负载利用） | 定义、表示、演化 |

**研究机会**：**没有任何现有方法覆盖案例库的完整生命周期**。定义、表示、匹配、应用/演化四个环节缺乏统一的理论框架，各方法只解决了局部问题。

#### 技术演进趋势

```
2020-2022: PLM-based NL2SQL (BERT, T5)
    ↓
2023-2024: LLM + Prompt Engineering + 语义缓存
    ↓
2024-2025: 案例/模板驱动 + 历史工作负载利用
    ↓
2025-2026: 案例库完整生命周期理论（本研究方向）
```

## 二、What（研究目标）

### 2.1 Goal（定位与理念）

**研究定位**：

本研究聚焦于 **Data Agent 案例库的完整生命周期理论**——从案例的定义、表示、匹配到应用/演化，建立系统化的形式化方法体系。研究目标不是"怎么做 NL2SQL"（那是基础设施），而是回答**"案例知识如何被定义、组织、检索和演化"**这一根本性问题。

**定位图谱**：

```
                    理论完备性
                       ↑
                       │    ┌─────────────────────┐
                       │    │ ★ 本研究             │ ← 完整生命周期理论
                       │    │ 定义+表示+匹配+演化   │
                       │    └─────────────────────┘
                       │         
                       │    ┌─────────────────────┐
                       │    │ CBR-to-SQL / TailorSQL│ ← 局部方法
                       │    │ 表示+匹配             │
                       │    └─────────────────────┘
                       │         
                       │    ┌─────────────────────┐
                       │    │ GPTCache / LangCache  │ ← 工程实现
                       │    │ 仅匹配               │
                       │    └─────────────────────┘
                       └────────────────────────────→ 实践覆盖
```

**理念与原则**：

| 原则 | 说明 | 实践方式 |
|------|------|---------|
| **理论先行** | 先建立形式化定义，再指导工程实现 | 案例等价性理论 → 匹配算法设计 |
| **生命周期完整** | 覆盖定义→表示→匹配→应用全流程 | 四个环节统一理论框架 |
| **渐进沉淀** | 案例库随使用自然增长并自我优化 | 自动沉淀 + 质量保证 + 演化机制 |
| **可解释性** | 案例匹配和应用过程可审计、可修正 | 形式化表示 + 结构化匹配 |

## 四、研究问题

基于以上分析，提出以下四个研究问题，对应案例库的完整生命周期：

### RQ1: 案例定义——什么构成一个有价值的案例？

> **如何形式化定义 Data Agent 场景中的"案例"？案例的等价性判别标准是什么？**

**核心科学问题**：案例的形式化定义与等价性边界

- 子问题 1.1：案例的数学定义是什么？（形式化表达：案例 = ⟨NL, SQL, Schema, Context⟩ 四元组？还是更复杂的结构？）
- 子问题 1.2：两个案例在什么条件下等价？等价性的层次（字面等价、语义等价、功能等价）如何定义？
- 子问题 1.3：案例的价值如何量化？（频率、覆盖度、复杂度、正确率等维度的综合评估）
- 子问题 1.4：语义边界附近的临界案例特征是什么？（GPTCache 报告 8% 边界误判的根源分析）

**验证路径**：Spider/BIRD 数据集 + 人工标注的等价查询对 + 边界测试集

### RQ2: 案例表示——如何抽象表示案例？

> **案例如何进行多粒度表示？不同粒度如何平衡泛化能力与匹配精度？**

**核心科学问题**：多粒度表示方法与泛化-精确权衡

- 子问题 2.1：案例表示的粒度层次如何定义？（从精确 SQL → 参数化模板 → 结构骨架 → 意图类型，共几层？）
- 子问题 2.2：各粒度层次的泛化能力与精确性的 Pareto 前沿在哪里？（CBR-to-SQL: 泛化增益 vs 错误风险）
- 子问题 2.3：复杂结构（嵌套查询、多表 JOIN、窗口函数）在不同粒度下的表示效果如何？
- 子问题 2.4：案例表示如何编码领域知识（Schema 语义、业务规则）？

**验证路径**：Spider 复杂查询子集 + 泛化测试（同模板不同实体）+ 表示层次消融实验

### RQ3: 案例匹配——如何高效准确地检索案例？

> **如何融合语义信息与结构特征实现高效准确的案例匹配？语义边界问题如何解决？**

**核心科学问题**：语义+结构混合匹配与边界判别

- 子问题 3.1：纯语义匹配 vs 纯结构匹配 vs 混合匹配的效果差异？各自的适用条件？
- 子问题 3.2：语义边界问题的本质是什么？如何检测和处理边界案例？（相似度连续空间中的离散决策问题）
- 子问题 3.3：匹配算法如何处理多粒度表示？不同粒度层的匹配策略如何协调？
- 子问题 3.4：匹配效率如何保证？大规模案例库（10K+）下的检索延迟控制？

**验证路径**：Spider/BIRD 匹配实验 + 构造的语义边界测试集 + 大规模检索性能测试

### RQ4: 案例应用与演化——匹配后如何使用？案例如何持续积累？

> **案例匹配后如何实例化应用？案例如何自动沉淀、质量保证和持续演化？**

**核心科学问题**：案例实例化机制与自动演化体系

- 子问题 4.1：匹配到的案例如何实例化为当前查询的具体 SQL？（参数替换、结构适配、上下文注入）
- 子问题 4.2：案例的自动沉淀机制如何设计？（触发条件、质量过滤、去重策略）
- 子问题 4.3：案例质量如何持续保证？（成功率跟踪、异常检测、自动清理）
- 子问题 4.4：Schema 变更后案例如何自动检测失效并修复？退化规律是什么？
- 子问题 4.5：案例库的长期演化模型是什么？（增长曲线、稳态规模、最优更新策略）

**验证路径**：模拟 Schema 变更实验 + 时间衰减模拟 + 自动沉淀质量评估

## 五、研究范围与边界

### 研究范围

| 维度 | 范围界定 |
|------|---------|
| **理论范围** | 案例的定义、表示、匹配、应用/演化四个生命周期环节 |
| **查询类型** | Select 查询（统计、聚合、筛选） |
| **数据库类型** | 关系型数据库（MySQL、PostgreSQL） |
| **语言** | 中文自然语言查询 |
| **场景** | 企业内部数据分析 |
| **集成环境** | 与主流 NL2SQL 方法（CHESS、DIN-SQL 等）的集成 |

### 研究边界（不包含）

| 排除项 | 原因 |
|--------|------|
| NL2SQL 方法本身的改进 | 本研究将其作为消费者/集成环境，不是研究对象 |
| Insert/Update/Delete 操作 | 企业场景以查询为主，写操作风险高 |
| NoSQL 数据库 | 聚焦关系型数据库，简化研究范围 |
| 多轮对话 | 聚焦单轮查询，多轮对话作为未来工作 |
| 图像/语音输入 | 聚焦文本查询 |

## 六、关键假设

| 假设 | 说明 | 验证路径 |
|------|------|---------|
| H1 | 案例知识可以被形式化定义，等价性可以被可靠判别 | 形式化定义 + 人工标注等价对验证 |
| H2 | 多粒度表示可以在泛化和精确之间找到有效平衡 | 多粒度消融实验 + Pareto 前沿分析 |
| H3 | 语义+结构混合匹配优于单一维度匹配 | Spider/BIRD 对比实验 |
| H4 | 案例库可以通过自动机制持续演化并保持质量 | 长期模拟实验 + 退化检测评估 |
| H5 | 约 33% 的查询存在可复用的案例模式 | 基于 Web 搜索研究 (Lempel & Moran 2003) 的类比推断 |

## 七、预期风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|---------|
| 等价性定义过于严格导致覆盖率低 | 中 | 高 | 多层次等价性（字面/语义/功能），逐步放宽 |
| 多粒度表示复杂度过高 | 中 | 中 | 从 2-3 个粒度起步，逐步扩展 |
| 混合匹配算法计算开销大 | 低 | 中 | 分层过滤策略，粗筛+精排 |
| 企业数据难以获取 | 中 | 高 | 使用公开数据集 + 合成数据 |
| 评估基准不适用 | 中 | 中 | 自建案例库生命周期评估框架 |
| 案例演化实验周期长 | 中 | 中 | 模拟加速实验 + 分阶段验证 |
