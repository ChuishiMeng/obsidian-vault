## 一、实验总体设计

### 1.1 实验目标

验证四个研究问题（RQ1-RQ4）对应的假设：

| 假设 | 对应RQ | 核心验证内容 |
|------|--------|-------------|
| **H1** | RQ1 | 案例知识可以被形式化定义，等价性可以被可靠判别（F1 ≥ 95%） |
| **H2** | RQ2 | 多粒度表示（L1/L2/L3）可以在泛化和精确之间找到有效平衡 |
| **H3** | RQ3 | 语义+结构混合匹配优于单一维度匹配（Precision ≥ 95%，边界误判 < 3%） |

## 二、实验一：案例定义与等价性判别（RQ1）

### 2.1 实验目的

验证案例形式化定义和等价性判别的有效性。具体回答：

- 三层等价性判别（结构等价 → 语义等价 → 意图等价）能否准确识别等价案例？
- 等价性判别相比纯文本比对和纯 Embedding 方法的优势有多大？
- 边界案例（语义近但逻辑不同）的误判率能否降至 3% 以下？

### 2.2 数据准备

从 BIRD-SQL 构造四类等价查询对：

| 等价类型 | 构造方法 | 示例 | 目标数量 |
|---------|---------|------|---------|
| **实体替换等价** | 替换具体实体值，保持SQL结构不变 | "北京的销售额" → "上海的销售额" | 200+ 对 |
| **同义表述等价** | 替换NL中的时间/条件表述 | "上个月" → "前30天"；"超过" → "大于" | 150+ 对 |
| **SQL重写等价** | 改写SQL结构，保持结果集相同 | LEFT JOIN → 子查询；IN → EXISTS | 150+ 对 |
| **非等价边界** | 语义相似但逻辑不同的查询对 | "top10" vs "bottom10"；"总量" vs "平均" | 200+ 对 |

**构造流程**：
```
BIRD-SQL原始数据 → 自动生成候选等价对(LLM辅助) → 人工验证标注 → 测试集
```

**质量保证**：
- SQL 重写等价对需在数据库上执行验证结果集一致
- 非等价边界案例需由两名标注员独立标注，Cohen's Kappa ≥ 0.8

### 2.3 评估方法

**评估指标**：

| 指标 | 定义 | 目标 |
|------|------|------|
| **Precision** | 判为等价的案例中真正等价的比例 | ≥ 95% |
| **Recall** | 真正等价的案例中被正确识别的比例 | ≥ 90% |
| **F1** | Precision 和 Recall 的调和平均 | ≥ 92% |
| **边界误判率** | 非等价边界案例被误判为等价的比例 | < 3% |

**对比基线**：

| 基线方法 | 说明 |
|---------|------|
| **B1: 纯SQL文本比对** | 字符串编辑距离 / Jaccard 相似度 |
| **B2: 纯NL Embedding** | Sentence-BERT / BGE 余弦相似度 + 阈值 |
| **B3: SQL AST比对** | 解析SQL AST后树编辑距离 |
| **Ours: 三层等价性判别** | 结构等价 → 语义等价 → 意图等价（P4方案） |

### 2.4 分析维度

- 按等价类型分别报告（实体替换 / 同义表述 / SQL重写 / 非等价边界）
- 按SQL复杂度分层报告（简单 / 中等 / 复杂 / 极复杂）
- 错误案例深度分析：哪些类型最容易误判？根因是什么？

## 三、实验二：多粒度案例表示（RQ2）

### 3.1 实验目的

验证多粒度表示（L1/L2/L3）对泛化能力的影响：

- 三个粒度层级各自的覆盖率和实例化准确率如何？
- 是否存在最优粒度选择策略？
- 复杂SQL（嵌套/多表JOIN/窗口函数）在不同粒度下表现差异？

### 3.2 实验设计

**三种粒度定义**（对应 P4 方案）：

| 粒度 | 表示内容 | 匹配方式 |
|------|---------|---------|
| **L1 具体实例** | 原始 NL + SQL | 向量精确匹配 |
| **L2 抽象模板** | NL模板 + SQL模板 + 实体槽位 | 模板匹配 + 实体填充 |
| **L3 查询模式** | SQL结构指纹 + 查询意图类型 | 结构哈希 + 意图匹配 |

**测试场景**：

| 场景 | 说明 | 测试查询数 |
|------|------|-----------|
| **同实体查询** | 与案例库中完全相同的查询 | 500 |
| **换实体查询** | 结构相同但实体不同（"北京"→"广州"） | 500 |
| **换表述查询** | 意图相同但NL表述不同 | 500 |
| **新结构查询** | 案例库中无直接匹配的全新结构 | 500 |

**核心指标**：

| 指标 | 定义 |
|------|------|
| **覆盖率** | 新查询能匹配到案例的比例（Hit Rate） |
| **实例化准确率** | 匹配到案例后，实例化生成的SQL执行正确的比例（EX） |
| **F1-Coverage** | 综合覆盖率和准确率的调和指标 |

### 3.3 预期结果

| 粒度 | 同实体 | 换实体 | 换表述 | 新结构 | 总体覆盖率 | 实例化准确率 |
|------|--------|--------|--------|--------|-----------|------------|
| **L1** | ★★★★★ | ★☆☆☆☆ | ★★☆☆☆ | ☆☆☆☆☆ | 低（~20%） | 极高（~98%） |
| **L2** | ★★★★★ | ★★★★★ | ★★★★☆ | ★☆☆☆☆ | 中高（~55%） | 高（~90%） |
| **L3** | ★★★★★ | ★★★★★ | ★★★★★ | ★★★☆☆ | 高（~70%） | 中（~75%） |
| **L1+L2+L3（级联）** | ★★★★★ | ★★★★★ | ★★★★★ | ★★★☆☆ | 高（~70%） | 高（~88%） |

**关键预期**：
- L2 模板在简单-中等查询上覆盖率最高，性价比最优
- L1 实例在精确匹配场景准确率最高，但覆盖率最低
- L3 模式在复杂查询和全新查询上有独特价值
- 三粒度级联（优先L1 → 降级L2 → 降级L3）综合表现最优

### 3.4 按SQL复杂度分层分析

| 复杂度 | SQL特征 | 数据来源 | 预期表现差异 |
|--------|---------|---------|-------------|
| **简单** | 单表 + 简单WHERE | BIRD-SQL easy | L1/L2差异小 |
| **中等** | 2-3表JOIN + GROUP BY | BIRD-SQL moderate | L2优势显现 |
| **复杂** | 嵌套子查询 + 窗口函数 | BIRD-SQL challenging | L3开始发挥作用 |
| **极复杂** | 3+层嵌套 + 4+表JOIN | BIRD-SQL extra hard | 所有粒度均退化 |

## 四、实验三：案例匹配算法（RQ3）

### 4.1 实验目的

验证语义+结构混合匹配优于单一匹配方法：

- 混合匹配在整体准确率上的提升幅度
- 关键贡献：混合匹配在**语义边界案例**上的区分能力
- 负例区分机制的独立贡献

### 4.2 实验设计

**四种匹配策略对比**：

| 方法 | 召回方式 | 精排方式 | 负例区分 |
|------|---------|---------|---------|
| **M1: 纯语义** | Embedding ANN | 余弦相似度排序 | ❌ |
| **M2: 纯结构** | SQL结构哈希 + 倒排索引 | AST结构相似度 | ❌ |
| **M3: 混合匹配** | 多路召回（向量+结构+关键词） | 多维相似度加权融合 | ❌ |
| **M4: 混合+负例** | 多路召回 | 多维相似度 + 对比检查 | ✅ |

**评估指标**：

| 指标 | 定义 | 目标 |
|------|------|------|
| **Hit@1** | Top-1 匹配正确率 | ≥ 80% |
| **Hit@5** | Top-5 中包含正确案例的比例 | ≥ 92% |
| **Precision@5** | Top-5 匹配结果的精确率 | ≥ 95% |
| **边界区分率** | 语义边界案例的正确区分率 | ≥ 97%（误判 < 3%） |
| **MRR** | 平均倒数排名 | ≥ 0.85 |

**对比基线（外部方法）**：

| 基线 | 来源 | 说明 |
|------|------|------|
| **GPTCache** | Bang et al. 2023 | 纯 Embedding 语义缓存 |
| **LangCache** | P01 | 领域 Embedding + 阈值 |
| **CBR-to-SQL** | P02 | 实体掩码 + NL Embedding |

### 4.3 边界案例测试

构造 20+ 组语义边界测试对，每组包含"语义近但逻辑不同"的查询对：

| # | 查询A | 查询B | 差异类型 |
|---|-------|-------|---------|
| 1 | 销售额最高的10个城市 | 销售额最低的10个城市 | 排序方向 |
| 2 | 本月总销售额 | 本月平均销售额 | 聚合类型（SUM vs AVG） |
| 3 | 新增用户数 | 活跃用户数 | 过滤条件 |
| 4 | 超过100万的客户 | 低于100万的客户 | 比较方向（GT vs LT） |
| 5 | 有多少订单 | 有没有订单 | 查询意图（COUNT vs EXISTS） |
| 6 | 上个月的数据 | 下个月的预测 | 时间方向 |
| 7 | 按部门分组 | 按城市分组 | 分组维度 |
| 8 | 去重后的用户数 | 所有用户数 | DISTINCT差异 |
| 9 | 最近7天 | 最近7个月 | 时间粒度 |
| 10 | 包含关键词的记录 | 不包含关键词的记录 | NOT条件 |
| ... | ... | ... | ... |

**核心度量**：各匹配方法在这 20+ 组边界案例上的区分准确率。

## 五、实验四：案例应用与自动演化（RQ4）

### 5.1 实验目的

验证案例库增强 NL2SQL 的端到端效果，以及自动沉淀机制的质量。

### 5.2 实验设计

#### 5.2a 端到端效果（案例库增强 NL2SQL）

**实验设置**：

| 配置 | NL2SQL 方法 | 案例库 | 说明 |
|------|-----------|--------|------|
| **基线** | CHESS / DIN-SQL | ❌ 无 | 纯 NL2SQL 生成 |
| **+案例库** | CHESS / DIN-SQL | ✅ DA-CaseBase | 案例库增强 |

**评估维度**：

| 指标 | 定义 |
|------|------|
| **EX（Execution Accuracy）** | SQL 执行结果与标准答案一致的比例 |
| **案例命中率** | 查询被案例库命中的比例（按匹配等级分层） |
| **LLM 调用次数** | 平均每查询的 LLM 调用次数 |
| **平均延迟** | 查询响应时间 |

**分复杂度报告**：

| 复杂度 | 基线 EX | +案例库 EX（预期） | 提升 |
|--------|---------|-------------------|------|
| 简单 | ~75% | ~90% | +15pp |
| 中等 | ~55% | ~68% | +13pp |
| 复杂 | ~35% | ~45% | +10pp |
| 极复杂 | ~20% | ~25% | +5pp |

**命中率增长曲线**：
- 模拟查询流到达顺序，记录案例库规模随时间增长
- 每 100 个查询为一个窗口，报告该窗口的命中率
- 预期曲线：初期命中率低 → 快速增长 → 趋于稳态

#### 5.2b 自动沉淀质量

**实验设置**：模拟冷启动场景

```
Round 0: 案例库为空
Round 1: 处理 500 查询，自动沉淀通过验证的案例
Round 2: 再处理 500 查询，利用 Round 1 的案例库
...
Round N: 重复，直到案例库增长趋于稳态
```

**评估指标**：

| 指标 | 定义 | 目标 |
|------|------|------|
| **沉淀准确率** | 自动入库案例中正确案例的比例 | ≥ 90% |
| **沉淀覆盖率** | 应入库的案例中实际入库的比例 | ≥ 70% |
| **EX 对比** | 自动沉淀案例库 vs 人工构建案例库的端到端 EX | 差距 < 5pp |
| **案例库增长曲线** | 案例数随查询轮次的增长趋势 | 呈 S 型收敛 |

**消融：有/无质量验证**

| 配置 | 质量验证 | 预期沉淀准确率 | 预期案例库噪声 |
|------|---------|---------------|--------------|
| **全验证** | 执行验证 + 结果合理性 + 一致性 + 对抗验证 | ≥ 90% | 低 |
| **仅执行验证** | 仅检查 SQL 可执行 | ~75% | 中（NL2SQL-BUGs 报告 25% 语义错误） |
| **无验证** | 所有成功查询直接入库 | ~60% | 高 |

#### 5.2c 退化检测

**实验设置**：模拟 Schema 变更场景

| 变更类型 | 模拟方式 | 影响范围 |
|---------|---------|---------|
| **列重命名** | 将 `amount` 改为 `total_amount` | 引用该列的案例 |
| **表删除** | 删除 `order_details` 表 | 引用该表的案例 |
| **列类型变更** | `price` 从 INT 改为 DECIMAL | 涉及该列比较的案例 |
| **新增表/列** | 添加 `returns` 表 | 不直接影响，但可能需要更新 |

**评估指标**：

| 指标 | 定义 | 目标 |
|------|------|------|
| **退化召回率** | Schema 变更后失效案例的检出率 | ≥ 90% |
| **退化精确率** | 标记为退化的案例中确实失效的比例 | ≥ 85% |
| **检测延迟** | 从 Schema 变更到检出退化的时间 | < 1 min（增量检测） |

## 六、消融实验汇总

| # | 消融内容 | 控制变量 | 对应创新点 | 验证问题 |
|---|---------|---------|-----------|---------|
| **A1** | 匹配算法 | 纯语义 / 纯结构 / 混合 / 混合+负例 | I2 | 混合匹配的每个组件贡献多大？ |
| **A2** | 表示粒度 | 仅L1 / 仅L2 / 仅L3 / L1+L2+L3级联 | I1 | 各粒度层级的独立贡献？ |
| **A3** | 质量验证 | 全验证 / 无验证 / 仅执行验证 | I3 | 多维验证比单一验证好多少？ |
| **A4** | 案例库大小 | 100 / 500 / 1K / 5K 案例 | 全部 | 案例库规模对效果的影响曲线？ |
| **A5** | 负例区分 | 有负例区分 / 无负例区分 | I2 | 负例机制对边界案例的改善？ |

### 消融实验详细设计

**A1：匹配算法消融**
- 固定案例库（相同案例集），仅改变匹配算法
- 报告 Hit@1、Hit@5、Precision、边界区分率
- 关注点：混合匹配在边界案例上的提升是否显著（McNemar 检验）

**A2：表示粒度消融**
- 固定匹配算法（混合+负例），仅改变可用粒度层级
- 报告覆盖率、实例化准确率、端到端 EX
- 关注点：级联策略是否优于任意单一粒度

**A3：质量验证消融**
- 固定沉淀触发条件，仅改变验证深度
- 运行 N 轮后比较案例库质量（人工抽样审核 200 条）
- 关注点：对抗验证和一致性验证的边际贡献

**A4：案例库规模消融**
- 从完整案例库中随机采样不同规模子集
- 报告每个规模下的 Hit Rate 和端到端 EX
- 关注点：是否存在拐点（边际收益递减的阈值）

**A5：负例区分消融**
- 固定其他所有组件，开/关负例区分模块
- 在全量测试集和边界测试集上分别报告
- 关注点：负例区分在全量测试集上的代价（是否增加误拒）vs 边界集上的收益

## 七、统计检验与结果报告规范

### 7.1 统计检验方法

| 场景 | 检验方法 |
|------|---------|
| 两种方法准确率对比 | McNemar's test（配对二分类） |
| 多种方法整体对比 | Friedman test + Nemenyi post-hoc |
| 效果量 | Cohen's d / Cliff's delta |
| 置信区间 | Bootstrap 95% CI |

### 7.2 结果报告规范

- 所有核心指标报告 **均值 ± 标准差**
- 主实验重复 **3 次**（不同随机种子），报告平均值
- 显著性检验标注：* p < 0.05, ** p < 0.01, *** p < 0.001
- 消融实验使用 **表格 + 柱状图** 双重呈现
