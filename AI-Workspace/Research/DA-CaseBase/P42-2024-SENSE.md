## 核心问题

### 解决什么问题

**开源LLM与闭源LLM在Text-to-SQL任务上的巨大性能差距**。

具体而言：
1. **性能差距大**：CodeLLaMA-13B-Instruct 在 BIRD 基准上比 GPT-4 低 30% 执行准确率
2. **数据稀缺**：高质量 Text-to-SQL 标注数据依赖人工专家，成本高昂
3. **跨域泛化难**：现有数据集域覆盖有限，模型难以泛化到新数据库/领域
4. **错误学习缺失**：强模型难以生成有价值的错误样本，模型无法从错误中学习

### 为什么重要

- **隐私与安全**：闭源模型（GPT-4）涉及数据外传，在政策敏感、隐私敏感场景不可用
- **成本问题**：闭源 API 调用成本高
- **开放性**：推动开源社区发展，使 Text-to-SQL 能力民主化
- **实际需求**：让非技术人员通过自然语言查询数据库是重要的应用场景

## 技术方法

### 核心思路：Strong + Weak 双阶段数据合成

SENSE 的核心创新是**融合强模型和弱模型生成的合成数据**，分两个阶段训练：

```
阶段1: SFT（监督微调）← Strong Data（GPT-4生成）→ 增强域多样性
阶段2: DPO（偏好学习）← Weak Data（小模型生成）→ 从错误中学习
```

### 阶段一：Strong Data → 监督微调 (SFT)

**目标**：利用 GPT-4 合成高质量、高多样性的 Text-to-SQL 数据

**关键设计**：
1. **域多样性控制**：Prompt 中明确排除过度代表的领域（教育、医疗、旅行、娱乐）
2. **难度控制**：分 Easy/Medium/Hard 三级，控制生成数据的复杂度
3. **模式引导**：GPT-4 同时生成数据库 Schema + NL 问题 + SQL 答案

**数据统计**（Table 1）：

| 数据集 | 样本数 | 数据库数 | 样本/库比 | 平均Token数 | 平均JOIN数 |
|--------|--------|----------|-----------|-------------|------------|
| Spider | 7,000 | 140 | 50.0 | 37.3 | 0.54 |
| BIRD | 9,428 | 69 | 136.6 | 64.0 | 1.02 |
| **Synthetic** | **8,216** | **503** | **16.3** | **60.3** | **1.13** |

关键观察：合成数据的**样本/库比仅16.3**（远低于Spider的50.0和BIRD的136.6），说明域多样性显著更高。平均JOIN数1.13也最高，SQL复杂度更大。

**SFT损失函数**：标准自回归 log-likelihood loss

$$\mathcal{L}_{SFT} = \mathbb{E}_{(x,y) \sim D_s} \left[ \sum_{t=1}^{T} \log p_\theta(y_t | y_{1:t-1}, x) \right]$$

### 阶段二：Weak Data → 偏好学习 (DPO)

**目标**：利用弱模型生成错误SQL，通过偏好学习让模型从错误中学习

**关键流程**：
1. 用弱模型（DeepSeek-Coder-1.3B）对训练集生成 SQL
2. 用 SQL 执行器验证每条生成的 SQL
3. 执行结果与 ground truth 匹配 → **正样本 $y_w$**
4. 执行结果不匹配 → **负样本 $y_l$**
5. 构建偏好数据集 $D_w = \{(x, y_w, y_l)\}$
6. 用 DPO 算法训练

**DPO 目标函数**：

$$\mathcal{L}_{DPO} = \mathbb{E}_{(x,y_w,y_l) \sim D_w} \left[ \log \sigma \left( \beta \log \frac{p_\theta(y_w|x)}{p_{ref}(y_w|x)} - \beta \log \frac{p_\theta(y_l|x)}{p_{ref}(y_l|x)} \right) \right]$$

**为什么用弱模型而非强模型生成错误样本？**
- 强模型（GPT-4）过度对齐（aligned），很难产生有价值的错误
- 弱模型天然会犯错，且错误模式更接近开源模型的典型错误
- 弱模型错误可通过执行器自动验证，无需人工标注

### 架构总结

```
输入数据源:
├── Human Annotated Data: Spider + BIRD 训练集
├── Strong Data: GPT-4 合成的 8216 条多样化数据
└── Weak Data: DeepSeek-Coder-1.3B 生成的偏好对

训练流程:
├── Phase 1: SFT on CodeLLaMA (Human + Strong Data)
│   ├── Optimizer: AdamW, lr=2e-5
│   ├── Scheduler: Cosine warmup
│   └── Epochs: 3
└── Phase 2: DPO on SFT模型 (Weak Data)
    ├── Optimizer: Adam, lr=2e-6
    ├── β = 0.2
    └── Hardware: 8×A100 GPUs

输出模型:
├── SENSE-7B (CodeLLaMA-7B → SFT → DPO)
└── SENSE-13B (CodeLLaMA-13B → SFT → DPO)
```

## 实验结果

### 数据集

| 数据集 | 类型 | 特点 |
|--------|------|------|
| **Spider** | 通用基准 | 7,000训练 + 1,034开发，200个数据库，138个领域 |
| **BIRD** | 挑战基准 | 95个大型真实数据库，33.4GB数据，37个领域，需外部知识推理 |
| **SYN** | 鲁棒性 | 用同义词替换问题标签 |
| **DK** | 鲁棒性 | 需要领域知识推理 |
| **REALISTIC** | 鲁棒性 | 更接近真实场景的 schema 替换 |

### 评估指标

- **EX (Execution Accuracy)**：SQL执行结果是否与gold SQL一致
- **TS (Test Suite)**：通过数据库增强创建的多组测试是否全部通过（更严格）

### 主要结果

#### Spider 基准（Table 2）

| 模型 | Dev-EX | Dev-TS |
|------|--------|--------|
| GPT-4 | 74.5 | - |
| DAIL-SQL + GPT-4 | 83.5 | 76.2 |
| Fine-tuned SQL-PaLM | 78.2 | - |
| CodeLLaMA-13B-Instruct | 62.3 | 54.5 |
| **SENSE-7B** | **83.2** | **81.7** |
| **SENSE-13B** | **84.1** | **83.5** |

**SENSE-13B 在 Spider 上超越了所有基于 GPT-4 的方法**，比 DAIL-SQL+GPT-4 高 0.6%（EX）和 7.3%（TS）。

#### BIRD 基准（Table 2）

| 模型 | Dev-EX | Test-EX |
|------|--------|---------|
| DAIL-SQL + GPT-4 | 54.8 | 57.4 |
| Fine-tuned SQL-PaLM | - | - |
| CodeLLaMA-13B-Instruct | 24.7 | - |
| **SENSE-7B** | **51.8** | **59.3** |
| **SENSE-13B** | **55.5** | **63.4** |

**SENSE-13B 在 BIRD Test 上超过 DAIL-SQL+GPT-4 达 5.98%**，展现了在挑战性场景下的优势。

#### 鲁棒性基准（Table 3）

| 模型 | SYN-EX | REALISTIC-EX | DK-EX | 平均 |
|------|--------|--------------|-------|------|
| RESDSQL-3B+NatSQL | 76.9 | 81.9 | 66.0 | 72.3 |
| Fine-tuned SQL-PaLM | 70.9 | 77.4 | 67.5 | 71.1 |

SENSE-13B 平均超过 RESDSQL-3B **5.4%**，尤其在 DK（领域知识）上优势明显（+14.2%），说明合成数据有效利用了基座模型的领域知识。

#### SQL难度分级分析（Table 4，Spider Dev TS）

| 难度 | DAIL-SQL+GPT-4 | Fine-tuned SQL-PaLM | **SENSE-13B** |
|------|----------------|---------------------|---------------|

| Medium | 81.8 | 85.2 | **88.6** |
| Hard | 67.2 | 68.4 | **75.9** |
| Extra Hard | 50.6 | 48.2 | **60.3** |

**越难的SQL，SENSE优势越大**：Easy +1.6%, Medium +3.4%, Hard +7.5%, Extra Hard +9.7%。

### 消融实验（Table 5）

| 配置 | Spider Dev-EX | Spider Dev-TS | BIRD Dev |
|------|---------------|---------------|----------|
| SENSE-13B (完整) | 84.0 | 83.5 | 55.5 |
| w/o Weak Data | 83.9 | 82.9 | 52.9 |
| w/o Strong Data | 61.7 | 53.5 | 22.9 |
| Base (CodeLLaMA-13B) | 61.7 | 53.5 | 22.9 |

**关键发现**：
- **Strong Data 是基础**：去掉后性能回到基座水平（-22.3% Spider EX, -32.6% BIRD）
- **Weak Data 锦上添花**：对 BIRD 提升显著（+2.6%），对 Spider 提升有限（+0.1% EX）
- **Weak Data 对复杂SQL更有价值**：BIRD 比 Spider SQL更复杂，所以提升更大

### 迁移性验证

在 Qwen 系列（同源模型）上验证：Qwen-7B 为基座 + Qwen-1.8B 为弱模型 → SENSE_Q-7B

| 模型 | Spider Dev-EX | BIRD Dev |
|------|---------------|----------|
| Qwen-7B (base) | 63.1 (w/ strong) | 26.1 |
| SENSE_Q-7B | **84.2** | **52.5** |

**方法在不同模型家族间均有效**，证明了方法的通用性。

### 通用能力保持（Table 6）

| 模型 | MMLU | ARC | GSM8K | HumanEval | 平均 |

## 技术亮点与创新点

### 1. 弱强模型互补的数据合成范式
- **Strong Data（GPT-4）**→ 提供域多样性和数据质量
- **Weak Data（小模型）**→ 提供有价值的错误样本用于偏好学习
- 两者互补，解决了"强模型难以生成错误"的痛点

### 2. 执行器对齐的偏好学习
- 不依赖人工标注正/负样本
- 用 SQL 执行器自动验证，实现**自动化**的偏好数据构建
- 这是一种**弱监督**方法，成本极低

### 3. 域多样性的系统控制
- Prompt 中显式排除过度代表的领域
- 控制生成数据的难度分布
- 实现了长尾分布的域覆盖（503个数据库 vs Spider的140个）

### 4. 关键Token级别的偏好学习效果（Figure 7）
- 论文通过案例分析展示了 DPO 在**关键Token**上的效果：
  - 减少列名选择错误（Wrong Column）
  - 减少表名选择错误（Wrong Table）
  - 修正算子错误（Wrong Operator: max→min）

## 局限性

1. **规模限制**：未在更大模型（如 70B）上验证
2. **任务单一**：仅验证了 Text-to-SQL，未扩展到代码生成、数学推理等类似任务
3. **时效性**：基于 2024年8月的模型和数据，未使用更新的基座模型
4. **弱模型选择**：未系统研究不同弱模型对偏好学习效果的影响

## 与本研究（DA-CaseBase）的关系

### 可借鉴的技术点

| 技术点 | SENSE 的做法 | 对 DA-CaseBase 的启发 |
|--------|-------------|----------------------|
| **弱强数据合成** | 强模型生成多样数据 + 弱模型生成偏好对 | 可用于合成 Data Agent 的训练数据：强模型生成高质量案例，弱模型生成错误案例用于偏好学习 |
| **执行器验证** | SQL 执行器自动判断正/负样本 | Data Agent 可用代码执行器（Python executor）验证生成代码的正确性 |
| **域多样性控制** | Prompt 控制领域分布，避免过度代表 | 构建 CaseBase 时需要覆盖多种数据分析场景，避免偏向某些领域 |
| **DPO 偏好学习** | 从错误中学习，对齐执行器 | 可用于训练 Data Agent 从错误的数据分析代码中学习 |
| **跨模型迁移性** | 方法在 CodeLLaMA 和 Qwen 系列都有效 | 方法论可推广到不同基座模型 |

### 本研究的切入点

1. **从 Text-to-SQL 扩展到 Text-to-Analysis**：
   - SENSE 专注于 SQL 生成，我们可以将类似的弱强数据合成方法扩展到更广泛的数据分析任务（Python 代码、可视化、统计分析等）

2. **案例库构建的数据合成**：
   - 利用 SENSE 的域多样性控制思路，系统地为 DA-CaseBase 生成覆盖多领域的数据分析案例
   - 用执行器验证确保案例的可执行性和正确性

3. **偏好学习用于 Agent 对齐**：
   - 将 DPO 方法应用于 Data Agent 的对齐训练
   - 弱模型生成的错误分析路径可作为负样本

4. **长尾分布覆盖**：
   - SENSE 的合成数据呈长尾分布（Figure 4），这与现实中数据分析需求的分布一致
   - DA-CaseBase 需要类似的长尾覆盖策略
