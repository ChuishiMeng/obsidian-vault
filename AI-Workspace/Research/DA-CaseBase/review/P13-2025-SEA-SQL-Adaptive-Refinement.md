## 一、研究背景

### 1.1 问题场景：LLM 生成的 SQL 经常出错，但修正很麻烦

**背景**：NL2SQL 系统用 LLM 生成 SQL，但生成的不一定对。

**核心问题**：SQL 出错了怎么办？

**错误类型**：

```
错误1：语法错误
- SQL 写错了，数据库直接报错
- 例如：SELECT * FORM users（FORM 应该是 FROM）

错误2：执行错误
- SQL 语法对，但数据库不支持
- 例如：用了不存在的函数

错误3：语义错误
- SQL 能执行，但结果不是用户想要的
- 例如：用户问"销售额"，SQL 算了"订单数"
```

---

### 1.2 修正 SQL 的困境

**现有修正方法的问题**：

| 方法 | 流程 | 问题 |
|------|------|------|
| **人工修正** | DBA 手动改 SQL | 无法规模化，成本高 |
| **重新生成** | 让 LLM 再生成一次 | 可能生成同样的错误 |
| **规则修正** | 预设修正规则 | 覆盖有限，难以泛化 |

**通俗理解**：

```
场景：LLM 生成的 SQL 执行失败

方法1：让 LLM 重新生成
问题：LLM 可能"固执"地生成同样的错误

方法2：人工分析错误原因，写修正规则
问题：错误类型太多，规则写不完

方法3：找 DBA 手动改
问题：DBA 不够用
```

---

### 1.3 为什么修正这么难？

**核心挑战**：

| 挑战 | 说明 |
|------|------|
| **错误类型多样** | 语法错误、语义错误、Schema 错误... |
| **原因不明** | 同样的错误信息，原因可能不同 |
| **重新生成无效** | LLM 不知道自己错了，可能重复错误 |

**示例**：

```
错误信息：ERROR: column "prodcut_name" does not exist

可能原因：
1. 列名拼写错误（应该是 product_name）
2. 表名错误（应该在 products 表）
3. 完全用错列

不同原因需要不同的修正策略！
```

---

### 1.4 理想的解决方案

**理想状态**：

```
系统发现 SQL 执行失败
    ↓
自动分析错误类型
    ↓
选择合适的修正策略
    ↓
修正 SQL
    ↓
再次执行，直到成功
```

**核心问题**：如何让系统"智能地"选择修正策略？

---

### 1.5 研究问题

**本文要解决的问题**：

1. 如何自动分析 SQL 错误类型？
2. 如何针对不同错误选择不同修正策略？
3. 如何降低修正成本（用小模型达到大模型效果）？

**核心贡献**：SEA-SQL —— 自适应修正策略 + Bias Eliminator

**目标**：

### 2.2 核心挑战

|------|------|
| **C1: 错误分类** | 如何识别 SQL 错误类型？|
| **C2: 自适应修正** | 不同错误类型需要不同修正策略 |
| **C3: 成本控制** | 如何平衡修正效果与成本？|

### 3.2 Bias Eliminator 设计

**双粒度信息融合**：

```python
class BiasEliminator:
    def analyze_error(self, pre_sql, execution_info):
        # 粗粒度信息: SQL 结构分析
        coarse_info = analyze_sql_structure(pre_sql)
        
        # 细粒度信息: 执行错误分析
        fine_info = analyze_execution_error(execution_info)
        
        # 融合分析
        error_type = classify_error(coarse_info, fine_info)
        fix_suggestion = generate_fix(error_type)
        
        return error_type, fix_suggestion
```

### 3.3 自适应修正策略

```python
def adaptive_refinement(pre_sql, execution_info):
    error_type, fix_suggestion = bias_eliminator.analyze_error(
        pre_sql, execution_info
    )
    
    # 根据错误类型选择修正策略
    if error_type == "SYNTAX_ERROR":
        return fix_syntax(pre_sql, fix_suggestion)
    elif error_type == "SCHEMA_ERROR":
        return fix_schema(pre_sql, fix_suggestion)
    elif error_type == "SEMANTIC_ERROR":
        return fix_semantic(pre_sql, fix_suggestion)
    else:
        return regenerate_sql(pre_sql)
```

### 3.4 Semantic-Enhanced Schema

**Schema 增强**：

```python
def enhance_schema(schema, question):
    enhanced = {}
    for table in schema.tables:
        enhanced[table.name] = {
            'description': get_table_description(table),
            'columns': {}
        }
        for column in table.columns:
            enhanced[table.name]['columns'][column.name] = {
                'description': get_column_description(column),
                'business_terms': extract_business_terms(question, column)
            }
    return enhanced
```
