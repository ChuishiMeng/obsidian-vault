## 一、研究背景

### 1.1 应用场景

**开源模型的困境**：

| 模型类型 | Spider 准确率 | 问题 |
|---------|--------------|------|
| GPT-4 | 86%+ | 闭源，成本高，数据隐私问题 |
| CodeLlama-13B | ~65% | 开源，但效果差距大 |
| DeepSeek-Coder | ~70% | 开源，仍有差距 |

**企业需求**：

| 需求 | 说明 |
|------|------|
| **数据隐私** | 不能将敏感数据发送给闭源模型 |
| **成本控制** | API 调用成本不可控 |
| **可定制** | 需要针对企业场景优化 |

### 1.2 现有方法的局限

**现有数据合成方法的问题**：

| 方法 | 问题 |
|------|------|
| **AnnotatedTables** | 直接 prompt 扩展，领域覆盖有限 |
| **MultiSQL** | SQL 结构多样性差 |
| **SENSE** | 新领域生成，执行失败率高 14% |

**核心挑战**：

| 挑战 | 说明 |
|------|------|
| **可靠性** | 生成的 SQL 执行失败率高 |
| **多样性** | SQL 结构单一，泛化能力差 |
| **语义对齐** | 问题和 SQL 语义不匹配 |

## 二、问题定义

### 2.1 问题形式化

**输入**：
- 种子数据集 D_seed = {(q₁, SQL₁), ...}
- 数据库 Schema S

**输出**：
- 合成数据集 D_synth（大规模、高质量、多样化）

**目标**：
- 执行成功率 > 95%
- SQL 结构多样性覆盖主流模式
- 问题-SQL 语义对齐

### 2.2 核心挑战

| 挑战 | 说明 |
|------|------|
| **C1: 可靠性** | 如何保证生成的 SQL 可执行？ |
| **C2: 多样性** | 如何覆盖更多 SQL 结构？ |
| **C3: 语义对齐** | 如何保证问题和 SQL 语义一致？ |
| **C4: 领域扩展** | 如何扩展到新领域？ |

**方法 2：SQL-to-Question 逆向翻译**

```
原始: SQL → Question（正向）
    问题: 语义不匹配

SQLForge: SQL → Question → 验证 → Question（逆向）
    保证语义对齐
```

```python
def reverse_translation(sql, llm):
    # 1. 从 SQL 生成问题
    prompt = f"""
    根据以下 SQL 查询，生成一个自然语言问题：
    SQL: {sql}
    """
    question = llm.generate(prompt)
    
    # 2. 验证语义对齐
    # 如果问题和 SQL 语义一致，则保留
    return question
```

```python
def domain_exploration(base_sql, new_domain):
    # 1. 识别领域相关实体
    entities = extract_entities(base_sql)
    
    # 2. 替换为新领域实体
    new_sql = replace_entities(base_sql, entities, new_domain.entities)
    
    # 3. 验证新 SQL
    if validate_sql(new_sql, new_domain.schema):
        return new_sql
    return None
```

## 六、对 DA-CaseBase 的启示

### 6.1 适用性分析

| 场景 | 适用性 | 原因 |
|------|--------|------|
| **模型 Fine-tuning** | ⭐ | 本研究不采用模型训练路线 |
| **数据合成** | ⭐⭐ | 可借鉴合成方法，但案例库依赖真实查询 |
| **SQL 验证** | ⭐⭐⭐ | 逆向翻译验证可借鉴 |

**核心原因**：SQLForge 的核心技术路线（数据合成 + Fine-tuning）与本研究（检索 + 生成）不匹配。

### 6.2 可借鉴的技术

| 技术 | 应用方式 |
|------|---------|
| **SQL-to-Question 逆向翻译** | 验证案例库中 SQL 的正确性 |
| **执行验证** | 入库前验证 SQL 可执行 |
| **模板扩充** | 从少量案例扩展更多变体 |

### 6.3 不适用的部分

| 功能 | 原因 |
|------|------|
| **模型 Fine-tuning** | 本研究采用检索 + 生成架构，不训练模型 |
| **大规模数据合成** | 案例库依赖真实用户查询，不宜完全依赖合成数据 |

### 6.4 与本研究的关系

```
SQLForge: 合成数据 → Fine-tune 模型

DA-CaseBase: 真实查询 → 检索 + 生成

借鉴点: SQL 验证方法（逆向翻译 + 执行验证）
不适用: 模型训练
```

## 八、研究价值总结

### 8.1 核心贡献

1. **可靠性增强**：SQL-to-Question 逆向翻译
2. **多样性增强**：模板扩充 + 领域探索
3. **开源模型 SOTA**：Spider 85.7%, BIRD 59.8%

### 8.2 与本研究的关系

**相关性低的原因**：
- 核心技术路线（数据合成 + Fine-tuning）与本研究不匹配
- 本研究采用「检索 + 生成」架构，不训练模型

**有限借鉴**：
- SQL 验证方法（逆向翻译 + 执行验证）可用于案例入库前验证
