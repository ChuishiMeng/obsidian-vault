## 一、研究背景

### 1.1 问题场景：NL2SQL 系统会"犯错"

**背景**：NL2SQL（自然语言转 SQL）系统已经广泛应用，用户用自然语言提问，系统自动生成 SQL 查询数据库。

**核心问题**：NL2SQL 系统生成的 SQL 并不总是正确的。

**错误类型**：

| 错误类型 | 说明 | 能否被发现？|
|---------|------|-----------|
| **语法错误** | SQL 写错了，无法执行 | ✅ 容易发现（执行报错）|
| **语义错误** | SQL 语法正确，但不符合用户意图 | ❌ 很难发现（能执行，但结果错误）|

**通俗理解**：

- **语法错误**：SQL 写得"不通"，数据库直接报错
- **语义错误**：SQL 写得"不对"，数据库能执行，但结果不是用户想要的

---

### 1.2 语义错误的隐蔽性与危害

**示例**：

```
用户问题: "查询销售额最高的前10个产品"

NL2SQL 系统生成的 SQL:
SELECT product_name, COUNT(*) as cnt
FROM orders
GROUP BY product_name
ORDER BY cnt DESC
LIMIT 10

执行结果: 返回了10个产品，系统认为"成功"

实际错误: 
- 用户问的是"销售额"，SQL 用的是 COUNT（计数）
- 正确应该是 SUM(amount)（求和）
- 结果：返回了"订单数最多的产品"，而非"销售额最高的产品"
```

**危害**：

| 危害 | 说明 |
|------|------|
| **决策错误** | 用户拿到错误数据，做出错误决策 |
| **难以发现** | SQL 能执行成功，用户可能不知道结果错了 |
| **隐蔽性高** | 只有懂 SQL 的人才能发现语义错误 |

---

### 1.3 现有评估方法的盲区

**现有基准的评估方式**：

```
Spider / BIRD 评估:
1. NL2SQL 生成 SQL
2. 执行 SQL
3. 对比执行结果与标准答案
4. 结果相同 → 正确；结果不同 → 错误
```

**盲区**：

| 情况 | 执行结果 | 用户意图 | 现有评估判断 | 实际判断 |
|------|---------|---------|-------------|---------|
| 情况1 | 正确 | 符合 | ✅ 正确 | ✅ 正确 |
| 情况2 | 错误 | - | ❌ 错误 | ❌ 错误 |
| **情况3** | 正确 | **不符合** | ✅ 正确 | ❌ **语义错误** |

**关键洞察**：

> 现有基准只能检测"执行是否成功"，无法检测"语义是否正确"
> 
> 如果 SQL 碰巧返回了正确结果（或接近正确），但语义不符合用户意图，现有评估会漏判！

## 三、核心方法

### 3.1 两级错误分类体系

**9 大类 + 31 子类**：

```
语义错误分类
├── 1. 属性相关错误 (Attribute-related)
│   ├── 1.1 属性缺失
│   ├── 1.2 属性错误
│   └── 1.3 属性冗余
├── 2. 条件相关错误 (Condition-related)
│   ├── 2.1 条件缺失
│   ├── 2.2 条件错误
│   └── 2.3 条件冗余
├── 3. 连接相关错误 (Join-related)
│   ├── 3.1 Join 缺失
│   ├── 3.2 Join 条件错误
│   └── 3.3 Join 冗余
├── 4. 分组相关错误 (Group-related)
├── 5. 聚合相关错误 (Aggregation-related)
├── 6. 排序相关错误 (Order-related)
├── 7. 嵌套相关错误 (Nesting-related)
├── 8. 集合相关错误 (Set-related)
└── 9. 其他错误 (Others)
```

### 3.2 基准数据集

**数据规模**：

| 指标 | 值 |
|------|-----|
| **总实例数** | 2,018 |
| **错误类型** | 9 大类 + 31 子类 |
| **数据来源** | Spider + BIRD 错误分析 |

**数据标注流程**：

```
1. 收集 NL2SQL 模型生成的错误 SQL
    ↓
2. 专家标注错误类型和位置
    ↓
3. 多人交叉验证
    ↓
4. 形成最终数据集
```

### 3.3 评估指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **Overall Accuracy** | 整体检测准确率 | 核心指标 |
| **Type-specific Accuracy** | 各类错误检测准确率 | 细粒度评估 |
| **Detection Rate** | 错误检出率 | 召回率 |
| **False Positive Rate** | 误报率 | 精确率相关 |

## 五、实验结果与分析

### 5.1 主要结果

**错误检出率**：

| 方法 | Overall Accuracy | Detection Rate |
|------|-----------------|----------------|
| 执行验证 | - | 0%（无法检测语义错误）|
| LLM 直接判断 | ~60% | 中等 |
| **GPT-4 + NL2SQL-BUGs** | **~75%** | 较高 |

### 5.2 错误类型分析

| 错误类型 | 检出率 | 难度 |
|---------|--------|------|
| 属性相关错误 | 80%+ | 较易 |
| 条件相关错误 | 70%+ | 中等 |
| Join 相关错误 | 60%+ | 较难 |
| 聚合相关错误 | 50%+ | 难 |

### 5.3 发现：现有基准中的错误

**Spider 和 BIRD 中的语义错误**：

| 基准 | 错误数量 | 占比 |
|------|---------|------|
| Spider | 16 | 1.55% |
| BIRD | 更多 | 更高 |

**关键发现**：

> 现有基准本身就包含语义错误！

| 技术 | 应用方式 |

**案例入库验证流程**：

```
新查询 + SQL

2. 语义错误检测（使用 NL2SQL-BUGs 分类体系）
    ↓
3. 通过 → 入库
   未通过 → 人工审核
```
