## 一、研究背景

### 1.1 问题场景：企业里用 NL2SQL，效果远不如论文里说的好

**背景**：企业有大量数据，业务人员需要频繁查询数据做分析。

**核心问题**：学术基准测试效果好（Spider 达到 90%+），但企业实际部署效果差。

**通俗理解**：

```
论文里：Spider 基准测试，准确率 90%+
企业里：同样的模型，准确率可能只有 50-60%

为什么差距这么大？
```

---

### 1.2 学术场景 vs 企业场景的巨大差距

**Schema 复杂度差距**：

| 维度 | 学术基准（Spider） | 企业现实 |
|------|------------------|---------|
| **表数量** | 几十张 | 数千张 |
| **字段命名** | `customer_name`, `order_date` | `consumer_1`, `dt_2024` |
| **业务术语** | "销售额"、"订单数" | "GMV"、"PV"、"UV"（企业黑话）|

**示例**：

```
学术场景：
- 数据库：10 张表，字段名清晰
- 查询："查询所有用户的姓名和年龄"
- SQL：SELECT name, age FROM users
- 难度：简单

企业场景：
- 数据库：500 张表，字段名是缩写
- 查询："查询上季度华东区 GMV"
- SQL：需要关联销售表、地区表、时间维表...
- 难度：复杂得多
```

---

### 1.3 企业特有的挑战

**挑战一：Schema 太大，字段名看不懂**

```
企业数据库字段名：
- consumer_1, consumer_2（这是什么？）
- amt_1, amt_2（金额？哪个金额？）
- dt, tm, ts（时间？哪个时间？）

业务人员根本不知道这些字段是什么意思
```

**挑战二：企业有"黑话"**

```
业务术语：
- "GMV" = 商品交易总额
- "PV" = 页面浏览量
- "UV" = 独立访客数

这些术语 LLM 训练数据里可能没有
```

**挑战三：需要上下文**

```
查询："本月的销售额"
系统需要知道：
- "本月" = 当前月份（不是固定的某个月）
- 用户角色 = 销售经理（可能只能看自己区域的数据）
```

---

### 1.4 学术基准的盲区

**Spider/BIRD 的局限**：

| 维度 | 学术基准 | 企业现实 | 差距 |
|------|---------|---------|------|
| Schema 规模 | 几十张表 | 数千张表 | 10-100x |
| 字段语义 | 清晰 | 模糊/缩写 | 无法理解 |
| 业务术语 | 通用词汇 | 企业黑话 | LLM 不认识 |
| 上下文 | 单次查询 | 时间/角色/对话上下文 | 需要上下文感知 |

**核心洞察**：

> 学术基准"过于干净"，无法反映企业的真实复杂性

---

### 1.5 另一个关键问题：系统需要持续学习

**观察**：企业业务在变化

```
情况1：新业务线 → 新的表结构
情况2：术语变化 → "GMV"改成"交易额"
情况3：用户反馈 → 之前生成的 SQL 有错
```

**核心问题**：

> NL2SQL 系统不是"部署一次就完事"，需要持续学习优化

**传统方法的问题**：

| 方法 | 问题 |
|------|------|
| 部署后不更新 | 效果会随业务变化而下降 |
| 重新训练模型 | 成本高，周期长 |

---

### 1.6 研究问题

**本文要解决的问题**：

1. 如何设计企业级 NL2SQL 系统架构？
2. 如何让系统持续学习、自我进化？
3. 如何评估企业场景的效果？

**核心贡献**：RubikSQL —— 终身学习框架 + 企业级架构 + RubikBench 评估基准
2. 业务术语不理解
3. 缺乏上下文感知
4. 无法处理隐式意图
```

## 二、问题定义

### 2.1 问题形式化

**输入**：
- 企业数据库 Schema S（数千张表）
- 历史查询日志 L
- 用户新问题 q
- 上下文 C（时间、用户角色等）

**输出**：
- 正确的 SQL

**目标**：
- 部署在企业环境
- 持续学习优化
- 处理复杂企业场景

### 2.2 四大核心挑战

| 挑战 | 说明 | 学术基准是否覆盖 |
|------|------|-----------------|
| **C1: 隐式意图** | 用户意图不明确 | ❌ |
| **C2: 私有领域知识** | 企业特定术语 | ❌ |
| **C3: 宽表 Schema** | 数千列的表 | ❌ |
| **C4: 上下文敏感性** | 时间、角色上下文 | ❌ |

### 2.3 终身学习视角

**核心思想**：

> NL2SQL 在企业中不是「一次性任务」，而是「终身学习任务」

```
传统视角: 训练模型 → 部署 → 使用

终身学习视角: 部署 → 使用 → 反馈 → 更新 → 使用 → ...
```

### 3.3 Stage 1: Database Context Engineering

**Schema 增强**：

```python
def enhance_schema(schema, domain_knowledge):
    enhanced = {}
    for table in schema.tables:
        enhanced[table.name] = {
            'description': domain_knowledge.get_table_description(table.name),
            'columns': {}
        }
        for column in table.columns:
            enhanced[table.name]['columns'][column.name] = {
                'description': domain_knowledge.get_column_description(column.name),
                'business_terms': domain_knowledge.get_business_terms(column.name),
                'example_values': get_sample_values(column)
            }
    return enhanced
```

**示例**：

```
原始 Schema:
  表: orders
  列: consumer_1, consumer_2, amt, dt

增强后 Schema:
  表: orders (订单表)
  列:
    consumer_1 → 下单客户ID (业务术语: 买家)
    consumer_2 → 收货客户ID (业务术语: 收货人)
    amt → 订单金额
    dt → 下单日期
```

### 3.4 Stage 2: User Query Augmentation

**上下文注入**：

```python
def augment_query(question, context):
    augmented = question
    
    # 时间上下文
    if context.time_reference:
        # "本月销售额" → "2025年3月销售额"
        augmented = resolve_time(question, context.current_time)
    
    # 用户角色上下文
    if context.user_role:
        # 根据角色调整权限
        augmented = apply_role_filter(augmented, context.user_role)
    
    return augmented
```

### 3.6 Stage 4: Knowledge Distillation

**知识蒸馏流程**：

```python
def knowledge_distillation(query, retrieved_docs, llm):
    # 1. 构建 Prompt
    prompt = build_prompt(query, retrieved_docs)
    
    # 2. LLM 生成 SQL
    sql = llm.generate(prompt)
    
    # 3. 执行验证
    success, result = execute_sql(sql)
    
    if not success:
        # 4. 自动修正
        sql = auto_correct(sql, result.error)
    
    return sql
```

## 六、对 DA-CaseBase 的深度启示

### 6.1 与本研究的高度相似性

| 维度 | RubikSQL | DA-CaseBase |
|------|----------|-------------|
| **场景** | 企业级 NL2SQL | ✅ 完全一致 |

### 6.2 终身学习机制对比

**思路一致**：双方都认为 NL2SQL 在企业中是「终身学习任务」，需要持续优化

**落地方式不同**：

| 维度 | RubikSQL | DA-CaseBase（本研究） |
|------|----------|----------------------|
| **知识更新** | 人工标注 + 知识库更新 | **自动沉淀机制** |
| **模型更新** | 定期 Fine-tune | 不训练模型，优化检索 |
| **反馈收集** | 显式用户反馈 | 隐式行为反馈 + 显式反馈 |
| **更新频率** | 批量更新 | **实时自动更新** |

**本研究创新点**：
- 自动沉淀：验证通过的查询自动入库，减少人工标注
- 实时更新：无需等待批量更新周期

### 6.2 可借鉴的技术

| 技术 | 应用方式 |
|------|---------|
| **四阶段工作流** | Database Context → Query Augmentation → KB Indexing → Distillation |
| **Schema 增强** | 添加业务描述、Join 路径、语义解释 |
| **上下文注入** | 时间上下文解析、角色权限过滤 |
| **终身学习机制** | 反馈 → 更新知识库 → 模型更新 |

### 6.4 与 CBR-to-SQL 的互补

```
CBR-to-SQL: 模板抽象 + 实体落地（处理个体查询）

### 6.5 本研究的关键差异

**终身学习落地方式对比**：

| 维度 | RubikSQL | DA-CaseBase |
|------|----------|-------------|
| **初始化** | 人工标注 Schema 增强 | 可从少量种子案例启动 |
| **知识更新** | 人工审核后更新 | **自动沉淀验证通过的查询** |
