## 一、研究背景

### 1.1 问题场景（一句话说清楚）

**核心问题**：两个写法不一样的 SQL，查出来的数据是否一样？怎么判断？

**通俗理解**：

想象你是一个"SQL 裁判"：
```
用户问："查一下2024年销售额前10的商品"

SQL A（标准答案）：
SELECT product_name FROM sales WHERE year=2024 ORDER BY amount DESC LIMIT 10

SQL B（AI生成的）：
SELECT product_name FROM sales WHERE year='2024' ORDER BY amount DESC LIMIT 10

问题：这两个 SQL 等价吗？
- 字符串不一样（2024 vs '2024'）
- 但查出来的数据可能完全一样
```

**传统"裁判"的问题**：

| 传统方法 | 怎么判断 | 问题 |
|---------|---------|------|
| **Execution Accuracy (EX)** | 执行两个 SQL，看结果是否一样 | 假阳性：两个错误 SQL 都返回空，误判为"等价" |
| **Exact Set Matching (ESM)** | 字符串完全匹配 | 假阴性：`WHERE age > 18` vs `WHERE age >= 19`，格式不同就判"不等价" |

**具体例子**：
```sql
-- 场景：数据库中没有符合条件的数据

-- SQL1（错误：应该是 age >= 18）
SELECT name FROM users WHERE age > 18

-- SQL2（正确）
SELECT name FROM users WHERE age >= 18

-- EX 的判断：两个都返回空结果 → 等价！❌（错误判断）
-- 实际上：语义不等价，只是碰巧数据为空
```

### 1.2 为什么这个问题重要？

| 维度 | 学术评估 | 企业应用 |
|------|---------|---------|
| 等价标准 | 严格语义等价 | 弱语义等价（业务等价） |
| 容错性 | 低（要求精确匹配） | 高（只要业务对即可） |
| 评估成本 | 人工标注 | 自动化优先 |
| 典型场景 | 基准测试 | Dataverse 数据平台 |

**企业场景举例**：
```
-- 用户问题："找出销售额前10的商品"

-- SQL A（参考）
SELECT product_name FROM sales ORDER BY amount DESC LIMIT 10

-- SQL B（生成）
SELECT product_name FROM sales ORDER BY amount DESC LIMIT 10 OFFSET 0

-- 企业判断：业务等价，应该判定为正确
-- 学术判断：严格等价，也是等价
```

### 1.3 核心挑战

| 挑战 | 描述 | 现有方案局限 |
|------|------|-------------|
| **歧义查询多解** | 同一问题多种 SQL 写法 | ESM 判定错误 |
| **假阳性** | 空结果匹配 | EX 误判为正确 |
| **假阴性** | 格式差异 | ESM 误判为错误 |
| **LLM 不一致** | 多次评估结果不同 | 评估稳定性差 |
| **弱等价判断** | 业务等价 vs 语义等价 | 缺乏明确标准 |

## 二、核心贡献

### 2.1 一句话说清楚

**用 LLM 理解 SQL 的"语义"，而不是只看表面写法。**

### 2.2 具体做了什么？

**思路**：以前的"裁判"要么太死板（ESM 只看字符串），要么太好骗（EX 看结果，空结果就认为等价）。这篇论文让 LLM 当"裁判"，理解 SQL 到底想干什么。

**两件核心事**：

**1. 让 LLM 理解 SQL 的"语义"**

```
传统方法：看字符串是否一样
  age > 18 vs age >= 19 → 字符串不同 → 不等价（错误）

LLM 方法：理解"语义"
  LLM 分析：age > 18 和 age >= 19 的边界值不同
  如果 age 是整数，语义等价
  如果 age 可能是小数，语义不等价
```

**2. 设计专门的验证步骤**

| 步骤 | 做什么 | 为什么 |
|------|--------|--------|
| **Miniature（小样本测试）** | 构造几个测试数据，看两个 SQL 结果是否一样 | 快速验证 |
| **Mull（多角度质疑）** | 让 LLM 从边界值、NULL、空表等角度质疑 | 避免遗漏 |
| **多次运行投票** | 跑 3 次，多数投票 | 提升稳定性 |

### 2.3 核心创新

| 创新 | 描述 | 效果 |
|------|------|------|
| **混合评估** | 规则处理简单情况（60%），LLM 处理复杂情况 | 快 + 准 |
| **Miniature 策略** | 构造边界值数据测试 | 发现隐藏问题 |
| **Mull 策略** | 多角度质疑等价性 | 提升准确性 |
| **弱等价定义** | 业务等价而非严格等价 | 适配企业场景 |

## 三、方法论

### 3.1 整体思路（类比理解）

**想象一个"SQL 裁判"的工作流程**：

```
拿到两个 SQL → 先看长得像不像（快速判断）
                    ↓ 不像
              让 LLM 深度分析
                    ↓
              构造测试数据验证
                    ↓
              多角度质疑
                    ↓
              判定是否等价
```

### 3.2 三步判断流程

**第一步：快速路径（处理 60% 简单情况）**

```python
# 如果两个 SQL 标准化后完全一样
if normalize(sql1) == normalize(sql2):
    return "等价"  # 直接判定，不用 LLM
```

**第二步：LLM 语义分析（处理复杂情况）**

```
让 LLM 理解两个 SQL：
1. 它们想查什么数据？
2. 用了什么条件？
3. 关联了哪些表？
4. 边界值是什么？
```

**第三步：Miniature + Mull 验证**

```
Miniature（小样本测试）：
  构造 age = 18, 19, 20 的测试数据
  执行两个 SQL，看结果是否一致

Mull（多角度质疑）：
  让 LLM 思考：
  - 如果 age 是小数怎么办？
  - 如果表是空的怎么办？
  - 如果有 NULL 怎么办？
```

### 3.3 具体例子

**场景**：判断 `age > 18` 和 `age >= 19` 是否等价

```
┌─────────────────────────────────────────────┐
│ 第一步：字符串匹配                           │
│   "age > 18" ≠ "age >= 19"                  │
│   → 不匹配，进入第二步                       │
├─────────────────────────────────────────────┤
│ 第二步：LLM 分析                             │
│   边界值分析：age=19 时，两个条件都满足      │
│   但 age=18.5 时，>18 满足，>=19 不满足      │
│   → 需要进一步验证                          │
├─────────────────────────────────────────────┤
│ 第三步：Miniature 测试                       │
│   测试数据：age = 18, 18.5, 19, 20          │
│   结果：age=18.5 时结果不同                  │
│   → 判定：不等价                            │
└─────────────────────────────────────────────┘

但如果知道 age 都是整数：
  age=18.5 不可能存在
  → 两个 SQL 在整数范围内等价
  → 弱语义等价
```

### 3.4 等价/不等价模式库

**系统预设的"等价模式"**（快速判断用）：

| 模式 | 示例 | 说明 |
|------|------|------|
| 表别名不同 | `FROM users u` vs `FROM users t` | 别名不影响语义 |
| 列顺序交换 | `SELECT a, b` vs `SELECT b, a` | 结果集相同（顺序可能不同） |
| 子查询转 JOIN | `IN (SELECT ...)` vs `JOIN` | 执行方式不同，结果相同 |
| 常量折叠 | `WHERE 1=1 AND age > 18` vs `WHERE age > 18` | 冗余条件可忽略 |

**系统预设的"不等价模式"**：

| 模式 | 示例 | 说明 |
|------|------|------|
| 过滤条件遗漏 | 缺少 WHERE 条件 | 结果集变大 |
| JOIN 类型错误 | `INNER JOIN` vs `LEFT JOIN` | NULL 处理不同 |
| 聚合函数不同 | `SUM()` vs `AVG()` | 完全不同的语义 |
| GROUP BY 缺失 | 没有 GROUP BY | 聚合逻辑错误 |

### 3.5 技术细节

| 组件 | 实现方式 | 备注 |
|------|---------|------|
| 预处理 | SQL 标准化（格式、别名、常量） | 统一表示 |
| 字符串匹配 | 精确匹配 + 模式匹配 | 快速路径，处理 60% |
| Miniature | 构造边界值测试数据 | 需要数据类型推断 |
| Mull | 多角度质疑 Prompt | 提升 LLM 准确性 |
| 多次运行 | n=3，多数投票 | 提升一致性 +8% |

## 四、实验结果

### 4.1 实验设置

**数据集**：

| 数据集 | 规模 | 特点 |
|--------|------|------|
| Dataverse | 手工标注 | 真实企业场景，业务等价判断 |
| Dev Set | 针对性设计 | 迭代优化，覆盖边缘情况 |
| Synthetic | 综合合成 | 覆盖多种等价/不等价模式 |

**Baseline**：

| Baseline | 描述 |
|----------|------|
| Execution Accuracy | 传统执行准确率 |
| Exact Set Matching | 传统字符串匹配 |
| GPT-3.5 | 直接 LLM 判断 |

### 4.2 主实验结果

| 数据集 | GPT-4 | GPT-3.5 | 规则方法 | EX | ESM |
|--------|-------|---------|---------|-----|-----|
| Dataverse | **92.3%** | 85.1% | 67.2% | 58.3% | 72.1% |
| Dev Set | **89.7%** | 82.4% | 64.8% | 55.7% | 69.8% |
| Synthetic | **88.1%** | 80.3% | 62.5% | 53.2% | 67.4% |

### 4.3 消融实验

| 变体 | Accuracy | 变化 | 说明 |
|------|----------|------|------|
| 完整方法 | 92.3% | - | Miniature + Mull + 多次运行 |
| w/o Miniature | 89.1% | ↓3.2% | 失去小样本测试能力 |
| w/o Mull | 87.2% | ↓5.1% | 失去多角度验证 |
| w/o 多次运行 | 84.3% | ↓8.0% | 一致性下降最多 |
| w/o 规则快速路径 | 88.5% | ↓3.8% | 效率下降 |

### 4.4 关键发现

1. **字符串匹配快速路径有效**：处理 60% 简单情况，大幅提升效率
2. **LLM 在复杂判断上表现优异**：GPT-4 达 92.3%，远超传统指标
3. **多次运行显著提升一致性**：+8%，是关键组件
4. **Mull 策略最重要**：移除后下降最多（5.1%）
