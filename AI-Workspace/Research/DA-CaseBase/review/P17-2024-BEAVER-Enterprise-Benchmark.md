## 一、研究背景

### 1.1 问题场景：学术基准"太干净"，企业场景"太复杂"

**背景**：Text-to-SQL 领域有一个尴尬的现象——论文发表的准确率越来越高，但企业实际部署效果却很差。

**核心问题**：学术基准测试与企业实际场景存在巨大差距。

**具体场景**：

```
论文发表：
- Spider 基准：GPT-4 达到 85%+ 准确率
- BIRD 基准：GPT-4o 达到 69% 准确率
- 结论："Text-to-SQL 已经解决了！"

企业部署：
- 同样的 GPT-4 模型
- 真实企业数据库
- 准确率可能只有 0-30%
- 现实："完全不可用"
```

**问题**：为什么差距这么大？

---

### 1.2 学术基准 vs 企业场景的对比

**Spider/BIRD 基准的特点**：

| 维度 | Spider | BIRD | 说明 |
|------|--------|------|------|
| **表数量** | 平均 4 张 | 平均 7 张 | 少，LLM 容易理解 |
| **字段命名** | `customer_name` | `student_name` | 清晰，LLM 能看懂 |
| **JOIN 数** | 平均 0.5 个 | 平均 1 个 | 简单，不需要复杂推理 |
| **数据来源** | Web 表格 | 公开数据 | LLM 训练数据里可能见过 |

**企业场景的特点**：

| 维度 | 企业实际 | 说明 |
|------|---------|------|
| **表数量** | 平均 **105 张** | 是 Spider 的 25 倍 |
| **JOIN 数** | 平均 **4.25 个** | 是 Spider 的 8 倍 |
| **字段命名** | `cons_1`, `amt_2`, `dt` | 缩写、看不懂 |
| **数据来源** | 企业私有数据 | LLM 从未见过 |

**示例**：

```
学术场景（Spider）：
- 数据库：student_record（学生记录）
- 字段：name, age, grade（清晰）
- 查询："查询所有学生的姓名"
- SQL：SELECT name FROM student_record
- 难度：简单

企业场景（BEAVER）：
- 数据库：FCLT_BUILDING（这是什么？）
- 字段：FCLT_BUILDING_KEY, BUILDING_NAME_LONG, date_built
- 查询："Which building has the most students?"
- SQL：需要 JOIN 3 张表，理解设施管理业务
- 难度：困难得多
```

### 1.3 BEAVER 基准的构建

**数据来源**：两个真实企业数据仓库

| 数据仓库 | 领域 | 表数 | 列数 | 查询数 |
|---------|------|------|------|--------|
| **DW** | 大学设施管理 | 99 | 1544 | 103 |
| **NW** | 计算基础设施 | 366 | 2708 | 100 |
| **合计** | - | 465 | 4252 | 203 |

**构建过程**：

```
1. 从企业查询日志收集真实 SQL
2. 四名研究生 + 两名 DBA 标注自然语言问题
3. 迭代审核直到所有问题通过
4. 额外标注列映射（Column Mapping）
```

**关键特点**：
- 真实企业数据库（不是合成数据）
- 真实用户查询（不是人工编造）
- 包含业务知识和隐式假设

## 二、核心发现

### 2.1 震撼的结果：LLM 准确率降至 0%

**实验设置**：

| 模型 | 设置 |
|------|------|
| GPT-4o | OpenAI 最强模型 |
| Llama-3.1-70B | 开源最强模型 |
| Llama-3.1-8B | 轻量级模型 |

**表检索结果**：

| 数据集 | Top-10 Recall | 完美召回率 |
|--------|--------------|-----------|
| Spider | 99.6% | 99.9% |
| BIRD | 97.5% | 95.1% |
| **BEAVER** | **48.7%** | **14.3%** |

> 📉 BEAVER 的表检索准确率比 Spider 低 50+ 个百分点

**SQL 生成结果**：

| 模型 | Spider | BIRD | BEAVER |
|------|--------|------|--------|
| GPT-4o | 69.5% | 30.9% | **0.0%** |
| Llama-70B | 60.3% | 25.8% | **0.0%** |
| Llama-8B | 51.1% | 13.8% | **0.0%** |

> 🚨 **所有模型在 BEAVER 上的执行准确率均为 0%**

**这意味着什么？**

```
学术基准：LLM 表现"很厉害"
企业场景：LLM 表现"完全失效"

差距：准确率下降 70-90 个百分点
```

---

### 2.2 为什么企业场景这么难？

**BEAVER 的错误分析**：

**表检索错误**：

| 错误类型 | 占比 | 说明 |
|---------|------|------|
| **信息不足** | 89.1% | 未检索到所有需要的表 |
| **缺失连接表** | 6.5% | 检索到的表无法直接 JOIN |
| **领域信息缺失** | 4.4% | 不理解领域特定术语 |

**示例**：

```
用户问题："What is the building name that accommodates the most students?"

检索结果：FCLT_BUILDING, STUDENT_DIRECTORY
问题：这两个表无法直接 JOIN！
缺失：FCLT_ROOMS（连接表）
```

**SQL 生成错误**：

| 错误类型 | 占比 | 说明 |
|---------|------|------|
| **列映射错误** | 59.1% | 将用户问题映射到错误的列 |
| **隐式假设遗漏** | 50.0% | 未推断出隐含条件 |
| **复杂查询失败** | 27.3% | 无法生成复杂语法 |
| **实例映射错误** | 22.7% | 将值映射到错误实例 |

**示例：列映射错误**：

```
用户问题："building street address"
错误映射：BUILDING_ADDRESS
正确映射：BUILDING_ADDRESS + ADDRESS_PURPOSE='STREET'

原因：不了解同一建筑有多种地址类型
```

**示例：隐式假设遗漏**：

```
用户问题："Provide information about virtual machine [id]"
金标准 SQL：WHERE instances.deleted = 0
预测 SQL：无此条件

原因：用户隐含假设只关心活跃实例
```

---

### 2.3 提供金标准信息后的效果

**实验**：如果告诉 LLM 正确的表和列，效果如何？

| 设置 | GPT-4o 准确率 |
|------|--------------|
| Baseline（自动检索表） | 0.0% |
| + 提供正确表 | 2.1% |
| + 提供正确列映射 | 4.2% |

> 📈 即使提供金标准信息，准确率也只有 4.2%

**这说明**：Schema Linking 只是问题的一部分，还有其他挑战：
- 复杂查询规划
- 领域知识理解
- 隐式假设推断

### 3.2 关键洞察

**洞察一：Schema 复杂度是核心挑战**

```
问题：案例检索时，如何处理 100+ 张表？

传统方法：语义相似度匹配
BEAVER 发现：语义相似度不够

原因：
1. 表数量太多，相似度计算困难
2. JOIN 关系需要理解，不只是语义匹配
3. 领域术语需要专门知识
```

**对 DA-CaseBase 的启示**：
- 案例检索不能只依赖语义相似度
- 需要 Schema 结构理解
- 需要 JOIN 关系推理

---

**洞察二：领域知识鸿沟**

```
问题：LLM 不认识企业特有术语

示例：
- "moira" = 邮件列表系统（MIT 特有）
- "FCLT" = 设施管理（大学特有）

LLM 训练数据里没有这些知识
```

**对 DA-CaseBase 的启示**：
- 案例库需要存储领域知识
- 检索时需要业务上下文
- 可能需要知识图谱辅助

---

**洞察三：隐式假设问题**

```
问题：用户问题有隐含条件

示例：
- "查询虚拟机信息" → 隐含只查活跃实例
- "本季度销售额" → 隐含当前季度、当前用户角色

这些假设不在问题文本中，但在业务上下文中
```

**对 DA-CaseBase 的启示**：
- 案例需要包含业务上下文
- 检索时需要用户角色信息
- 可能需要业务规则库
