## 一、研究背景

### 1.1 应用场景

**Text-to-SQL 的成本问题**：

| 成本维度 | 说明 |
|---------|------|
| **LLM API 成本** | 按 Token 计费 |
| **执行成本** | 云数据仓库按扫描数据量计费 |
| **时间成本** | 延迟影响用户体验 |

**核心问题**：

> 推理模型（如 o1）效果更好，但成本更高；非推理模型成本更低，但效果可能不足

### 1.2 研究问题

| 问题 | 说明 |
|------|------|
| **RQ1** | 推理模型 vs 非推理模型的效果差异？|
| **RQ2** | 成本差异有多大？|
| **RQ3** | 如何选择合适的模型？|

### 2.1 评估维度

def calculate_cost(model, query, database):
    # LLM API 成本
    tokens = count_tokens(query, model)
    llm_cost = tokens * model.price_per_token
    
    # 执行成本
    bytes_scanned = execute_and_measure(query, database)
    execution_cost = bytes_scanned * database.price_per_byte
    
    return llm_cost + execution_cost

## 四、实验结果

### 4.1 主要发现

**效果对比**：

| 模型类型 | Execution Accuracy |
|---------|-------------------|
| 推理模型 (o1) | 更高 |
| 非推理模型 (GPT-4) | 略低 |

**成本对比**：

| 维度 | 推理模型 | 非推理模型 |
|------|---------|-----------|
| **Token 消耗** | 高 | 低 |
| **执行字节数** | **少 44.5%** | 多 |
| **总成本** | 高 API 成本 | 低 API 成本 |

**关键发现**：

> 推理模型生成的 SQL 更高效，执行时扫描的数据量少 44.5%！

### 4.2 成本-效果权衡

| 选择 | 效果 | 成本 | 适用场景 |
|------|------|------|---------|
| **推理模型** | 高 | 高 | 复杂查询、高价值场景 |
| **非推理模型** | 中 | 低 | 简单查询、高频场景 |

### 4.3 执行时间与成本关系

**发现**：执行时间与成本效率弱相关

> 执行快的 SQL 不一定成本更低（取决于扫描数据量）

## 五、对 DA-CaseBase 的启示

### 5.1 适用性分析

| 场景 | 适用性 | 原因 |
|------|--------|------|
| **模型选择** | ⭐⭐⭐⭐⭐ | 直接指导三层架构的模型选择 |
| **成本优化** | ⭐⭐⭐⭐ | 案例命中可降低执行成本 |

### 5.2 与三层架构的结合

```
Layer 1: 案例命中
    → 执行预验证 SQL
    → 执行成本可控

Layer 2: 模板匹配
    → 使用非推理模型
    → API 成本低

Layer 3: LLM 生成
    → 选择推理模型（复杂查询）
    → 或非推理模型 + 精炼（平衡成本）
```

### 5.3 关键决策

| 层级 | 模型选择 | 理由 |
|------|---------|------|
| Layer 1 | 无需 LLM | 直接执行缓存 SQL |
| Layer 2 | 非推理模型 | 模板实例化，简单任务 |
| Layer 3 | 推理模型（可选）| 复杂查询需要推理 |

### 5.4 本研究优化建议

**成本优化策略**：

```
1. 提高案例命中率 → 减少 LLM 调用
2. 简化 SQL → 减少执行成本
3. 选择合适模型 → 平衡 API 成本和效果
```
