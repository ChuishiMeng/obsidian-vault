## 一、研究背景

### 1.1 问题场景（一句话说清楚）

**核心问题**：生成的 SQL 虽然正确，但执行很慢，如何既准确又高效？

**通俗理解**：

想象你是一个"SQL 优化师"：
```
用户问："查一下华东区每个客户的订单总额"

SQL A（AI生成的）：
SELECT * FROM orders WHERE customer_id IN (
  SELECT customer_id FROM customers WHERE region = 'East'
)
-- 执行时间：5秒

SQL B（优化后的）：
SELECT o.customer_id, SUM(o.amount) 
FROM orders o 
JOIN customers c ON o.customer_id = c.customer_id 
WHERE c.region = 'East'
GROUP BY o.customer_id
-- 执行时间：0.5秒

问题：两个 SQL 都"正确"，但效率差10倍！
```

**传统方法的问题**：

| 传统方法 | 关注什么 | 问题 |
|---------|---------|------|
| **Text-to-SQL 研究** | 只关注"对不对" | 不在乎"快不快" |
| **SQL 优化器** | 只优化已有 SQL | 不参与生成过程 |

**具体例子**：
```sql
-- 低效 SQL（子查询 + 全表扫描）
SELECT * FROM orders 
WHERE customer_id IN (SELECT customer_id FROM customers WHERE region = 'East')

-- 问题：
-- 1. 子查询导致全表扫描
-- 2. 没有用索引
-- 3. 大表场景下执行很慢

-- 高效 SQL（JOIN + 索引利用）
SELECT o.* FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE c.region = 'East'

-- 优化：
-- 1. JOIN 可以利用索引
-- 2. 数据库优化器更容易优化
-- 3. 执行计划更高效
```

### 1.2 为什么这个问题重要？

| 场景 | 为什么重要 |
|------|-----------|
| **企业生产** | 高峰期查询慢，用户体验差 |
| **成本控制** | 慢查询占用更多计算资源 |
| **学术评估** | 现有基准只看准确性，忽略效率 |

**企业场景**：
```
场景：电商大促，每秒1000次查询

慢查询（5秒）：
  - 用户体验差
  - 数据库压力大
  - 需要更多服务器

快查询（0.5秒）：
  - 用户体验好
  - 数据库压力小
  - 成本更低
```

### 1.3 核心挑战

| 挑战 | 描述 | 现有方案局限 |
|------|------|-------------|
| **准确 vs 效率** | 两个目标可能冲突 | 只优化准确性 |
| **复杂度自适应** | 简单查询不需要复杂推理 | 统一处理，效率低 |
| **结构对齐** | 如何评估 SQL 结构质量 | 只有结果评估 |

## 二、核心贡献

### 2.1 一句话说清楚

**让 Text-to-SQL 既"对"又"快"：简单查询直接生成，复杂查询推理生成，同时优化准确性和效率。**

### 2.2 具体做了什么？

**思路**：以前的 Text-to-SQL 只管"对不对"，这篇论文让它也管"快不快"。

**核心工作**：

**1. 自适应推理模式**

```
简单查询（1-2表，无子查询）：
  直接生成 SQL → 快！

复杂查询（多表、子查询、聚合）：
  先推理（CoT）→ 再生成 SQL → 准！

怎么判断简单/复杂？
  - 表数量 ≤ 2 → 简单
  - JOIN ≤ 1 → 简单
  - 无子查询 → 简单
  其他 → 复杂
```

**2. 骨架完整性评分**

```
传统评估：看结果是否正确
  问题：结果对了，但 SQL 结构可能很差

骨架评估：看 SQL 结构是否对齐
  - SELECT 字段对不对？
  - JOIN 结构对不对？
  - WHERE 条件全不全？
  - GROUP BY/ORDER BY 有没有？
```

**3. 查询延迟感知奖励**

| 奖励类型      | 怎么算        | 作用   |
| --------- | ---------- | ---- |
| **准确性奖励** | SQL 是否正确   | 主要目标 |
| **骨架奖励**  | 结构对齐程度     | 提升质量 |
| **延迟奖励**  | 1/(1+执行时间) | 鼓励高效 |

### 2.3 核心创新

| 创新 | 描述 | 效果 |
|------|------|------|
| **双重优化** | 准确性 + 效率同时优化 | 准确率 +8%，延迟 -15% |
| **自适应模式** | 简单直接，复杂推理 | 简单查询快，复杂查询准 |
| **骨架评分** | 结构对齐评估 | 提升生成质量 |
| **延迟奖励** | 奖励高效 SQL | 鼓励优化 |

## 三、方法论

### 3.1 整体思路（类比理解）

**想象一个"智能 SQL 生成器"**：

```
用户问："查华东区客户订单"
         ↓
    ┌────────────┐
    │ 复杂度判断  │
    │ 表数？JOIN？│
    └────────────┘
         ↓
    ┌────┴────┐
    ↓         ↓
 简单查询   复杂查询
    ↓         ↓
 直接生成   CoT推理
    ↓         ↓
    └────┬────┘
         ↓
    ┌────────────┐
    │ 骨架评分   │
    │ 延迟奖励   │
    └────────────┘
         ↓
    高效正确的 SQL
```

### 3.2 分步流程

**第一步：复杂度判断**

```
输入：用户问题 + Schema
输出：简单/复杂

判断标准：
  简单：表数 ≤ 2 AND JOIN ≤ 1 AND 无子查询
  复杂：其他情况

示例：
  "查用户表所有数据" → 简单（1表，无JOIN）
  "查华东区客户的订单总额" → 复杂（2表，JOIN，聚合）
```

**第二步：自适应生成**

```
简单查询模式：
  用户问题 → 直接生成 SQL
  优点：快（~50ms）
  适用：简单查询

复杂查询模式：
  用户问题 → CoT推理 → SQL
  优点：准（+5%准确率）
  适用：复杂查询
```

**第三步：骨架评分 + 延迟奖励**

```
骨架评分：
  提取 SQL 骨架 = {SELECT, JOIN, WHERE, GROUP BY, ORDER BY}
  与最优 SQL 对比，计算对齐度

延迟奖励：
  执行 SQL，测量时间
  时间越短，奖励越高

总奖励 = 准确性奖励 + 骨架奖励 + 延迟奖励
```

### 3.3 具体例子

**场景**：判断"查华东区客户的订单"应该用哪种模式

```
┌─────────────────────────────────────────────┐
│ 第一步：复杂度判断                           │
│   涉及表：customers, orders（2表）           │
│   JOIN：需要 1 次                           │
│   子查询：无                                │
│   → 判定：中等复杂度                         │
├─────────────────────────────────────────────┤
│ 第二步：选择推理模式                         │
│   不算特别简单，使用推理模式                  │
│   CoT:                                      │
│   1. 先找华东区客户                          │
│   2. 再关联订单                              │
├─────────────────────────────────────────────┤
│ 第三步：生成 SQL                             │
│   SELECT o.* FROM orders o                   │
│   JOIN customers c ON o.customer_id = c.id   │
│   WHERE c.region = 'East'                    │
├─────────────────────────────────────────────┤
│ 第四步：评估                                 │
│   骨架评分：SELECT ✓, JOIN ✓, WHERE ✓        │
│   延迟奖励：执行时间 0.3s，奖励 0.77          │
└─────────────────────────────────────────────┘
```

### 3.4 骨架提取示例

```python
# 原始 SQL
SELECT o.customer_id, SUM(o.amount)
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE c.region = 'East'
GROUP BY o.customer_id
ORDER BY SUM(o.amount) DESC

# 提取骨架
skeleton = {
    'select': ['customer_id', 'SUM(amount)'],
    'joins': [('orders', 'customers', 'customer_id')],
    'where': ['region = East'],
    'group_by': ['customer_id'],
    'order_by': ['SUM(amount) DESC']
}

# 与最优骨架对比
score = jaccard(generated_skeleton, optimal_skeleton)
# score = 0.85 → 骨架对齐度 85%
```

## 四、实验结果

### 4.1 实验设置

**数据集**：

| 数据集 | 规模 | 用途 |
|--------|------|------|
| BIRD | 12,750 问题 | 主实验 |
| KaggleDBQA | 企业查询 | 验证 |

### 4.2 主实验结果

| 数据集 | Baseline | HES-SQL | 准确率提升 | 延迟降低 |
|--------|---------|---------|----------|---------|
| BIRD | 71.2% | **79.14%** | +8% | -15% |
| KaggleDBQA | 49.8% | **54.9%** | +5% | -11% |

### 4.3 消融实验

| 变体 | 准确率 | 延迟 | 说明 |
|------|--------|------|------|
| 完整方法 | 79.14% | 基线 | 骨架 + 延迟 + 自适应 |
| w/o 骨架奖励 | 76.2% | +5% | 结构质量下降 |
| w/o 延迟奖励 | 78.5% | +12% | 效率优化减弱 |
| w/o 自适应 | 77.8% | +8% | 简单查询变慢 |

### 4.4 关键发现

1. **双重优化成功**：准确率提升 + 延迟降低
2. **自适应有效**：简单查询快，复杂查询准
3. **骨架评分重要**：结构对齐提升质量
4. **延迟奖励有效**：鼓励生成高效 SQL
