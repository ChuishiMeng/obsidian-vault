## 一、研究背景

### 1.1 问题场景：NL2SQL 用起来很贵

**背景**：NL2SQL 系统依赖大模型（如 GPT-4），每次查询都要调用 API。

**核心问题**：用得越多，成本越高，企业用不起。

**通俗理解**：

```
场景：企业每天 10,000 次数据查询

每次查询成本：
- Token 消耗：约 2,000 tokens
- 按 GPT-4 价格：$0.03/1K tokens
- 单次成本：$0.06

每月成本：10,000 × 30 天 × $0.06 = $18,000/月

一年就是 $216,000！这还不算高峰期的用量
```

**企业的实际困境**：

| 角色 | 困境 |
|------|------|
| CTO | "NL2SQL 效果不错，但账单太吓人" |
| 产品经理 | "用户很喜欢，但公司不敢推广" |
| 运维 | "高峰期 API 调用量爆表" |

---

### 1.2 成本为什么这么高？

**根本原因**：每次查询都要把整个数据库 Schema 塞给 LLM

**示例**：

```
用户问："查询华东区销售额"

传统方法：
1. 获取数据库 Schema（500 张表）
2. 把 500 张表的结构全部塞进 Prompt
3. LLM 分析后生成 SQL

问题：
- 用户只问 1 张表，却塞了 500 张表
- 大量 Token 是无用的
- 每 1K tokens 就要钱
```

**Token 消耗分析**：

| 内容 | Token 占比 | 必要性 |
|------|-----------|--------|
| 数据库 Schema | 80-90% | 大部分不相关 |
| 用户问题 | 5% | 必要 |
| 生成的 SQL | 5-10% | 必要 |

---

### 1.3 现有方法的浪费

**传统方法的 Token 浪费**：

| 方法 | 流程 | 浪费 |
|------|------|------|
| **直接 Prompt** | 全量 Schema → LLM → SQL | 90% Schema 不相关 |
| **DIN-SQL** | 多步推理，多次调用 | 重复加载 Schema |
| **RAG 增强** | 检索 + 生成 | 检索结果可能不完整 |

**核心矛盾**：

> 每次都加载全量 Schema，但用户问题通常只涉及几张表
> 
> 就像"只想去楼下便利店，却要看整个城市的地图"

---

### 1.4 理想的解决方案

**理想状态**：

```
用户问："查询华东区销售额"

系统智能分析：
1. 这个问题涉及"订单表"和"地区表"
2. 只加载这 2 张表的 Schema
3. LLM 基于精简 Schema 生成 SQL

Token 消耗：从 2000 → 200（降低 90%）
```

**核心问题**：如何知道用户问题涉及哪些表？

---

### 1.5 研究问题

**本文要解决的问题**：

1. 如何按需加载 Schema，减少 Token 消耗？
2. 如何在不降低准确率的前提下降低成本？
3. 大规模 Schema 场景下，优化效果如何？

**核心贡献**：Agentic NL2SQL —— **交互式按需探索匹配**，Token 降低 87%
| 直接 Prompt | 2,000+ | $0.06+ | 全量 Schema，冗余大 |
| DIN-SQL | 1,500+ | $0.045+ | 多次调用，累积成本 |
| RAG 增强 | 1,000+ | $0.03+ | 检索也有成本 |

**核心问题**：

> 每次查询都加载全量 Schema，大量冗余 Token

### 3.2 核心贡献：交互式按需探索匹配

**与传统 Schema Linking 的关键区别**：

| 维度 | 传统 Schema Linking | Agentic NL2SQL（本文）|
|------|-------------------|---------------------|
| **时机** | 一次性静态链接 | **交互式、动态探索** |
| **信息获取** | 全量 Schema 一次性加载 | **按需请求元信息** |
| **探索方式** | LLM 基于 Prompt 推断 | **Agent 主动询问数据库元数据** |
| **Token 消耗** | 高（包含大量无关信息）| **低（只请求必要信息）** |

**通俗理解**：

```
传统 Schema Linking：
1. 把整个数据库 Schema 塞给 LLM
2. LLM 在 Prompt 里分析哪些表相关
3. 问题：Schema 太大，Token 浪费严重

Agentic 方法：
1. Agent 先问数据库："有哪些表？"
2. 数据库只返回表名列表（很少的 Token）
3. Agent 分析后问："给我订单表的列信息"
4. 数据库只返回这一张表的 Schema
5. 循环直到收集足够信息
```

**核心创新**：

> 不是让 LLM 在 Prompt 里"猜"哪些表相关，而是让 Agent **主动探索**，按需请求

---

### 3.3 交互式获取流程

**流程对比**：

```
传统方法（一次性的、静态的）：
用户问题 → 全量 Schema 塞进 Prompt → LLM 一次性分析 → 生成 SQL
问题：Prompt 太长，Token 浪费

Agentic 方法（交互式、动态的）：
用户问题 → Agent 请求表名列表 → 分析哪些表相关
           → Agent 请求相关表的列信息 → 分析哪些列相关
           → Agent 请求主外键关系 → 分析 Join 路径
           → 构建精简 Schema → LLM 生成 SQL
优势：每一步只请求必要信息，Token 最小化
```

**代码示例**：

```python
def interactive_schema_exploration(question, database):
    # 第1轮：获取表名列表（极少 Token）
    all_tables = database.list_tables()  # 返回 ["orders", "users", "products", ...]
    
    # 第2轮：Agent 分析哪些表相关
    relevant_tables = agent.analyze_relevant_tables(question, all_tables)
    # 返回 ["orders", "users"]
    
    # 第3轮：只请求相关表的详细信息
    schema = {}
    for table in relevant_tables:
        # 按需获取，不获取无关表的 Schema
        schema[table] = database.get_table_schema(table)
    
    # 第4轮：请求主外键关系
    fk_relations = database.get_foreign_keys(relevant_tables)
    
    # 构建 Prompt，Token 大幅减少
    return build_minimal_prompt(question, schema, fk_relations)
```

---

### 3.4 为什么这种方式更高效？

**Token 对比**：

```
传统方法：
- 请求 1 次，返回全量 Schema（500 张表 = 20,000 tokens）

Agentic 方法：
- 第1轮请求表名：500 个表名 = 200 tokens
- 第2轮请求 2 张表的 Schema = 400 tokens
- 第3轮请求外键关系 = 100 tokens
- 总计：700 tokens（降低 96%！）
```

**关键洞察**：

> Agent 的多次"小请求"比一次"大请求"更节省 Token
> 
> 因为每一步都只请求必要信息，避免冗余

### 3.3 Token 优化效果

**优化前后对比**：

| 场景 | 传统方法 Tokens | Agentic Tokens | 降低比例 |
|------|----------------|----------------|---------|
| **小 Schema（10 表）** | 1,000 | 300 | 70% |
| **中 Schema（50 表）** | 5,000 | 500 | **90%** |
| **大 Schema（200 表）** | 20,000 | 1,000 | **95%** |
| **平均** | - | - | **87%** |

## 五、实验结果与分析

### 5.1 主要结果

**Token 和成本优化**：

| 方法 | Tokens/查询 | 成本/查询 | 降低 |
|------|------------|----------|------|
| Baseline | 2,000 | $0.060 | - |
| CHESS | 800 | $0.027 | 57% |
| **Agentic NL2SQL** | **260** | **$0.008** | **87%** |

**准确率对比**：

| 方法 | Execution Accuracy |
|------|-------------------|
| Baseline | 46.3% |
| CHESS | 64.6% |
| **Agentic NL2SQL** | **68.5%** |

**关键发现**：

> Token 降低 87% 的同时，准确率反而提升！

### 5.2 Schema 规模影响

| Schema 规模 | Token 降低 | 准确率影响 |
|------------|-----------|-----------|
| 小（< 20 表）| 70% | 无影响 |
| 中（20-100 表）| 90% | 略有提升 |
| 大（> 100 表）| 95% | 显著提升 |

**原因分析**：

> Schema 越大，冗余越多，精简后噪声更少，准确率反而提升

## 七、论文局限性

### 7.1 论文承认的局限

- Schema 裁剪有额外开销
- 极端场景可能裁剪过度
- 需要平衡准确率和效率

### 7.2 更深层局限

| 局限 | 说明 |
|------|------|
| **裁剪成本** | Schema 裁剪本身需要 LLM 调用 |
| **裁剪风险** | 可能裁剪掉重要信息 |
| **依赖 Schema 质量** | Schema 元数据不完整时效果差 |

## 八、研究价值总结

### 8.1 核心贡献

1. **交互式按需探索匹配**：Agent 主动请求数据库元信息，而非一次性加载全量 Schema
2. **与传统 Schema Linking 的区别**：动态交互 vs 静态链接，多次小请求 vs 一次大请求
3. **Token 降低 87%**：显著降低成本，准确率不降反升

### 8.2 与本研究的关系

**借鉴价值**：
- Schema 裁剪方法 → Layer 3 生成时降低 Token
- 交互式获取 → 动态加载必要信息

**本研究优势**：
- 案例命中时 **0 Token**（比 Agentic 更彻底）
- 三层架构全面覆盖
