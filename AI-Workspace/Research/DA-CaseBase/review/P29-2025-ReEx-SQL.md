## 一、研究背景

### 1.1 问题场景

**场景描述**：用户问"找出销售部门中工资高于平均值的员工"，Text-to-SQL 生成 SQL 后执行报错。

**具体示例**：
```
生成 SQL：
SELECT name FROM employees WHERE dept = 'Sales' AND salary > AVG(salary)

执行错误：AVG() 不能直接用在 WHERE 子句中
```

**现有方案的问题**：
- 生成后才执行，错误发现太晚
- 推理错误累积，无法中途纠正
- 执行反馈只用于后处理，未融入生成过程

### 1.2 学术场景 vs 企业场景对比

| 维度 | 学术基准 | 企业实际 |
|------|---------|---------|
| 执行验证 | 事后验证 | 需要即时反馈 |
| 错误处理 | 标注为错误 | 需要自动修复 |
| 数据库依赖 | 简化环境 | 复杂异构数据库 |

---

## 二、核心贡献

### 2.1 主要贡献

1. **执行感知推理**：每步推理后执行验证，错误即时回退
2. **树状解码策略**：多路径探索，选择最优 SQL
3. **复合奖励函数**：正确性 + 探索质量 + 效率

### 2.2 创新点

- **方法创新**：执行验证融入生成过程，而非后处理
- **技术创新**：树状解码实现多路径探索和剪枝
- **应用创新**：推理时间减少 51.9%

---

## 三、方法论

### 3.1 整体框架

```
标准推理：Step1 → Step2 → SQL → 执行 → Error!（太晚）

执行感知推理：
Step1 → 执行✓ → Step2 → 执行✗ → 回退 → Step2' → 执行✓ → SQL
```

### 3.2 核心方法

#### 方法1：执行感知提示结构

**输入**：用户问题 + Schema
**输出**：带执行验证的推理链
**核心步骤**：
1. 生成推理步骤
2. 提取 SQL 片段执行
3. 根据结果调整推理

**关键示例**：
```
<reasoning>
Step 1: 找出销售部门员工
<execute>SELECT id FROM employees WHERE dept = 'Sales'</execute>
<result>[1, 3, 5, 7]</result>
Step 2: 计算平均工资
<execute>SELECT AVG(salary) FROM employees</execute>
<result>8500</result>
</reasoning>
<sql>...</sql>
```

#### 方法2：树状解码策略

| 维度 | 线性解码 | 树状解码 |
|------|---------|---------|
| 探索能力 | 低 | 高 |
| 成功率 | 中 | 高 |
| 计算开销 | 低 | 高（需剪枝） |

### 3.3 技术细节

| 组件 | 实现方式 | 备注 |
|------|---------|------|
| 执行引擎 | 数据库连接 | 需要实际 DB |
| 奖励函数 | R_total = R_correct + αR_explore + βR_efficiency | 复合奖励 |
| 剪枝策略 | 基于执行结果 | 降低开销 |

## 四、实验结果

### 4.1 实验设置

**数据集**：

| 数据集 | 规模 | 用途 |
|--------|------|------|
| Spider | 7,000 问题 | 主实验 |
| BIRD | 12,750 问题 | 验证 |
| Spider-Realistic | 1,000 问题 | 真实场景 |

### 4.2 主实验结果

| 方法 | Spider | BIRD | Spider-Realistic |
|------|--------|------|------------------|
| Baseline | 86.1% | 62.3% | 82.5% |
| ReEx-SQL | **88.8%** | **64.9%** | **85.2%** |
| 提升 | +2.7% | +2.6% | +2.7% |

### 4.3 消融实验

| 变体 | Accuracy | 变化 |
|------|----------|------|
| 完整方法 | 88.8% | - |
| w/o 执行验证 | 87.0% | ↓1.8% |
| w/o 树状解码 | 87.6% | ↓1.2% |
| w/o 奖励函数 | 87.9% | ↓0.9% |

### 4.4 关键发现

1. **执行验证是核心**：移除后下降最多
2. **推理时间减少 51.9%**：剪枝策略有效
3. **多路径探索有效**：提高成功率
