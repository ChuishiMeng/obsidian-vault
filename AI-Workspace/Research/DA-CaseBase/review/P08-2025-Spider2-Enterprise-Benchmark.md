## 一、研究背景

### 1.1 应用场景

**学术基准 vs 企业现实**：

| 维度 | Spider 1.0（学术） | Spider 2.0（企业） |
|------|------------------|-------------------|
| **数据库规模** | 几十张表 | 数百张表 |
| **Schema 复杂度** | 清晰的字段命名 | 业务缩写、内部术语 |
| **查询类型** | 单次查询 | 工作流（多步骤） |
| **SQL 方言** | 标准 SQL | Snowflake, BigQuery, DuckDB |

**企业用户的真实需求**：

| 角色 | 需求 | 学术基准无法反映的问题 |
|------|------|----------------------|
| 数据分析师 | 分析复杂业务数据 | Schema 过于简化 |
| 业务运营 | 跨系统数据查询 | 单数据库，无跨系统 |
| 高管 | 实时业务监控 | 静态数据，无工作流 |

### 1.2 现有基准的局限

**Spider 1.0 的问题**：

| 问题 | 说明 |
|------|------|
| **已被"解决"** | SOTA 模型达到 90%+ 准确率 |
| **脱离现实** | 学术合成数据，非真实企业场景 |
| **无工作流** | 单次查询，无多步骤任务 |

**Gap 分析**：

```
学术基准高分 ≠ 企业部署成功

原因：
1. Schema 复杂度差异巨大
2. 企业有特定业务术语
3. 需要跨数据源查询
4. 需要处理工作流
```

## 二、问题定义

### 2.1 研究目标

**目标**：构建真实企业级 Text-to-SQL 评估基准

**设计原则**：

| 原则 | 说明 |
|------|------|
| **真实性** | 真实企业数据库、真实查询需求 |
| **复杂性** | 复杂 Schema、复杂 SQL |
| **工作流** | 多步骤任务，非单次查询 |
| **多方言** | 支持多种 SQL 方言 |

### 2.2 核心挑战

| 挑战 | 说明 |
|------|------|
| **C1: 数据获取** | 企业数据敏感，难以公开 |
| **C2: 标注成本** | 企业级 SQL 需要专家标注 |
| **C3: 评估标准** | 如何定义"正确"？执行结果 vs 语义等价 |
| **C4: 工作流定义** | 如何表示和评估多步骤任务？ |

## 三、核心方法

### 3.1 Spider 2.0 基准设计

**数据来源**：

| 来源 | 说明 |
|------|------|
| **公开数据平台** | Snowflake, BigQuery 示例数据库 |
| **开源项目** | DuckDB, dbt 项目 |
| **数据竞赛** | Kaggle 等平台 |

**数据规模**：

| 指标 | 值 |
|------|-----|
| **任务数** | 632 |
| **数据库数** | 多个企业级数据库 |
| **SQL 方言** | Snowflake, BigQuery, DuckDB |
| **工作流比例** | 高（多步骤任务） |

### 3.2 任务类型

**工作流任务示例**：

```
任务: 分析销售趋势
├── Step 1: 查询过去12个月的销售额
├── Step 2: 计算环比增长率
├── Step 3: 找出增长最快的地区
└── Step 4: 生成报告

每个步骤都是一个独立的 Text-to-SQL 任务
```

**任务复杂度分布**：

| 复杂度 | 占比 | 说明 |
|--------|------|------|
| **简单** | 20% | 单表查询 |
| **中等** | 50% | 多表 JOIN、聚合 |
| **复杂** | 30% | 嵌套子查询、窗口函数、工作流 |

### 3.3 评估方式

**多维度评估**：

| 维度 | 评估方式 |
|------|---------|
| **SQL 正确性** | 执行结果对比 |
| **工作流完成度** | 所有步骤是否完成 |
| **效率** | 执行时间、Token 消耗 |

**评估代码**：

```python
def evaluate_workflow(tasks, model):
    results = []
    for task in tasks:
        # 1. 执行工作流
        workflow_success = True
        for step in task.steps:
            sql = model.generate_sql(step.question, task.schema)
            success, result = execute_sql(sql, task.database)
            if not success:
                workflow_success = False
                break
        
        # 2. 记录结果
        results.append({
            'task_id': task.id,
            'workflow_success': workflow_success,
            'sql_correct': ...,
            'execution_time': ...
        })
    
    return results
```

## 五、实验结果与分析

### 5.1 主要结果

**模型表现对比**：

| 模型 | Spider 1.0 EX | Spider 2.0 Workflow Success |
|------|--------------|----------------------------|
| GPT-4 | 86%+ | **~30%** |
| Claude | 85%+ | ~25% |
| 开源 SOTA | 80%+ | ~15% |

**关键发现**：

> Spider 2.0 上所有模型表现大幅下降！
> 
> Spider 1.0 "已解决"（90%+）→ Spider 2.0 "仍困难"（<30%）

### 5.2 错误分析

| 错误类型 | 占比 | 说明 |
|---------|------|------|
| **Schema 理解错误** | 40% | 无法理解复杂 Schema |
| **SQL 语法错误** | 25% | 方言差异导致 |
| **逻辑错误** | 20% | 查询逻辑不正确 |
| **工作流错误** | 15% | 步骤间依赖处理失败 |

### 5.3 难度因子分析

| 因子 | 难度影响 |
|------|---------|
| **Schema 大小** | 表越多越难 |
| **SQL 复杂度** | JOIN/子查询越多越难 |
| **方言差异** | 特殊语法导致错误 |
| **工作流长度** | 步骤越多越难 |

## 六、对 DA-CaseBase 的深度启示

### 6.1 与本研究的高度相关性

| 维度 | Spider 2.0 | DA-CaseBase |
|------|-----------|-------------|
| **场景** | 企业级 Text-to-SQL | ✅ 完全一致 |
| **评估基准** | 需要验证企业场景效果 | ✅ **必要的评估参考** |
| **复杂度** | 真实企业复杂度 | ✅ 目标场景 |

### 6.2 可借鉴的评估方式

| 技术 | 应用方式 |
|------|---------|
| **工作流评估** | 评估多步骤查询的完整度 |
| **多方言支持** | 支持不同数据库方言 |
| **错误分类** | 用于错误分析和改进 |

### 6.3 对本研究的关键意义

**1. 评估基准选择**：

```
DA-CaseBase 评估应使用：
- Spider 2.0（企业级真实场景）
- 而非 Spider 1.0（已被"解决"）
```

**2. 难度认知**：

| 基准 | SOTA 表现 | 本研究目标 |
|------|----------|-----------|
| Spider 1.0 | 90%+ | 不适用（过于简单）|
| Spider 2.0 | ~30% | **验证真实企业场景效果** |

**3. 错误分析参考**：

- Schema 理解是最大难点（40% 错误）
- 案例库可缓解 Schema 理解问题

### 6.4 与 CBR-to-SQL 的结合

```
CBR-to-SQL 在 Spider 1.0 上效果优秀（82.8%）

但 Spider 2.0 的企业场景挑战：
1. 更复杂的 Schema
2. 工作流任务
3. 多 SQL 方言

DA-CaseBase 需要在 Spider 2.0 上验证
```

## 七、论文局限性

### 7.1 论文承认的局限

- 标注成本高，数据规模有限
- 无法覆盖所有企业场景
- 工作流定义有主观性

### 7.2 更深层局限

| 局限 | 说明 |
|------|------|
| **无企业私有数据** | 敏感数据无法公开 |
| **工作流简化** | 真实工作流更复杂 |
| **无用户反馈** | 缺少用户满意度评估 |

## 八、研究价值总结

### 8.1 核心贡献

1. **揭示真实差距**：学术基准高分 ≠ 企业成功
2. **企业级基准**：真实数据库、工作流任务
3. **评估框架**：多维度评估方法

### 8.2 与本研究的关系

**作为测试数据集**：
- Spider 2.0 提供企业级真实场景测试数据
- 可用于验证 DA-CaseBase 的效果

**评估建议**：
- 使用 Spider 2.0 工作流任务
- 报告 Workflow Success Rate
- 分析不同复杂度的表现

**核心参考**：
- DA-CaseBase 需要在企业级基准上验证
- Spider 2.0 是目前最合适的企业级测试集
