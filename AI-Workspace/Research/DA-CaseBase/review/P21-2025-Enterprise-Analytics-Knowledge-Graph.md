## 一、研究背景

### 1.1 问题场景：LinkedIn 需要全公司都能用的数据助手

**背景**：LinkedIn 有数百万张表，员工需要频繁查询数据。

**核心问题**：如何让不会 SQL 的员工也能自助查询数据？

**具体场景**：

```
现实情况：
- 产品经理想看"上周新用户注册趋势"
- 工程师想查"最近 API 错误率"
- 运营想看"各渠道转化率"

问题：
- 公司有数百万张表
- 字段名是缩写，看不懂
- 不同团队关心不同的表
- 业务术语（黑话）很多
```

---

### 1.2 企业级部署的四大挑战

| 挑战 | 具体表现 | 解决方案 |
|------|---------|---------|
| **Schema 规模** | 数百万表，热门表 100+ 列 | 用户-表聚类 |
| **语义歧义** | 同一术语不同领域含义不同 | 个性化上下文 |
| **动态变化** | 表定期弃用、信息重叠 | 知识图谱定期刷新 |
| **辅助任务** | 找表、解释查询、调试 | 多 Agent 架构 |

```
└──────────────┴───────────────┴───────────────┘
```

---

### 2.2 知识图谱构建

**数据源**：

| 来源 | 内容 | 用途 |
|------|------|------|
| **数据库元数据** | Schema、外键、列类型 | 表/列属性 |
| **历史查询日志** | 访问频率、常见 JOIN | 使用模式 |
| **Wiki/文档** | 表说明、业务背景 | 语义增强 |
| **代码仓库** | 查询示例、使用上下文 | Few-shot 示例 |
| **众包知识** | 领域知识、用户偏好 | 个性化 |

**知识图谱结构**：

```
节点:
- 用户 (User)
- 表集群 (Table Cluster)
- 表 (Table)
- 列 (Column)
- 产品领域 (Product Area)

关系:
- 用户 → 表集群 (访问模式)
- 表集群 → 产品领域 (业务关联)
- 表 → 表 (常见 JOIN)
- 表 → 列 (Schema)
```

---

### 2.3 用户-表聚类

**问题**：数百万表，但每个用户只关心一部分

**方法**：

```
1. 提取 3 个月查询历史的用户-表访问矩阵
2. 过滤低频表（中间表、管道输出表）
3. ICA 降维 (N_comp = 200)
4. 软聚类：每组件取 Top T_c 表 (T_c = 20)
5. 为用户分配 Top 组件作为个性化表集
6. 为产品领域分配相关组件
```

**优势**：
- 运行时间 < 15 分钟
- 表可属于多个集群
- 适应表弃用和团队变化

### 2.4 Query Writer Agent

**四步流程**：

```
┌─────────────────────────────────────────────┐
│           Query Writer Agent               │
├───────────┬───────────┬──────────┬─────────┤
│  Retrieve │   Rank    │  Write   │   Fix   │
│   检索    │   排序    │   生成   │   修正  │
└───────────┴───────────┴──────────┴─────────┘
```

**Retrieve Context**：
- 候选表：用户/产品领域的表集群
- K_ret = 20 表（最大化召回）
- 示例查询、领域知识、术语解释

**Rank Context**：
- LLM 评分表相关性 (1-10)
- K_rnk = 7 表（平衡精度和延迟）
- 列分为"相关"和"可能相关"两级

**Write Query**：
- 使用排序后的 Schema + 示例 + 领域知识
- 输出：假设、SQL、解释、表、列

**Fix Query**：
- Trino EXPLAIN VALIDATE 检测错误
- Researcher Agent 检索缺失表/列
- 最多 2 轮修正

---

### 2.5 多 Agent 架构

**Agent 分工**：

| Agent | 职责 | 模型 |
|-------|------|------|
| **Query Writer** | SQL 生成 | GPT-4o |
| **Researcher** | 检索补充信息 | GPT-4o-mini |
| **Data Finder** | 找表 | GPT-4o-mini |
| **Query Fixer** | 修复错误 | GPT-4o |
| **Q&A Agent** | 其他问题 | GPT-4o-mini |

**Intent Router**：根据用户意图路由到对应 Agent

## 三、实验结果

### 3.1 企业内部基准

| 指标 | 数值 |
|------|------|
| **周活用户** | 300+ |
| **专家评估准确率** | 53% 正确或接近正确 |
| **问题覆盖** | 10 个产品领域 |

### 3.2 消融实验

| 组件移除 | 准确率下降 |
|---------|----------|
| 知识图谱 | 显著下降 |
| 用户-表聚类 | 中等下降 |
| Researcher Agent | 小幅下降 |
| 领域知识 | 小幅下降 |

## 四、对 DA-CaseBase 的启示

### 4.1 知识图谱用于案例组织

**方法借鉴**：

```
案例库组织:
- 表节点 → 相关案例
- 用户节点 → 常用案例
- 领域节点 → 领域案例

检索增强:
- 表描述 → 语义理解
- 常见 JOIN → 关系推理
- 历史日志 → 使用模式
```

### 4.2 用户-案例聚类

- 根据用户访问模式聚类案例
- 为不同用户/团队提供个性化案例集
- 适应案例库动态变化

### 4.3 Researcher Agent

- 检索缺失的案例
- 验证案例有效性
- 推荐相似案例
