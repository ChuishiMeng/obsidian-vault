## 核心问题

### 解决什么问题？
构建**开源**的 Text-to-SQL 专用语言模型，替代对 GPT-4/ChatGPT 等闭源模型的依赖。

### 为什么重要？
闭源 LLM 用于 Text-to-SQL 存在三大限制：
1. **L1 - 不透明**：架构和训练细节不公开，无法针对特定应用持续优化
2. **L2 - 隐私风险**：数据需发送到模型提供商的 API
3. **L3 - 成本高昂**：175B+ 参数导致推理开销巨大（GPT-4 每个样本约60秒）

### CodeS 的优势
- **全开源**：基于 StarCoder，模型/代码/数据全部公开
- **小而强**：1B/3B/7B/15B 参数，比 GPT-4 小 10x-100x
- **新 SOTA**：在几乎所有 Text-to-SQL 基准上达到 SOTA

## 技术方法

### 整体架构（三大组件）

```
┌─────────────────────────────────────────────────┐
│  (a) 增量预训练                                    │
│  StarCoder → SQL语料训练 → CodeS (1B/3B/7B/15B)    │
├─────────────────────────────────────────────────┤
│  (b) 数据库 Prompt 构建（在线）                      │
│  Schema Filter + Value Retriever + Metadata       │
├─────────────────────────────────────────────────┤
│  (c) 新领域适应（离线）                              │
│  双向数据增强：Question→SQL + SQL→Question          │
├─────────────────────────────────────────────────┤
│  使用方式：SFT微调 / Few-shot ICL                   │
└─────────────────────────────────────────────────┘
```

### 组件一：增量预训练

**基座模型**：StarCoder（1B/3B/7B/15B）

**预训练语料（21.5GB）**：
| 类型 | 大小 | 来源 | 训练轮次 |
|------|------|------|---------|
| SQL相关数据 | 11GB | StarCoder预训练语料中的SQL部分 | 2 epochs |
| NL-to-Code数据 | 6GB | CoNaLa, StaQC, CodeAlpaca-20k, Jupyter notebooks, **NL-SQL-458K（自建）** | 1 epoch |
| NL相关数据 | 4.5GB | Alpaca-cleaned, Unnatural-instructions, UltraChat | 1 epoch |

**NL-SQL-458K（亮点）**：从 The Pile、The Stack、GitHub Code 中用正则提取 SELECT 语句，过滤语法错误后得到 458K SQL，再用 GPT-3.5 生成对应自然语言问题。

**训练细节**：
- 优化器：AdamW（β1=0.9, β2=0.95）
- 学习率：5e-5，cosine衰减
- Batch size：4M tokens
- 混合精度：BF16 + DeepSpeed Zero-3
- FlashAttention-2
- 上下文长度：1B/3B/7B = 8192，15B = 6144
- 训练耗时：1.5/3/8/16 天（1B/3B/7B/15B）

### 组件二：数据库 Prompt 构建

**目标**：为每个用户查询构建高质量的数据库上下文，缓解 Schema Linking 难题。

#### Schema Filter（模式过滤器）
- 训练一个 schema item classifier（基于问题预测表/列的相关性分数）
- 保留 top-k1 个表（默认6），每个表保留 top-k2 列（默认10）
- 训练集中根据 ground truth SQL 识别使用的表/列，不足时随机填充

#### Value Retriever（值检索器）— 粗到细匹配
1. **粗粒度**：用 BM25 索引（Lucene/Pyserini）从全库快速检索数百个候选值
2. **细粒度**：用最长公共子串（LCS）算法精确匹配

解决了传统 LCS 在大规模数据库（如 BIRD 的 Donor 库有 1.16 亿值）上的效率瓶颈。

#### 元数据（Metadata）
- **列数据类型**：INTEGER/REAL/TEXT 等，指导 CAST 使用
- **注释（Comments）**：解决缩写/歧义列名问题（如 `w1st` → "word id of the first word"）
- **代表性值**：每列取 2 个 DISTINCT 非空值，帮助理解值格式
- **主键/外键**：指导 JOIN ON 子句生成

**Prompt 格式示例**：
```
table movie, columns = [ movie.mid (int | primary key | comment: movie id | values: 101, 102), ...]
foreign keys: rating.rid = reviewer.rid ...
matched values: reviewer.name (Sarah Martinez)
Question: What are the names of all directors...
```

### 组件三：双向数据增强（新领域适应）

**目标**：用最少标注（仅30个样本）为新领域数据库生成大量训练数据。

#### Question→SQL 方向（保真性）
1. 收集少量真实用户问题
2. 人工标注对应 SQL
3. 用 GPT-3.5 两阶段扩展：
   - Stage 1：模拟用户偏好生成新问题
   - Stage 2：为新问题生成对应 SQL

#### SQL→Question 方向（通用性）
1. 从 Spider 提取 75 个通用 (question, SQL) 模板
2. 用新数据库的表/列/值填充模板
3. 用 GPT-3.5 润色模板问题使其更自然

**效果**：从30个标注样本扩展到 ~5000 个训练样本。

### 使用方式

#### 有充足训练数据 → SFT 微调
- 4 epochs，batch size 128，lr 5e-6
- Beam search 生成 4 个候选 SQL，选第一个可执行的

#### 训练数据不足 → Few-shot ICL
- **Question-pattern-aware demonstration retriever**：
  - 同时考虑问题语义相似度和**问题模式相似度**（去掉实体后的骨架）
  - 避免仅基于实体匹配导致的偏差
  - 使用 SimCSE 计算句子相似度
  - 相似度 = max(原始问题相似度, 模式相似度)

## 实验结果

### 评估数据集

| 数据集 | 训练集 | 测试集 | 特点 |
|--------|--------|--------|------|
| Spider | 8,659 | 1,034 (dev) | 200数据库，138领域，经典基准 |
| BIRD | 9,428 | 1,534 (dev) + hidden test | 95数据库，33.4GB，平均549K行/库 |
| Spider-DK/Syn/Realistic | - | Spider变体 | 鲁棒性测试 |
| Dr.Spider | - | 17种扰动 | 全面鲁棒性评估 |
| Bank-Financials | 4,901(增强) | 91 | 金融领域，最大表65列 |
| Aminer-Simplified | 5,427(增强) | 97 | 学术领域，复杂表关联 |

### 评估指标
- **EX（Execution Accuracy）**：预测SQL和标准SQL执行结果是否一致
- **TS（Test-suite Accuracy）**：多数据库实例上的EX（减少假阳性）
- **VES（Valid Efficiency Score）**：考虑执行效率的分数
- **HE（Human Evaluation）**：人工评判SQL正确性

### 核心实验结果

#### Few-shot ICL（增量预训练效果）

| 模型 | Spider dev (TS%, 5-shot) | BIRD dev (EX%, 5-shot) |
|------|--------------------------|------------------------|
| StarCoderBase-7B | 64.6 | 30.44 |
| **CodeS-7B** | **71.8 (+7.2)** | **34.49 (+4.05)** |
| StarCoder-15B | 69.6 | 33.64 |
| **CodeS-15B** | **73.4 (+3.8)** | **37.03 (+3.39)** |

→ 增量预训练对小模型提升更显著（1B提升10.5% vs 15B提升3.8%）

#### SFT 微调

**Spider dev set：**

| 方法 | EX% | TS% |
|------|-----|-----|
| DIN-SQL + GPT-4 | 82.8 | 76.6 |
| DAIL-SQL + GPT-4 | 83.6 | 76.2 |
| **SFT CodeS-3B** | 84.1 | 78.1 |
| **SFT CodeS-7B** | **85.4** | **80.3** |
| SFT CodeS-15B | 84.9 | 79.4 |

→ **3B 模型就超过了 GPT-4 方案**，7B 达到新 SOTA

**BIRD test set（无外部知识）：**

| 方法 | EX% |
|------|-----|
| ChatGPT + CoT | 28.95 |
| DIN-SQL + GPT-4 | 50.72 |
| **SFT CodeS-15B** | **52.15** |

→ 比 ChatGPT+CoT 提升 23.2%

**BIRD test set（有外部知识）：**

| 方法 | EX% |
|------|-----|
| DIN-SQL + GPT-4 | 54.76 |
| **SFT CodeS-7B** | **58.47** |
| **SFT CodeS-15B** | **59.25** |

#### 鲁棒性评估

| 数据集 | 之前最佳 | SFT CodeS-7B |
|--------|---------|-------------|
| Spider-DK (EX%) | 67.5 | **72.0 (+4.5)** |
| Spider-Syn (TS%) | 67.4 | **70.0 (+2.6)** |
| Spider-Realistic (TS%) | 73.2 | **77.2 (+4.0)** |
| Dr.Spider 全局平均 | 74.9 | **75.0** |

#### 真实场景（新领域适应）

| 方法 | Bank-Financials HE% | Aminer HE% |
|------|---------------------|------------|
| 3-shot GPT-3.5 + comments | 84.6 | 62.8 |
| DIN-SQL + GPT-4 | 79.1 | 67.0 |
| **SFT CodeS-7B (aug. data)** | **85.7** | **64.9** |
| **SFT CodeS-7B (merged data)** | 84.6 | **67.0** |

→ 仅30个标注样本 + 双向增强即可达到/超过 GPT-4 水平

### 推理效率

| 模型 | 推理时间/样本 | GPU显存需求 |
|------|-------------|-----------|
| DIN-SQL + GPT-4 | ~60秒 | - |
| SFT CodeS-1B | 0.6秒 | 10GB |
| SFT CodeS-3B | 0.9秒 | 13GB |
| SFT CodeS-7B | 1.1秒 | 20GB |
| SFT CodeS-15B | 1.5秒 | 35GB |

→ **比 GPT-4 快 40-100 倍**，可本地部署

### 消融实验要点（3-shot ICL, CodeS-15B）

| 消融项 | Spider dev TS% 变化 | BIRD dev EX% 变化 |
|--------|---------------------|-------------------|
| 去掉模式相似度 | -0.1 | +0.65 |
| 去掉 demonstration retriever | -0.4 | +0.39 |
| 去掉 schema filter | -1.3 | -1.57 |
| 去掉 value retriever | -0.4 | -1.83 |
| 去掉 comments | -0.5 | -1.11 |
| 去掉代表性值 | -1.1 | -3.20 |
| 去掉主键/外键 | -1.5 | -3.00 |

→ **代表性值和主键/外键对BIRD影响最大**（复杂数据库场景）

## 与 DA-CaseBase 的关系

### 可借鉴点

#### 1. CodeS 作为案例库的 NL2SQL 执行引擎
CodeS 是**开源可控的 NL2SQL 模型**，天然适合作为 DA-CaseBase 系统的底层执行环境：
- **开源可定制**：可以在案例库的特定领域上增量预训练或微调
- **本地部署**：7B 模型仅需 20GB 显存，1.1秒/查询，适合生产环境
- **隐私安全**：数据不出域

#### 2. Prompt 构建策略 → 案例表示方法
CodeS 的数据库 Prompt 构建是一个精心设计的**数据库知识表示方案**，可借鉴用于案例库的案例表示：
- **Schema Filter** → 案例库的**相关性检索**（根据问题过滤相关表/列）
- **Value Retriever** → 案例库的**值匹配**（BM25 + LCS 粗到细策略）
- **Metadata 整合** → 案例库的**上下文丰富化**（类型、注释、代表值、关系）

#### 3. 双向数据增强 → 案例库的案例生成
CodeS 的双向数据增强思路可直接用于扩充案例库：
- **Question→SQL**：从少量真实案例扩展（保真性）
- **SQL→Question**：从模板生成通用案例（覆盖度）
- 仅需 30 个标注样本即可生成 ~5000 个训练案例

#### 4. Demonstration Retriever → 案例检索策略
CodeS 的 **question-pattern-aware retriever** 是经典的 CBR（Case-Based Reasoning）检索范式：
- 同时考虑表面相似度（问题语义）和结构相似度（问题模式/骨架）
- 用 SimCSE 计算句子嵌入相似度
- 这正是案例库理论中**案例检索（Case Retrieval）**的核心问题

### 本研究切入点

1. **CodeS 作为集成环境验证案例库理论**：将案例库方法集成到 CodeS 的 demonstration retrieval 和 prompt construction 流程中，验证 CBR 是否能进一步提升 NL2SQL 性能

2. **案例表示的系统化**：CodeS 的 prompt 构建本质上是一种"问题+数据库上下文"的案例表示，但缺乏显式的案例结构化存储和索引。DA-CaseBase 可以在此基础上引入**结构化案例库**，实现更高效的案例管理和检索

3. **案例适应（Case Adaptation）**：CodeS 的模板填充 + GPT 润色本质是一种案例适应过程，DA-CaseBase 可以将其形式化为 CBR 理论框架下的 Case Adaptation

4. **增量学习与案例维护**：CodeS 的新领域适应方法可以与案例库的**案例维护（Case Maintenance）**结合——当新领域的案例积累到一定量时，触发增量微调

## 关键评价

### 优势
- ✅ 工程完整度极高，开源生态友好
- ✅ 小模型（3B-7B）超越 GPT-4，性价比极高
- ✅ 双向数据增强方法实用性强（30个标注样本）
- ✅ 评估全面（8个基准 + 2个真实场景 + 详细消融）

### 局限
- ⚠️ 基于 StarCoder（2023），底座模型已过时
- ⚠️ Schema Filter 依赖训练好的分类器，跨域迁移有限
- ⚠️ 未涉及多轮对话、纠错等交互式场景
- ⚠️ 仅支持 SQLite，未测试其他数据库方言
- ⚠️ 15B 模型并不总是优于 7B（过拟合风险），模型规模边际效应递减
