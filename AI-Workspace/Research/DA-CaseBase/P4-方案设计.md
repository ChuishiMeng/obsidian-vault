## 二、案例定义与等价性判别（RQ1）

### 2.1 案例定义

一个 NL2SQL 案例是一个 **<自然语言查询, SQL查询, 数据库Schema>** 三元组，附带元信息。核心问题是：**两个NL-SQL对何时应视为"同一个案例"？**

### 2.2 等价性判别方法

**核心挑战**：同一查询意图可以有多种NL表述和多种SQL写法，需要在语义层面判定等价性。

```python
class CaseEquivalenceJudge:
    """案例等价性判别器：判定两个NL-SQL对是否表达同一查询意图"""
    
    def judge_equivalence(self, case_a, case_b) -> EquivalenceResult:
        """三层判别策略"""
        
        # Layer 1: SQL结构等价（快速）
        # 两个SQL在抽象掉具体值后，AST结构完全一致
        structural_eq = self.check_structural_equivalence(
            case_a.sql_template, case_b.sql_template
        )
        if structural_eq:
            return EquivalenceResult(equivalent=True, level='structural',
                                     confidence=0.95)
        
        # Layer 2: SQL语义等价（中等）
        # 结构不同但语义等价，如 WHERE a AND b ≡ WHERE b AND a
        semantic_eq = self.check_semantic_equivalence(
            case_a.sql_original, case_b.sql_original
        )
        if semantic_eq:
            return EquivalenceResult(equivalent=True, level='semantic',
                                     confidence=0.85)
        
        # Layer 3: 意图等价（深度）
        # NL表述不同但查询意图相同
        intent_eq = self.check_intent_equivalence(
            case_a.nl_original, case_b.nl_original,
            case_a.sql_original, case_b.sql_original
        )
        return EquivalenceResult(equivalent=intent_eq.score > 0.9,
                                 level='intent', confidence=intent_eq.score)
    
    def check_structural_equivalence(self, sql_tmpl_a, sql_tmpl_b):
        """SQL模板的AST归一化后比较"""
        ast_a = normalize_ast(parse_sql(sql_tmpl_a))
        ast_b = normalize_ast(parse_sql(sql_tmpl_b))
        return ast_a == ast_b
    
    def check_semantic_equivalence(self, sql_a, sql_b):
        """语义等价检测：处理交换律、分配律等"""
        # 1. WHERE子句交换律：A AND B ≡ B AND A
        # 2. JOIN交换律：A JOIN B ≡ B JOIN A
        # 3. 子查询等价：IN (SELECT ...) ≡ EXISTS (SELECT ...)
        # 4. 聚合等价：COUNT(*) WHERE x ≡ SUM(CASE WHEN x THEN 1)
        normalized_a = semantic_normalize(sql_a)
        normalized_b = semantic_normalize(sql_b)
        return normalized_a == normalized_b
    
    def check_intent_equivalence(self, nl_a, nl_b, sql_a, sql_b):
        """意图等价：基于NL语义 + SQL执行结果双重验证"""
        # 1. NL语义相似度（Embedding cosine）
        nl_sim = cosine_similarity(encode(nl_a), encode(nl_b))
        # 2. SQL执行结果一致性（在测试数据上执行并比较）
        result_consistent = compare_execution_results(sql_a, sql_b)
        # 3. 综合判定
        return IntentEquivalenceScore(
            nl_similarity=nl_sim,
            result_consistent=result_consistent,
            score=0.6 * nl_sim + 0.4 * float(result_consistent)
        )
```

### 2.3 等价案例合并策略

当判定两个案例等价时，需要合并而非简单去重：

| 场景 | 合并策略 |
|------|---------|
| 同NL同SQL | 直接去重，保留质量分更高者 |
| 不同NL同SQL | 保留多个NL变体，共享同一SQL模板 |
| 同NL不同SQL | 选择质量分更高的SQL，标注替代写法 |
| 不同NL不同SQL但意图等价 | 保留两者，标记等价关系 |

### 3.2 多粒度案例表示方法（RQ2 核心）

案例库需要支持不同粒度的匹配，因此每个案例在三个粒度层级进行表示：

```
粒度层级:
  
  L1 具体实例层    "查询上个月销售额超过100万的北京客户"
       ↓ 实体抽象
  L2 模板层        "查询{DATE_RANGE}销售额超过{THRESHOLD}的{CITY}客户"
       ↓ 结构抽象
  L3 模式层        [聚合查询] + [条件过滤] + [分组] + [HAVING阈值]
```

| 粒度层级 | 表示内容 | 索引方式 | 匹配场景 |
|---------|---------|---------|---------|
| **L1 具体实例** | 原始NL + SQL | 向量索引 | 完全相同的查询 |
| **L2 抽象模板** | NL模板 + SQL模板 + 实体槽位 | 向量索引 + 模板索引 | 相同模式不同参数 |
| **L3 查询模式** | SQL结构指纹 + 查询意图 | 结构哈希 + 倒排索引 | 结构相似的查询 |

**粒度选择策略**：
```python
def select_granularity(query, candidates):
    """根据匹配情况动态选择合适的粒度"""
    # 优先尝试L1精确匹配
    l1_match = find_exact_match(query, candidates)
    if l1_match and l1_match.score >= 0.95:
        return l1_match, 'L1'
    
    # 尝试L2模板匹配
    l2_match = find_template_match(query.nl_template, candidates)
    if l2_match and l2_match.score >= 0.85:
        return l2_match, 'L2'
    
    # 降级到L3模式匹配
    l3_match = find_pattern_match(query.sql_fingerprint, candidates)
    if l3_match and l3_match.score >= 0.70:
        return l3_match, 'L3'
    
    return None, 'no_match'
```

## 四、案例匹配机制（RQ3）

### 4.1 匹配流程

```
Phase 1: 查询预处理
  ┌──────────────────────────────────────────┐
  │  实体识别 → 模板抽象 → 意图分类           │
  │  "销售额top10" → "{METRIC}{RANKING}"     │
  └──────────────────────────────────────────┘

Phase 2: 多路召回
  ┌────────────┬────────────┬──────────────┐
  │向量语义召回 │模板结构召回 │关键词召回     │
  │(Top-K=50)  │(Top-K=30)  │(Top-K=20)    │
  └────────────┴────────────┴──────────────┘
         合并去重 → 候选集(~60-80条)

Phase 3: 精排匹配
  ┌────────────┬────────────┬──────────────┐
  │NL语义精排   │SQL结构对比 │负例区分       │
  │Cross-Encoder│AST diff   │Contrastive   │
  └────────────┴────────────┴──────────────┘

Phase 4: 匹配质量分级 → 案例应用
  ┌──────────────────────────────────────────┐
  │ 高质量(≥0.95) → 直接复用SQL              │
  │ 中等(0.85-0.95) → 模板实例化             │
  │ 低质量(<0.85) → few-shot示例             │
  │ 无匹配 → 纯NL2SQL生成                    │
  └──────────────────────────────────────────┘
```

### 4.2 多路召回设计

| 召回通道 | 召回方式 | Top-K | 用途 |
|---------|---------|-------|------|
| **向量召回** | Embedding相似度+ANN索引 | 50 | 语义相近候选 |
| **结构召回** | SQL结构哈希+倒排索引 | 30 | 相同结构模板 |
| **关键词召回** | BM25+Schema关键词 | 20 | 表名/列名命中 |

**合并去重策略**：
```python
def merge_recall_results(vector_hits, structure_hits, keyword_hits):
    # 优先级：结构命中 > 向量命中 > 关键词命中
    candidates = {}
    
    # 结构命中（高优先级）
    for case in structure_hits:
        candidates[case.case_id] = (case, 'structure', 0.9)
    
    # 向量命中
    for case in vector_hits:
        if case.case_id not in candidates:
            candidates[case.case_id] = (case, 'vector', 0.8)
    
    # 关键词命中（低优先级）
    for case in keyword_hits:
        if case.case_id not in candidates:
            candidates[case.case_id] = (case, 'keyword', 0.6)
    
    return list(candidates.values())
```

### 4.3 相似度计算

**多维度相似度融合**：

```python
def compute_similarity(query, case):
    # 1. NL语义相似度
    nl_sim = cosine_similarity(query.nl_embedding, case.nl_embedding)
    
    # 2. 模板结构相似度
    template_sim = cosine_similarity(query.nl_template_embedding, 
                                     case.nl_template_embedding)
    
    # 3. SQL结构相似度
    structure_sim = euclidean_similarity(
        query.sql_fingerprint.to_feature_vector(),
        case.sql_fingerprint.to_feature_vector()
    )
    
    # 4. 意图一致性
    intent_sim = jaccard_similarity(query.query_intents, case.query_intents)
    
    # 5. Schema覆盖度
    schema_sim = len(set(query.tables) & set(case.tables)) / len(query.tables)
    
    # 加权融合
    final_sim = 0.3 * nl_sim + 0.25 * template_sim + 0.2 * structure_sim 
              + 0.15 * intent_sim + 0.1 * schema_sim
    
    return final_sim
```

### 4.4 负例区分机制（核心创新）

**问题**："销售额top10" vs "销售额bottom10" 语义极近但逻辑相反

**解决方案**：

```python
def contrastive_check(query, candidate_case):
    """对比检查：区分相似但不同的查询"""
    
    # 1. 检查负例关联
    if candidate_case.case_id in query.confusable_case_ids:
        # 触发深度对比
        return deep_contrastive_analysis(query, candidate_case)
    
    # 2. 关键语义差异检测
    critical_diffs = detect_critical_differences(query, candidate_case)
    # 检测项：排序方向、聚合类型、时间范围方向
    
    # 3. 结构差异验证
    if candidate_case.confusion_notes:
        for key, note in candidate_case.confusion_notes.items():
            if applies_to(query, key):
                return False, note
    
    return True, "匹配通过"
```

**关键差异检测项**：

| 检测项 | 相似但不同示例 | 检测方法 |
|--------|---------------|---------|
| **排序方向** | "top10" vs "bottom10" | ORDER BY方向检查 |
| **聚合类型** | "总量" vs "平均值" | 聚合函数检查(SUM vs AVG) |
| **时间方向** | "最近7天" vs "过去7天" | 时间边界语义分析 |
| **比较方向** | "超过100万" vs "低于100万" | WHERE操作符检查(GT vs LT) |
| **存在性** | "有没有" vs "有多少" | 查询意图检查(EXISTS vs COUNT) |

### 4.5 匹配质量分级

将匹配结果按质量分为四个等级，驱动不同的案例应用策略：

```python
MATCH_QUALITY_CONFIG = {
    'high':   0.95,    # 高质量匹配 → 直接复用SQL
    'medium': 0.85,    # 中等匹配 → 模板实例化
    'low':    0.70,    # 低质量匹配 → 作为few-shot示例

def classify_match_quality(similarity_score):
    if similarity_score >= MATCH_QUALITY_CONFIG['high']:
        return 'high', '直接复用SQL'
    elif similarity_score >= MATCH_QUALITY_CONFIG['medium']:
        return 'medium', '模板实例化'
    elif similarity_score >= MATCH_QUALITY_CONFIG['low']:
        return 'low', '作为few-shot示例'
    else:
        return 'none', '纯NL2SQL生成'
```

## 五、案例应用与NL2SQL集成（RQ3）

### 5.1 核心定位

**案例库不替代NL2SQL生成器，而是增强它。** 根据匹配质量等级，案例库以不同方式与NL2SQL系统集成：

```
用户查询 → 案例匹配 → 匹配质量分级 → 应用策略选择
                                          ↓
              ┌──────────────────────────────────────────┐
              │ 高质量(≥0.95)  → 直接复用SQL（0 LLM调用）│
              │ 中等(0.85-0.95) → 模板实例化（轻量推理）  │
              │ 低质量(0.70-0.85)→ few-shot示例增强NL2SQL│
              │ 无匹配(<0.70)   → 不干预，纯NL2SQL生成   │
              └──────────────────────────────────────────┘
                                          ↓
                                    执行验证 → 反馈
```

### 5.2 各等级应用策略

#### 高质量匹配（≥0.95）：直接复用

```python
def apply_high_quality_match(query, matched_case):
    """精确匹配 → 直接复用SQL，完全替代LLM调用"""
    # 1. 提取实体差异
    entity_mapping = align_entities(query.entity_slots, matched_case.entity_slots)
    
    # 2. 实例化SQL模板
    sql = instantiate_template(matched_case.sql_template, entity_mapping)
    
    # 3. 语法检查
    if not validate_sql_syntax(sql):
        # 降级到中等匹配策略
        return apply_medium_quality_match(query, matched_case)
    
    return sql, {'source': 'case_reuse', 'llm_calls': 0}
```

#### 中等匹配（0.85-0.95）：模板实例化

```python
def apply_medium_quality_match(query, matched_case):
    """模板匹配 → 减少LLM推理负担"""
    # 1. 使用匹配案例的SQL模板作为骨架
    skeleton = matched_case.sql_template
    
    # 2. LLM只需填充差异部分（而非从零生成）
    prompt = f"""
    参考SQL模板: {skeleton}
    用户查询: {query.nl_original}
    差异点: {identify_differences(query, matched_case)}
    
    请基于模板修改生成最终SQL。
    """
    sql = llm_generate(prompt)
    
    return sql, {'source': 'template_instantiation', 'llm_calls': 1}
```

#### 低质量匹配（0.70-0.85）：few-shot示例

```python
def apply_low_quality_match(query, matched_cases):
    """低匹配 → 提供few-shot示例辅助NL2SQL生成"""
    # 1. 选择Top-3相关案例作为示例
    examples = select_best_examples(matched_cases, k=3)
    
    # 2. 构造few-shot prompt
    prompt = build_fewshot_prompt(
        query=query.nl_original,
        examples=[(c.nl_original, c.sql_original) for c in examples],
        schema=query.schema_context
    )
    sql = llm_generate(prompt)
    
    return sql, {'source': 'fewshot_enhanced', 'llm_calls': 1}
```

#### 无匹配（<0.70）：不干预

```python
def apply_no_match(query):
    """无匹配 → 案例库不干预，由NL2SQL系统自行生成"""
    sql = nl2sql_generator.generate(query.nl_original, query.schema_context)
    
    return sql, {'source': 'pure_nl2sql', 'llm_calls': 1}
```

## 六、案例自动沉淀与演化（RQ4）

### 6.1 自动沉淀流程

案例库通过自动沉淀机制持续增长，每一次成功的NL2SQL交互都是潜在的新案例来源：

```
新查询 → NL2SQL生成 → 执行验证 → 质量评分 → 入库/拒绝
                                     ↓
                              ┌──────────────┐
                              │ 质量分 ≥ 0.8 │──→ 自动入库
                              │ 质量分 0.6-0.8│──→ 标记待审
                              │ 质量分 < 0.6 │──→ 拒绝入库
                              └──────────────┘
```

```python
class CaseAutoSedimenter:
    """案例自动沉淀器：将验证通过的查询自动转化为案例"""
    
    def try_sediment(self, query, generated_sql, execution_result):
        """尝试将一次成功的NL2SQL交互沉淀为案例"""
        
        # Step 1: 执行验证
        if not execution_result.success:
            return SedimentResult(accepted=False, reason='execution_failed')
        
        # Step 2: 质量评分
        quality = self.quality_scorer.score(query, generated_sql, execution_result)
        
        # Step 3: 等价性检查（避免重复入库）
        existing = self.equivalence_judge.find_equivalent(query, generated_sql)
        if existing:
            # 已有等价案例 → 更新统计而非新增
            self.update_existing_case(existing, query)
            return SedimentResult(accepted=False, reason='equivalent_exists',
                                  merged_to=existing.case_id)
        
        # Step 4: 按质量分处理
        if quality.score >= 0.8:
            new_case = self.create_case(query, generated_sql, quality)
            self.case_store.insert(new_case)
            return SedimentResult(accepted=True, case_id=new_case.case_id)
        elif quality.score >= 0.6:
            self.pending_review.add(query, generated_sql, quality)
            return SedimentResult(accepted=False, reason='pending_review')
        else:
            return SedimentResult(accepted=False, reason='low_quality')
    
    def create_case(self, query, sql, quality):
        """从原始查询构建完整案例"""
        # 1. 实体识别与槽位提取
        entity_slots = self.entity_extractor.extract(query, sql)
        # 2. 模板抽象
        nl_template = self.template_builder.abstract_nl(query, entity_slots)
        sql_template = self.template_builder.abstract_sql(sql, entity_slots)
        # 3. 结构指纹生成
        fingerprint = SQLStructureFingerprint.from_sql(sql)
        # 4. 向量编码
        nl_emb = self.encoder.encode(query)
        tmpl_emb = self.encoder.encode(nl_template)
        # 5. 组装案例
        return NL2SQLCase(
            case_id=generate_id(),
            nl_original=query, sql_original=sql,
            entity_slots=entity_slots,
            nl_template=nl_template, sql_template=sql_template,
            nl_embedding=nl_emb, nl_template_embedding=tmpl_emb,
            sql_fingerprint=fingerprint,
            quality_score=quality.score,
            # ... 其他字段
        )
```

### 6.2 质量评分机制

```python
class CaseQualityScorer:
    """案例质量评分器：多维度自动验证"""
    
    def score(self, query, sql, execution_result) -> QualityScore:
        checks = {}
        
        # 1. 执行成功性（基础门槛）
        checks['exec_success'] = float(execution_result.success)
        
        # 2. 结果合理性
        checks['result_reasonable'] = self.check_result_reasonable(
            execution_result.data
        )
        # 检查项：count不能为负、时间范围合理、数值范围合理
        
        # 3. 一致性验证
        checks['consistency'] = self.check_consistency(sql, query)
        # 同一模板的历史案例结果是否一致
        
        # 4. 对抗验证
        checks['robustness'] = self.check_robustness(sql, query)
        # 微扰问题是否得到一致结果
        
        # 5. 模板可抽象性
        checks['abstractable'] = self.check_abstractable(query, sql)
        # 能否成功提取实体槽位并形成模板
        
        # 加权计算
        weights = {'exec_success': 0.3, 'result_reasonable': 0.25,
                   'consistency': 0.2, 'robustness': 0.15, 'abstractable': 0.1}
        total = sum(checks[k] * weights[k] for k in checks)
        
        return QualityScore(score=total, details=checks)
```

### 6.3 案例演化机制

案例库是一个活的系统，需要持续演化以适应变化：

```python
class CaseEvolutionManager:
    """案例演化管理器：管理案例的新增、更新、退化和废弃"""
    
    # ===== 1. 新增：验证通过的查询自动入库 =====
    def on_new_case(self, case):
        """新案例入库后的处理"""
        # 1. 建立与已有案例的关联
        self.link_confusable_cases(case)
        # 2. 更新索引
        self.index_manager.add(case)
        # 3. 触发相关案例的负例关系更新
        self.update_confusable_links(case)
    
    # ===== 2. 更新：使用反馈调整质量分 =====
    def on_feedback(self, case_id, feedback):
        """根据使用反馈更新案例质量"""
        case = self.case_store.get(case_id)
        case.hit_count += 1
        
        if feedback.success:
            case.success_count += 1
            # 成功使用 → 质量分微调上升
            case.quality_score = min(1.0, 
                case.quality_score + 0.01 * (1 - case.quality_score))
        else:
            # 失败 → 质量分下降
            case.quality_score = max(0.0,
                case.quality_score - 0.05)
            # 记录失败原因
            case.failure_log.append(feedback.reason)
        
        case.last_used = datetime.now()
        self.case_store.update(case)
    
    # ===== 3. 退化检测：Schema变更 + 冲突检测 =====
    def detect_degradation(self):
        """定期扫描，检测退化案例"""
        degraded = []
        
        for case in self.case_store.all():
            # 3a. Schema变更检测
            current_schema = self.schema_registry.get_current()
            if self.schema_changed(case.schema_subset, current_schema):
                degraded.append((case, 'schema_outdated'))
                continue
            
            # 3b. 表/列失效检测
            invalid_refs = self.check_invalid_refs(case, current_schema)
            if invalid_refs:
                degraded.append((case, 'refs_invalid', invalid_refs))
                continue
            
            # 3c. 使用率异常检测
            if case.hit_count > 10 and case.success_count / case.hit_count < 0.5:
                degraded.append((case, 'low_success_rate'))
            
            # 3d. 冲突检测：同一查询匹配到多个互相矛盾的案例
            conflicts = self.detect_conflicts(case)
            if conflicts:
                degraded.append((case, 'conflicting', conflicts))
        
        return degraded
    
    # ===== 4. 废弃：低质量/过期案例清理 =====
    def cleanup(self):
        """清理低质量和过期案例"""
        removed = []
        
        for case in self.case_store.all():
            should_remove = False
            reason = ''
            
            # 4a. 质量分过低
            if case.quality_score < 0.3:
                should_remove = True
                reason = f'quality_too_low ({case.quality_score:.2f})'
            
            # 4b. 长期未使用
            days_unused = (datetime.now() - case.last_used).days
            if days_unused > 180 and case.hit_count < 5:
                should_remove = True
                reason = f'stale ({days_unused} days, {case.hit_count} hits)'
            
            # 4c. Schema已失效且无法修复
            if case.schema_version != self.schema_registry.current_version:
                if not self.can_migrate(case):
                    should_remove = True
                    reason = 'schema_migration_failed'
            
            if should_remove:
                # 不物理删除，标记为废弃
                case.status = 'deprecated'
                case.deprecated_reason = reason
                self.case_store.update(case)
                removed.append((case.case_id, reason))
        
        return removed
```

### 6.4 演化生命周期状态图

```
  ┌──────┐    沉淀    ┌──────┐   使用/反馈  ┌──────┐
  │ 候选 │──────────→│ 活跃 │←──────────→│ 成熟 │
  └──────┘           └──────┘            └──────┘
     ↑                  │                    │
     │              退化检测              退化检测
     │                  ↓                    ↓
     │              ┌──────┐            ┌──────┐
     │              │ 退化 │───修复───→│ 活跃 │
     │              └──────┘            └──────┘
     │                  │
     │              无法修复
     │                  ↓
     │              ┌──────┐
     └──待审核──────│ 废弃 │
                    └──────┘
```

| 状态 | 条件 | 可用性 |
|------|------|--------|
| **候选** | 新沉淀，质量分0.6-0.8 | 不参与匹配，等待审核 |
| **活跃** | 质量分≥0.8，Schema有效 | 正常参与匹配 |
| **成熟** | 质量分≥0.9，成功率>80%，使用次数>50 | 高优先级匹配 |
| **退化** | Schema变更/成功率下降 | 降权参与匹配 |
| **废弃** | 质量分<0.3/长期未用/Schema失效 | 不参与匹配 |
