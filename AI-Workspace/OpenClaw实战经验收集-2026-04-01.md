### 一、生产级自动化案例（$15k → $271/月）

**来源**：Reddit r/GrowthHacking

**核心架构**：
- 34 cron jobs + 71 scripts
- 4个专业agent（content, SEO, deals, recruiting）
- 共享"大脑"记忆系统

**关键模式**：

#### 1. 决策Inbox模式（5AM-11AM）
```
5 AM – Quote mining: 从历史内容挖掘线索
6 AM – Content angles: 生成hooks和outline
7 AM – SEO/AEO actions: 识别keyword gaps
8 AM – Deal of the day: CRM扫描，高价值lead跟进
9–11 AM – Recruiting drop, product pulse, connection
```

**设计原则**：每个cron终点是30秒内可做的yes/no决策

#### 2. 共享大脑架构
- 所有agent读写同一记忆源
- 一次拒绝，所有agent学习
- 防止重复推荐和矛盾

#### 3. 失败设计原则
- **静默失败监控**：重建delivery pipeline，显式失败通知
- **数据artifacts强制**：agent必须先运行脚本读实际输出，禁止虚构
- **14天去重**：防止重复推荐同一prospect

#### 4. 成本优化策略
- 机械任务用便宜模型（ETL, compaction, dedup）
- 策略/创意用premium模型
- 添加"成本审计"cron job
- 案例：infra jobs从$36/周降至$7/周

#### 5. 维护层Agents
- **Monthly meta-agent**：质疑系统存在，删除无价值agent
- **Weekly self-healing**：自动修复failed jobs
- **Weekly system janitor**：清理文件，跟踪成本

#### 6. 并行化策略
- 6个sub-agents并行运行
- 8分钟完成，人类团队需一周

### 二、10条硬学习经验（Reddit实战）

**来源**：Reddit r/clawdbot "10 things I Learned"

#### 1. 工具配置是第一步
```json
"tools": { "profile": "full" }
```
默认profile是messaging only，不给工具agent什么也做不了。

#### 2. Context截断的隐形杀手
- **20,000字符限制**：超长文件被静默截断
- **截断策略**：保留前70% + 后20%，中间10%丢失
- **关键规则放顶部**：中间是"kill zone"

检查方法：`/context list`，看到`TRUNCATED`说明有内容被截。

#### 3. 80% Token是浪费
- Input token占比148:1（139M input, 935K output）
- Agent不是在思考，是在反复review
- **解决方案**：
  - 核心规则放SOUL.md
  - 其他内容移到memory folder
  - 用semantic search按需fetch
  - Token降低40-60%

#### 4. 模型选择策略
- **人类交互agent → Opus**：tone, emotion, 写作质量
- **数据扫描agent → GPT-5.4**：更快、更稳定、更听话
- 不是省钱，是right tool right job

#### 5. 记忆系统崩溃周期
- MEMORY.md第1天干净，第3周变成junk drawer
- 新旧信息混杂，agent无法区分
- **解决方案**：
  - 开启hybrid search + temporal decay
  - 新记忆权重高，旧记忆自动衰减

#### 6. 无下游消费者的agent是成本黑洞
- 每个agent必须回答三个问题：
  1. 我产出什么
  2. 谁接收我的输出
  3. 我从不碰什么
- 不能回答问题2 → 该角色不该存在
- 案例：删除1个agent，Token降44%，速度提升62%

#### 7. Telegram多Topic配置
```yaml
channels:
  telegram:
    groups:
      "-100xxxxxxxxxx":
        topics:
          "daily-brief": { agentId: "nexus" }
          "radar": { agentId: "scout" }
          "content": { agentId: "quill" }
          "runtime": { agentId: "forge" }
          "support": { agentId: "guide" }
```
每个topic独立session/workspace/memory

#### 8. Agent不知道消息来源
- ACP bridge provenance模式：
  - `meta`：系统知道消息来源
  - `meta+receipt`：agent也能看到

#### 9. 备份是生存刚需
```bash
openclaw backup create
openclaw backup create --only-config
openclaw backup verify
```
默认备份：local state, config, credentials, sessions

#### 10. 先运行，再架构
- Day 1：一个agent + Telegram + 一个日常任务
- Week 1：修复工具profile、截断、响应慢
- Week 2：检查token账单，瘦身prompt
- Week 3：添加第二个agent
- Week 4：架构自然生长

**关键洞察**：架构是从解决问题中生长出来的，不是设计出来的。

### 三、14条Power User实践（200+小时）

**来源**：MindStudio Blog

#### Tips 1-3：架构设计

1. **先画Agent Graph**
   - 纸上/白板先画出来
   - Map inputs, outputs, decision points
   - Flag failure cascading点

2. **单一责任原则**
   - Research agent只research
   - Formatting agent只format
   - 一个sentence描述不了 → split

3. **子Agent并行化**
   - 串行45秒 → 并行20秒
   - 独立任务同时fire

#### Tips 4-6：模型路由

4. **模型分层策略**
   - **Tier 1**：Classification/routing/yes/no → GPT-4o Mini/Haiku/Flash
   - **Tier 2**：Extraction/summarization → Mid-tier
   - **Tier 3**：Complex reasoning/code/writing → Frontier

5. **专用Router Agent**
   - 入口点统一路由
   - 集中管理thresholds和fallback
   - 换模型改一处，不用hunt dozen workflows

6. **缓存重复查询**
   - TTL设置
   - 高volume workflow降30-50%成本

#### Tips 7-8：Telegram管理

7. **Topic线程分离**
   - Errors & Alerts（不mute）
   - Completed Runs
   - Approvals
   - Info（可mute，定时review）

8. **标准化消息格式**
```
[STATUS] Agent Name
Task: Brief description
Result: One-line outcome
Time: HH:MM:SS
```

#### Tips 9-10：子Agent通信

9. **结构化Context传递**
   - 用JSON，不用raw text
   - 定义output schema
   - 每步validate

10. **幂等性设计**
    - Check-before-write
    - Unique ID deduplication
    - 防止重复执行副作用

#### Tips 11-12：Cron防错

11. **错峰调度**
    - 不用:00，用:07, :15, :23, :34
    - 避免rate limit和throttle

12. **Retry逻辑**
    - Retry count（默认3）
    - Retry delay（exponential backoff: 1s→4s→16s）
    - Failure notification → Telegram Errors thread
    - Recovery behavior定义

#### Tips 13-14：安全实践

13. **Credential禁止在Prompt中**
    - 用环境变量store
    - 定期审计embedded "just for testing" credentials

14. **季度权限审计**
    - Read-only够不够？
    - API还在用吗？
    - Credentials过期了吗？

### 四、OWASP Top 10 for Agentic Applications

**来源**：Akamai Security Research

#### 10大威胁

| ID | 威胁 | OpenClaw案例 | 防御方案 |
|---|------|-------------|----------|
| ASI01 | Agent Goal Hijack | 邮件/文档内嵌指令劫持 | 指令/数据通道分离 |
| ASI02 | Tool Misuse | Shell滥用，skills全权限 | Capability-based access |
| ASI03 | Privilege Abuse | 继承用户全部权限 | 专用service account |
| ASI04 | Supply Chain | ClawHub恶意skills | Verify before execute |
| ASI05 | RCE | Bash tool任意命令 | Zero-access sandboxing |
| ASI06 | Memory Poisoning | SOUL.md/MEMORY.md注入 | 静态知识/动态指令分离 |
| ASI07 | Inter-Agent通信 | Moltbook恶意agent对话 | 认证+加密+语义验证 |
| ASI08 | Cascading Failures | 单点故障传播 | Circuit breakers |
| ASI09 | Trust Exploitation | 用户信任weaponize | Sensitive操作需确认 |
| ASI10 | Rogue Agents | 内部逻辑被篡改 | SOUL.md/MEMORY.md integrity check |

#### 三层防御策略

**Layer 1: 传统安全**
- Network: Bind localhost, authenticate, encrypt
- Credentials: Secret management, rotate, scope
- Dependencies: Audit, update, monitor

**Layer 2: 架构控制**
- Capability restriction: 最小权限
- Context separation: 指令vs数据通道
- Execution gates: 确认机制
- Failure isolation: Circuit breakers

**Layer 3: Runtime保护**
- Input filtering: 注入检测
- Behavioral monitoring: 资源访问模式
- Memory auditing: 持久状态审查

### 五、常见3大错误

**来源**：Towards Data Science

#### 1. 不用Docker运行
- 风险：无隔离，环境依赖问题
- 解决：直接让coding agent设置Docker

#### 2. 培训不足
- 案例：给AWS access但不告诉怎么用
- 结果：Slack乱回复，被tag不知道该干嘛
- 解决：
  - 明确what to do / what NOT to do
  - 具体任务执行步骤
  - 交互协议定义

#### 3. 权限不足
- 案例：要求AWS任务但只给DynamoDB不给S3
- Agent不知道要申请权限
- 解决：像对待intern一样给完整权限

## 🔑 核心模式总结

### 1. 生产级系统的特征
- **决策导向**：终点是yes/no action，不是document
- **共享记忆**：防止agent重复和矛盾
- **维护层**：meta-agent质疑、healing、清理
- **成本审计**：持续监控token和API成本

### 2. 架构生长法
- Day 1: 一个agent + 一个任务
- Week 1-4: 从问题中学习，架构自然生长
- 不要先画图再运行

### 3. Token优化核心
- Context瘦身（MEMORY.md从22KB→按需fetch）
- 模型分层路由
- 无消费者agent删除

### 4. 可靠性设计
- 幂等性（防重复执行）
- Retry + exponential backoff
- 错峰调度（避rate limit）
- 失败隔离（circuit breakers）

### 5. 安全核心
- OWASP Top 10框架
- 指令/数据分离
- Capability-based access
- 季度审计
