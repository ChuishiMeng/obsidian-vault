## 一句话总结

作者花 3 个月周末从零搭建 Data Agent，迭代三版后放弃，最终转向 Skills 方案——**把业务隐性知识固化进 Skill，只服务自己，反而解决了最难的问题**。

---

## 三版 Data Agent 迭代历程

### 第一版：语义层约束（12天）

**架构**：用 YAML 预定义指标、维度、别名，LLM 只负责选择

**实现功能**：
- 自然语言查询
- 指标/维度自助配置
- 版本回滚
- 事件库做异常归因

**问题**：
- 约束太死，只能做 L2 层的数据获取
- 无法做 L3 层的分析诊断
- 把 LLM 当成高级搜索框，没发挥推理能力

---

### 第二版：Agent Loop

**架构**：Plan→Execute→Reflect 三步循环

**实现功能**：
- LLM 先规划分析路径
- 执行 SQL 获取结果
- 反思结果是否正确、是否需要修正
- WebSocket 流式输出
- RAG 上下文检索

**效果**：用 DeepSeek 验证，终于有了"Agent 在做分析"的感觉

**踩坑**：
- 前端 loading 很久才出结果（异步处理不够好）
- 上下文理解不到位，追问时丢 context
- 对话设计没有做到 AI native

---

### 第三版：产品化重构（大几万行代码）

**架构**：三层解耦（编排/推理/执行）

**设计要点**：
- 推理层统一适配器，支持 DeepSeek、OpenAI、Claude、Ollama 等任意模型
- 知识库分层管理（语义层、记忆、模板、元数据各司其职）

**产品定位**：
- 开源工具，本地优先
- 数据不上服务器
- 过程透明、结果可验证、知识可修正

**做到一半停了**，因为摸到了绕不过去的坎：

---

## 四个绕不过去的核心坎

| 问题 | 具体表现 |
|------|----------|
| **业务隐性知识** | 指标口径、业务上下文、分析惯例，不在数据库里，在人脑子里。没有这层知识，AI 分析看着对但经不起追问 |
| **语义转化** | "最近转化率怎么样"——"最近"是多久？"转化率"是哪个环节的？模糊性需要大量映射和消歧 |
| **模型依赖** | OpenAI 的 Data Agent 做得好，是因为 GPT 5.4 够强 + 内部环境打通了业务策略隐性知识。换模型换环境，效果差不少 |
| **Tool Use 能力** | 可视化、上下文解读、多步分析编排，依赖模型工具调用能力，不同模型差距很大 |

## 从 Agent 到 Skills

作者把 Data Agent 经验做成了一套数据 Skills：

**核心差异**：
- 不解决"所有人都能用"
- 只解决"我自己用得好"

**消掉的变量**：
- 业务知识不用从零构建（就是自己的）
- 语义映射不用消歧（自己知道问什么）

**实际案例**：
做数据异动分析时，Skill 主动点出某天某策略改动，与数据现象时间方向吻合。作者确认后确实存在——这个 context 作者当时不知道，但之前持续更新到知识库，Skill 自己关联上了。

## 关键洞察

### 1. 大模型在吃软件

> "花几个月做的产品，用 Skills 几天就能实现核心诉求"

作者不看好需要独立部署、独立维护的工具软件，倾向于把能力直接做进 Skills。

### 2. Demo vs 产品的难度差距

| 层级 | 难度 |
|------|------|
| Demo | 1 |
| 能用的产品 | 100 |
| 好用的产品 | 10000 |

> "GitHub 是所有人秀 demo 的地方，但真正能用好用的产品没几个"

### 3. 数据岗位的 AI 暴露度

- Karpathy 职业暴露度评分：数据岗位 **9分**（满分10）
- 但就业增长预期 **+34%**（所有职业最高）
- Anthropic：AI 理论上能覆盖 94% 知识工作，实际渗透率只有 33%

**结论**：暴露度高 ≠ 要消失，反而说明与 AI 交集大，能借力的空间也大。
