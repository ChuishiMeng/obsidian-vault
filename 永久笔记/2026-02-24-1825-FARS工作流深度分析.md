## FARS的四大设计原则

### 1️⃣ First Principles思维

#### 人类研究的约束（我们受困于此）

| 约束类型 | 具体表现 | 我们P9中的体现 |
|----------|----------|--------------|
| **人力有限** | 高失败率、入门门槛高 | 每个P9阶段需人工确认，效率低 |
| **注意力有限** | 无法同时跟进多个方向 | 单线程执行，无并行探索 |
| **制度激励** | 只发成功实验、负面结果不发表 | 实验失败即停止，不写失败报告 |

#### FARS的解放（First Principles）

> "Instead of optimizing for compatibility with existing academic formats, FARS is built from first principles: efficiently and reliably expand the frontier of knowledge."

**核心问题**：
- 我们的目标应该是"拓展知识前沿"，不是"兼容学术格式"

---

### 2️⃣ Short Paper策略

#### FARS的论文格式定义

| 维度 | FARS的定义 | 我们P9的目标 |
|------|-----------|-------------|
| **长度** | Short paper（无约束） | 传统顶会格式（8页+） |
| **贡献** | 单一聚焦贡献 | 多贡献单元 |
| **负面结果** | **显式报告** ✅ | 避免提及 ✗ |
| **结构** | 最小必要结构 | 严格Introduction→Method→...格式 |

#### 对我们的帮助

**问题**：
- v1：写完整论文（Persona+RAG+评估）
- v2：写完整论文（四模块架构）
- v3：写完整论文（跨问题一致性）
- 每次失败成本高（完整论文投不中=2-3个月浪费）

**改进方案**：
```
P9流程改为：
P1 → P2 → P3 → P4 → P5a（单一假设）
                              ↓
                          P6a → P7a → P8a → P9a（short paper）
                              ↓
                    验证成功/有意义 → P5b（扩展为完整论文）
```

**好处**：
- 失败成本低：只浪费假设，不浪费完整论文
- 迭代速度快：short paper 2天 vs 完整论文2个月
- 知识密度高：每篇一个明确贡献
- 负面结果可发表：FA0121模式

### 3️⃣ 共享文件系统协作

#### FARS的Agent协作机制

```
传统多Agent系统（AgentLab模式）：
Agent A ──▶ 直接通信 ──▶ Agent B
           ↑        消息队列      ↑
        复杂通信协议            成本高

FARS的共享文件系统：
Agent A ──┐
          ├─▶ 读写共享文件系统 ──┤
Agent B ──┘          ↑
                 无需直接通信
```

#### 对我们的帮助

**现状问题**：
- P9各阶段文档独立
- P2文献与P4空白无共享状态
- P7实验结果与P9写作无数据通道

**改进方案**：
```yaml
# P9共享工作区设计
workspace: ~/.openclaw/workspace/research/shared/

project_template:
  project-{ID}:
    # Ideation共享
    literature_cache/         # P2读取/写入
    hypothesis_log/          # P2生成的所有假设

    # Planning共享
    design_docs/            # P5读取
    data_requirements/       # P5→P6传递

    # Experiment共享
    code_experiments/        # P7读取/写入
    results_data/           # P7→P8传递

    # Writing共享
    paper_drafts/           # P9读取
    figures/               # P9生成

    # 跨阶段共享
    shared_memory/          # 全局上下文
    failed_hypotheses/      # 失败案例库
    common_baselines/       # 共享基线

    # 元数据
    metadata.yaml           # 项目状态、当前阶段、耗时
```

**好处**：
- 无需Agent间复杂通信
- 文件系统即持久化记忆
- 支持大规模并发项目
- 可随时恢复中断状态

### 4️⃣ 负责任发布机制

#### FARS的三层安全网

```
Layer 1: 自动审查（Agent）
    ↓ 通过后
Layer 2: 人工审查（≥3名5年+经验研究者）
    ↓ 通过后
Layer 3: 标签化（AI-generated）
    ↓
提交到arXiv
```

#### 对我们的帮助

**现状问题**：
- 无发布前审查机制
- v1、v2论文写完即投（无内部评审）
- 无AI生成标签

**改进方案**：
```python
# P9新增：发布控制器
class P9Publisher:
    def publish_workflow(self, paper_path):
        # Layer 1: 自动审查
        if not self.auto_review(paper_path):
            print("❌ 自动审查未通过：改进P7实验深度")
            return False

        # Layer 2: 人工审查（用户）
        if not self.user_confirm(paper_path):
            print("⏸️ 等待人工审查确认")
            return False

        # Layer 3: 标签化
        labeled_path = self.add_ai_label(paper_path)
        print(f"✅ 准备提交：{labeled_path}")

        return True

    def auto_review(self, paper_path):
        """基于Agentic Reviewer的自动审查"""
        # 检查点：
        # 1. 实验完整性（有对比实验吗？）
        # 2. 结果显著性（有统计检验吗？）
        # 3. 写作清晰度（有明确贡献陈述吗？）
        # 4. 引用完整性（核心论文都引用了吗？）
        return self.checkpoints(paper_path)
```

### 1️⃣ Short Paper模板

创建 `templates/short_paper.md`：

```markdown
# [Hypothesis]

## Abstract
[1-2段：问题+方法+结果]

[问题背景 → 相关工作 → 贡献声明]

[实验设置 + 主要结果表/图]

[结果分析 + 对比baseline]

## Discussion
[结果解读 + 局限性]

[总结 + 未来工作]

## References
```

**应用场景**：
- v3验证ACS（跨问题态度一致性）
- 先写short paper验证核心假设
- 验证成功后再扩展为KDD论文

**FA0121诊断报告模板**：
```markdown
# Negative Results: [Experiment Name]

## Hypothesis
[原始假设]

## Method
[实现方法]

## Results
[实验结果：无提升]

## Diagnosis
[系统性诊断]
- 是否实现正确？[是]
- 是否超参充分？[是]

- Gate和n-gram embedding训练是耦合动力学问题
- 简单加监督无法解决

- 需要从架构层面（而非监督层面）解决偏置问题
```

### 3️⃣ P9共享工作区实现

创建 `workspace/shared/` 架构：

```python
# shared_workspace.py
class SharedWorkspace:
    def __init__(self, project_id):
        self.project_dir = f"workspace/shared/project-{project_id}"
        self.setup_structure()

    def setup_structure(self):
        """创建标准目录结构"""
        dirs = [
            "literature_cache",
            "hypothesis_log",
            "design_docs",
            "code_experiments",
            "results_data",
            "paper_drafts",
            "figures",
            "shared_memory",
            "failed_hypotheses"
        ]
        for d in dirs:
            os.makedirs(f"{self.project_dir}/{d}", exist_ok=True)

    def get_current_state(self):
        """读取当前项目状态"""
        with open(f"{self.project_dir}/metadata.yaml") as f:
            return yaml.safe_load(f)

    def update_stage(self, stage, outputs):
        """更新阶段状态"""
        state = self.get_current_state()
        state['current_stage'] = stage
        state[f"{stage}_outputs"] = outputs
        with open(f"{self.project_dir}/metadata.yaml", 'w') as f:
            yaml.dump(state, f)
```

## 对我们三个项目的具体应用

### Virtual Users (v3) - 最紧迫

#### 问题
- v3聚焦"跨问题态度一致性"，实验失败风险高
- 当前模式：完整KDD论文，失败成本极高

#### 改进方案
```
立即行动：
1. 先写short paper验证ACS假设
   - 标题：Attitude Consistency Across Questions in LLM-based Virtual Users
   - 目标：KDD Workshop或arXiv预印本
   - 周期：1-2周

2. 建立失败案例库
   - 每次实验无论成功/失败都记录
   - 使用FA0121的"诊断性实验"模板

3. 共享工作区
   - v3项目迁移到 `workspace/shared/project-v3/`
   - 各阶段通过文件系统通信
```

**预期收益**：
- 降低失败成本：short paper失败浪费1周 vs 完整论文2月
- 提高迭代速度：每2周验证一个新想法
- 积累负结果知识：失败案例库成为研究资产

## 核心收获

1. **Short Paper是策略不是妥协**
   - 快速验证假设
   - 降低失败成本
   - 聚焦单一贡献

2. **共享文件系统优于直接通信**
   - 无需复杂协议
   - 持久化记忆
   - 可大规模并发

3. **三层安全网确保负责任**
   - 自动审查过滤低质量
   - 人工审查确保准确性
   - 标签化保持透明

4. **失败案例库是资产不是负担**
   - 避免重复踩坑
   - 展示研究诚实性
   - 可能为他人启发
